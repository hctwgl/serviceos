# ADR-018：配置 Schema 与表达式运行时

## 状态

Proposed

## 上下文

ServiceOS 需要让业务人员配置流程、表单、资料、SLA、派单和价格。若直接允许任意脚本，将带来安全、性能、可解释性、版本兼容和运维风险；若完全写死，则无法支持多车企差异。

## 决策

采用分层配置执行模型：

1. **结构层**：所有配置由 JSON/YAML 表达，并使用版本化 JSON Schema 校验；
2. **条件层**：普通布尔条件使用受限表达式语言；
3. **决策层**：多条件业务规则优先使用决策表；
4. **公式层**：价格与指标公式使用受限数值表达式；
5. **脚本层**：只有平台管理员可使用受控脚本，默认关闭；
6. **扩展层**：高复杂逻辑通过经过审核的插件扩展点实现。

禁止业务配置直接：

- 访问数据库；
- 发起任意网络请求；
- 读写文件系统；
- 启动线程；
- 反射加载类；
- 修改历史配置版本；
- 获取未授权敏感字段。

## 运行时要求

表达式运行时必须具备：

- 白名单函数；
- 类型检查；
- 执行超时；
- 内存和复杂度限制；
- 确定性；
- 输入输出审计；
- 命中解释；
- 版本锁定；
- 沙箱隔离；
- 批量样本模拟。

## 表达式上下文

上下文由平台显式构造，至少区分：

- `workOrder`
- `customer`
- `vehicle`
- `asset`
- `project`
- `region`
- `task`
- `fulfillmentFacts`
- `actor`
- `clock`

配置不能访问未声明属性。敏感字段默认不进入上下文。

## 备选方案

### 任意 Groovy/JavaScript

优点是灵活；缺点是攻击面大、不可控、难解释，拒绝作为默认方案。

### 全部自研 DSL

长期可控，但首期成本高。只在稳定需求形成后逐步演进，不作为 MVP 前置条件。

### 全部写代码

短期简单，但会复现旧系统问题，拒绝。

## 后果

正面：

- 大多数车企差异可配置；
- 配置可静态校验和模拟；
- 规则结果可解释；
- 风险与权限可控。

负面：

- 部分复杂需求需要插件；
- 需要建设 Schema 注册表、模拟器和发布流水线；
- 表达式能力必须克制，不能追求“什么都能写”。

## 复评触发条件

- 受控表达式无法覆盖超过 20% 的新增需求；
- 插件数量持续增长且重复；
- 性能无法满足批量派单或计价；
- 需要标准 BPMN/DMN 互操作。
