# PR226：任务型操作系统重构实施计划

## 1. 分析结论

PR226 的目标不是扩大 ServiceOS 的业务模型，而是把 PR225 已经提供的项目与履约方案能力重新组织成两个稳定的工作台：

- 项目经理进入项目后，优先得到项目状态、当前方案/版本、风险和下一步动作；
- 履约配置人员进入方案后，优先看到流程、节点责任、规则引用和版本发布路径。

本次改造只发生在 `serviceos-frontend/apps/admin`。不新增领域、不新增 `fulfillment` 模块、不修改后端模型和 API，不用本地伪造数据填充缺失的业务事实。

基线为 PR225 的 Admin 实现（当前分支 `codex/pr226-task-os`，基线提交 `c4d83d36`）。设计基准采用 DEC-018 的“经典专业任务型”方向：轻量壳层、高信息密度、克制的状态色、以任务和责任为视觉主线，避免营销式 Hero、大面积说明文字、重复指标卡和大屏化表达。

## 2. PR225 审计

### 保留

- `/projects/:projectId` 项目工作区路由、项目权限和现有 query/presenter 数据链路；
- `/projects/:projectId/fulfillment` 方案工作台路由，以及草稿、发布、预览、版本历史的既有 API 和命令；
- Project Summary、履约阶段条、风险、业务时间线、Blueprint Sidebar、Workflow Canvas、任务模板、表单、SLA、版本时间线这些业务组件的职责；
- 现有设计系统 token、App Shell、路由级权限、`If-Match`/幂等/版本校验，不把业务页面改写到 Vben 基础工程中；
- 工单 Workspace 和后端实现。本次只保留项目页到工单页的任务入口，不把运行态数据复制到 Blueprint Designer。

### 重构

#### Project Workspace

PR225 的项目页由摘要、上下文条、六个带大段描述的 Tab、四张指标卡、阶段条、时间线、SLA 卡、风险卡和“履约责任边界”说明组成。它虽然信息完整，但优先级被卡片和说明文字打散，风险与下一动作没有成为首屏任务。

PR226 将其收敛为：

1. 紧凑项目摘要：项目、客户、区域、状态、当前方案和版本；
2. 一个明确的下一动作入口；
3. 履约阶段条；
4. 左侧业务时间线；
5. 右侧风险和 SLA 健康度；
6. 资源网络、质量异常、数据分析作为明确的后续工作入口，而不是解释型占位卡。

项目页继续使用真实项目/工单查询结果，时间线呈现业务事件，不新增日志 API，也不把项目字段逐项铺开。

#### Fulfillment Blueprint Designer

PR225 的方案页已经有三栏雏形，但仍有以下问题：

- 设计页显示“运行工单、成功率”等运行态指标，违反 DEC-008/009 的设计态与运行态边界；
- `seedDesignSurfaces()` 生成了任务、表单字段和 SLA 规则，造成伪造的规则事实；
- 左侧只有七个栏目，缺少阶段管理、验收、结算、模拟运行等设计入口；
- 流程设计器、长篇说明和大面积卡片仍然像配置后台；
- 真实版本历史存在，但没有成为版本发布路径的主入口。

PR226 将其改为固定三栏工作台：左侧方案导航，中间规则/流程设计面，右侧节点属性。顶部保留继续编辑草稿、创建新版本（无活动草稿时明确禁用）、模拟运行、发布和查看历史等任务动作。流程节点从真实草稿或编译预览读取，节点属性显示责任角色、任务模板、SLA、表单和证据引用。

## 3. 组件拆分与调整

`components/serviceos/` 作为业务组件边界，页面只负责 query、状态和路由编排。

| 组件 | PR226 处理 |
| --- | --- |
| `ProjectSummaryHeader` | 保留，改为紧凑摘要与任务入口的头部 |
| `ProjectRiskPanel` | 新增项目专用风险面板；不影响工单页继续使用的 `RiskPanel` |
| `ProjectTimeline` | 新增/替换 `BusinessTimeline`，语义固定为业务事件 |
| `FulfillmentStageBar` | 保留，作为项目首屏阶段状态条 |
| `BlueprintSidebar` | 保留并压缩为固定设计导航 |
| `WorkflowCanvas` | 作为流程设计主画布，使用真实阶段节点 |
| `WorkflowNodePanel` | 新增右侧节点属性面板 |
| `TaskTemplatePanel` | 改为责任、输入、输出、表单、证据、SLA、完成条件的高密度视图 |
| `FormDesigner` | 保留三栏设计结构；没有字段定义时明确显示“接口未返回”，不创建假字段 |
| `SlaRulePanel` | 新增方案 SLA 规则面板；只展示真实 SLA 摘要和未返回的目标/预警/升级字段 |
| `VersionTimeline` | 保留真实版本历史，强化查看历史/创建版本/发布入口 |

如果替代组件完成且无调用方，删除 PR225 中重复的页面专用实现；不保留第二套流程、任务或表单领域模型。

## 4. 数据与接口边界

继续复用 PR225 已有接口：项目详情、项目工单/业务活动、履约方案详情、草稿、编译预览、校验、影响分析、发布和版本历史。Blueprint Designer 不再请求工单统计，不展示完成率或运行统计。

当前接口能返回阶段的任务类型、责任角色、表单/证据数量与摘要、SLA 摘要、异常和下一阶段摘要；不能返回表单字段定义，也不能返回 SLA 的目标/预警/升级拆分字段。对此采用显式的未配置/接口未返回状态，并引导进入已有草稿维护或发布流程，不使用 mock、默认成功或猜测性标签。

本 PR 不修改 Controller、OpenAPI、事件、Flyway、后端领域模型和共享 API 包。

## 5. 删除/禁止项

- 删除 Blueprint Designer 中的运行工单数、成功率、完成率等运行态指标；
- 删除 `seedDesignSurfaces()` 及其派生的伪造任务、字段和 SLA 数据；
- 删除或收敛大段 Hero/说明文字、重复指标卡和装饰性空白；
- 不新增普通列表、详情、编辑页面，不新增业务大屏，不新增后端接口；
- 不把表单 JSON、接口字段名或数据库字段作为用户界面主语。

## 6. 风险与控制

| 风险 | 控制 |
| --- | --- |
| 真实 API 没有返回表单字段详情 | 表单设计器显示结构化的“字段定义未返回”状态，不伪造字段 |
| 真实 API 没有拆分 SLA 目标/预警/升级 | 展示已有 SLA 摘要和引用关系，属性区明确缺失数据 |
| 设计态误用运行态 | 方案页只使用 profile/draft/preview/revision 数据；项目运行数据只留在 Project Workspace |
| 页面仍然被旧 CSS 卡片化 | 以现有 design-system token 为基础，减少固定高度和 overflow 裁切；在内置浏览器实测当前可用视口，并保留 1280/1080/760 断点的响应式布局 |
| 本地种子无法提供真实项目 | 不在前端增加 mock fallback；记录本地环境阻塞，真实数据截图只能来自可用 API 状态 |

## 7. 实施与验证

实施顺序：

1. 收敛 Project Workspace 的首屏任务结构和业务组件；
2. 重构 Blueprint Designer 三栏布局、导航、流程画布、节点属性和版本入口；
3. 删除运行态指标与本地伪造设计数据；
4. 执行 `git diff --check`、前端静态/类型/构建验证；
5. 用内置浏览器验证项目页和方案页的导航、节点选择、版本发布边界、长内容滚动和横向溢出；视口验证以当前可用的 1280×720 为实测证据，其他宽度由既有响应式断点和静态构建门禁覆盖；
6. 保存真实运行截图，并创建 Draft PR，明确本地验证与 CI 验证边界。

最低命令：

```bash
bash scripts/agent-verify.sh frontend
```

浏览器数据准备曾遇到一个本地环境事实：首次执行 `product-data:reset` 在配置基础发布阶段因“没有匹配的已发布 configuration bundle”失败。随后仅重启既有后端并通过现有 Admin 创建了真实项目履约方案，未添加前端 mock 或后端修复；当前方案仍是草稿，发布检查真实阻塞于“未绑定 Workflow 版本”和“未绑定 Configuration Bundle”，该边界在页面中保持可见。
