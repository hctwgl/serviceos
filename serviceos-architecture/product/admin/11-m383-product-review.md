# M383 项目履约配置产品审查

## 1. 审查结论

M378～M382 已形成有价值的纵向业务主链路：项目履约 Profile、不可变 Revision、校验、预览、发布、建单解析、工单冻结与配置快照均已有真实后端能力和前端入口。

但当前前端状态只能标记为：

- 技术：`RUNTIME_CONNECTED`；
- 前端：`FRONTEND_CONNECTED`；
- 产品：`PRODUCT_REJECTED`；
- 质量：功能与 a11y 部分通过，视觉未批准。

不得标记 `PRODUCT_ACCEPTED` 或“项目工单履约配置产品化完成”。

## 2. 保留的实现

- ProjectFulfillmentProfile / Revision / Manifest / Resolver 领域能力；
- 草稿、校验、发布和生效时间；
- 新旧工单配置版本冻结；
- 工单配置快照 API；
- blockedActions 与表单/资料阻塞原因；
- 项目详情进入“工单类型与履约配置”的产品路径；
- 独立列表、详情、编辑、预览、发布和快照路由；
- 生成的 `@serviceos/core-client` 接入；
- 专用发布流程页面而非普通 Modal。

## 3. P0 产品问题

### 3.1 新增工单类型不是产品流程

当前“新增工单类型”直接硬编码创建 `HOME_CHARGING_SURVEY_INSTALL` 和“标准家充履约方案”，没有让用户选择工单类型、起始模板、复制来源和业务说明。

目标：独立 FormPage/短向导，至少支持：

1. 选择工单类型；
2. 填写方案名称和说明；
3. 从标准方案、复制已有方案或空白方案开始；
4. 确认后创建草稿。

### 3.2 发布流程暴露 Manifest JSON

发布第二步直接显示 Manifest 摘要和完整 `manifestJson`。普通项目配置管理员不应阅读 JSON、Digest 或内部引用。

目标：展示业务运行说明书，包括阶段、责任、表单、必传资料、动作、下一阶段、异常路径、SLA 和客户端支持。Manifest JSON 只能进入开发诊断。

### 3.3 运行预览仍由前端解析 Manifest

前端读取 `manifestJson` 后自行转换 ownerType、targetStage、slaRef 和数量，仍可能把技术编码暴露为业务内容，也形成第二套展示解释逻辑。

目标：服务端 Compile Preview 返回产品化 DTO；前端只负责渲染，不解析运行 Manifest JSON。

### 3.4 编辑器仍以技术文档结构为中心

编辑页直接读写 `documentJson`，使用 `stageCode` 作为主键，并在普通上下文展示 aggregateVersion、原始状态等技术信息；页面文件过大，阶段、表单、资料、动作、审核和 SLA 没有形成独立业务组件和实体选择器。

目标：

- 页面使用服务端结构化 Draft DTO；
- 技术版本进入诊断区；
- 表单、资料、SLA、动作和角色使用真实目录选择器；
- 普通模式不输入资产 ID、stageCode、taskType code、slaRef；
- 拆分为阶段导航、阶段规则、表单绑定、资料要求、动作流转、责任派单、审核、SLA 和异常路径组件。

### 3.5 发布“影响分析”不是实际差异

当前影响页只是固定说明“发布新 Revision、只影响新工单”，没有列出与当前发布版本相比的阶段、表单、资料、动作和 SLA 差异。

目标：服务端提供 Revision Compare/Impact DTO；发布第三步展示真实差异和影响。

### 3.6 操作没有统一使用 allowed-actions

编辑、预览、发布等按钮主要按页面存在性直接显示，尚未形成 Profile 级 allowed-actions 产品模型。

目标：查询详情时返回可执行动作及阻塞原因；前端只呈现服务端允许的编辑、校验、预览、发布、暂停、恢复和停用动作。

## 4. P1 产品问题

- 配置详情通过解析草稿 JSON 展示阶段，而不是使用结构化只读 DTO；
- 责任类型、目标阶段、SLA 引用仍可能显示英文或内部引用；
- 发布版本列表缺少版本状态、版本说明、差异入口和当前生效标识；
- 工单配置快照只显示数量，不能解释“为什么需要某项资料、为什么动作被阻塞”；
- 工单概览新增独立“配置来源”卡片，但应并入摘要/配置来源 Tab，避免继续堆卡片；
- 错误反馈缺少统一问题编号；
- 缺少只读、无权限、冲突恢复、已暂停、已停用等完整产品状态；
- 缺少 1440×1024 与 1280px 的履约配置视觉审查证据。

## 5. 下一里程碑建议

### M384：Admin 产品蓝图接受与 M383 产品审查

- 合并 Admin 产品蓝图；
- 重新标记 M383 的技术、前端、产品和质量状态；
- 建立履约配置页面逐路由验收矩阵；
- 不修改后端运行语义。

### M385：履约配置母版六产品化修正

必须完成：

- 新增工单类型向导；
- 服务端产品化 Preview DTO；
- 服务端 Revision Compare/Impact DTO；
- Profile allowed-actions；
- 结构化 Draft DTO；
- 编辑器组件拆分和实体选择器；
- 发布流程移除 JSON；
- 工单快照解释能力；
- 完整页面状态；
- 视觉和人工产品验收。

M385 通过后，项目履约配置才可标记 `PRODUCT_ACCEPTED`。

### M386：其余五张母版

依次完成运营工作台、用户管理、项目管理、工单中心和工单详情，并逐张产品负责人验收。

## 6. 视觉验收清单

M385 至少提交：

1. 履约配置列表正常/空状态；
2. 新增工单类型向导；
3. 配置详情；
4. 编辑器正常状态；
5. 编辑器校验错误状态；
6. 运行说明书；
7. 发布四步骤；
8. 发布差异与影响；
9. 发布成功；
10. 工单配置快照；
11. 无权限只读；
12. 并发冲突。

所有页面提供 1440×1024；编辑器和列表额外提供 1280px。产品负责人明确批准后才能建立视觉金标。
