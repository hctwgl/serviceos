# Admin 产品定位与设计原则

## 1. 产品定位

ServiceOS Admin 是平台内部统一运营管理后台，用于管理新能源家充服务从收单、审核、派单、预约、勘测、安装、整改、终审到后续处理的完整履约过程，同时承担客户项目、组织资源、配置发布、权限治理和审计追溯。

它服务于长期、日常、高频运营，不是：

- 开发调试控制台；
- API Explorer；
- 数据库管理器；
- 领域对象浏览器；
- 大屏驾驶舱；
- 配置 JSON 编辑器；
- 将全部角色塞入同一页面的万能门户。

## 2. 用户进入页面后应立即获得的答案

任何主要业务页面都应优先回答：

1. 这是什么业务对象；
2. 当前处于什么业务状态；
3. 当前由谁负责；
4. 下一步要做什么；
5. 有什么风险或阻塞；
6. 最晚什么时候完成；
7. 当前允许执行哪些动作；
8. 操作成功后会发生什么。

技术版本、内部 ID、接口路径、幂等键、乐观锁、原始表达式和服务端枚举只能作为诊断或高级配置信息渐进披露。

## 3. 核心设计原则

### 3.1 任务优先

工作台先展示待办、风险、异常和最近处理；详情页先展示当前任务、责任、SLA 和下一步；不能先展示领域对象和技术元数据。

### 3.2 业务语言优先

用户看到“分配服务网点”“提交安装资料”“驳回整改”，而不是 `assign-candidates`、`submit-evidence`、`reject-review`。

### 3.3 标准管理后台优先

用户、项目、角色、网点、师傅、模板等基础对象采用成熟的查询、列表、分页、详情、创建、编辑、启停和变更记录模式。长创建表单不常驻列表首屏。

### 3.4 复杂流程使用专用工作区

终审、改派、履约配置发布、事实更正、批量操作和整改复审使用全页专用流程；普通 Modal 只做短确认。

### 3.5 技术能力不直接决定菜单

后端存在 ReviewCase、EvidenceRevision、ConfigurationAsset，不意味着普通菜单必须使用这些名称。菜单围绕用户任务组织，领域对象作为实现和审计依据。

### 3.6 权限由服务端决定，体验由产品层解释

服务端 Navigation、Capability、Scope 和 allowed-actions 是事实源。前端不按角色名推断权限，但必须把权限和阻塞原因翻译为用户可理解的状态。

### 3.7 一个产品，而不是 Agent 页面集合

所有页面必须共享：

- AppShell；
- 页面头；
- 列表工具栏；
- 表格密度；
- 表单布局；
- 状态语义；
- 空、错、无权限、冲突状态；
- 操作容器；
- 中文业务文案。

### 3.8 先产品验收，再冻结视觉基线

视觉回归用于防止已批准设计被意外破坏，不能把未经审查的半成品自动升级为金标。

## 4. 产品范围

### 4.1 V1 必须覆盖

- 工作台与待办；
- 工单运营；
- 服务履约与调度；
- 审核与整改；
- 车企、项目和项目履约配置；
- 组织、网点、师傅和服务区域；
- 用户、角色、菜单、权限和数据范围；
- 配置模板与发布；
- 操作、登录、配置和系统诊断审计。

### 4.2 不属于普通运营主界面

- API Path；
- UUID 主标识；
- If-Match / Idempotency-Key；
- aggregateVersion / resourceVersion；
- 原始 capability code；
- 原始 Workflow token；
- Manifest JSON；
- SERVICEOS_EXPR_V1 原文；
- correlationId 字段名；
- 英文后端异常。

以上内容进入系统诊断、开发诊断抽屉或高级配置模式。

## 5. 设计成功标准

由不了解 ServiceOS 代码实现的业务运营人员完成以下任务时，不需要开发人员解释技术概念：

- 找到待处理工单；
- 查看当前责任和 SLA；
- 分配网点或师傅；
- 审核资料并发起整改；
- 创建项目；
- 配置某项目某类工单的流程、表单、资料和 SLA；
- 创建用户并授予角色和数据范围；
- 查询一项配置或操作的变更记录。
