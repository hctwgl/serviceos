# Admin 信息架构与菜单

## 1. 一级菜单

```text
工作台
工单运营
服务履约
审核与整改
客户与项目
组织与资源
配置中心
系统管理
审计与监控
```

一级菜单按用户业务任务组织，不按后端模块、表名、聚合或 API 分组。

## 2. 二级菜单

### 2.1 工作台

- 运营工作台
- 我的待办
- SLA 风险
- 运营异常

### 2.2 工单运营

- 工单中心
- 待初审工单
- 待分配工单
- 待预约工单
- 暂停工单
- 已完成工单

除工单中心外，其余项优先实现为服务端受控的预设视图，而不是复制六套列表代码。

### 2.3 服务履约

- 服务任务
- 预约管理
- 网点派单
- 师傅调度
- 勘测管理
- 安装管理

### 2.4 审核与整改

- 审核队列
- 勘测审核
- 安装审核
- 平台终审
- 整改跟踪
- 审核记录

勘测审核、安装审核和平台终审可以是审核队列的预设视图；复杂审核进入专用工作区。

### 2.5 客户与项目

- 车企管理
- 项目管理
- 工单类型与履约配置
- 服务范围
- 接口与字段映射
- 配置发布记录

“工单类型与履约配置”优先从项目详情进入，也可为有权限的项目配置管理员提供跨项目入口。

### 2.6 组织与资源

- 企业组织
- 合作组织
- 服务网点
- 师傅档案
- 技能与资质
- 服务区域

### 2.7 配置中心

- 履约方案模板
- 流程模板
- 表单模板
- 资料模板
- SLA 模板
- 业务日历
- 派单规则
- 配置版本

配置中心面向高级配置管理员。普通运营人员不直接编辑 JSON、表达式、Manifest 或内部资产 ID。

### 2.8 系统管理

- 用户管理
- 组织管理
- 角色管理
- 菜单管理
- 权限管理
- 数据权限
- 授权与委托
- 登录与安全

“企业组织”是业务基础资料；“组织管理”是统一用户中心中的账号归属、权限继承和组织树治理。实现时必须在文案和页面说明中消除歧义。

### 2.9 审计与监控

- 操作日志
- 登录日志
- 配置变更
- 接口记录
- 任务失败
- 系统诊断

系统诊断承载 API Path、correlationId、资源版本、队列 checkpoint 等技术信息，普通业务详情页不直接展示。

## 3. 导航规则

1. 菜单条目来自服务端 Navigation / Page Registry；前端不按角色名称硬编码菜单。
2. 角色影响默认首页、常用入口和排序；真正可见性仍由 capability、scope 和 feature gate 决定。
3. 详情页和专用流程页通常不作为侧栏菜单，但必须保持父菜单选中。
4. 预设视图必须保留当前筛选、返回位置和深链。
5. 未实现或未启用功能不得显示空入口。
6. 同一业务目标不因为状态不同复制多套页面；优先使用保存视图、Tabs 或专用队列。
7. 菜单名称必须使用业务语言，禁止使用 `ReviewCase`、`ConfigurationAsset`、`IntegrationDelivery` 等领域对象名作为普通入口。

## 4. AppShell 信息层级

### 顶栏

- 侧栏折叠；
- 面包屑；
- 全局搜索；
- 当前租户/项目/区域范围的紧凑摘要；
- 数据新鲜度；
- 帮助；
- 用户菜单。

正常新鲜度不占据持续黄色警告区域；只有数据延迟、未知或重建时强化提示。

### 侧栏

- 品牌区；
- 一级/二级菜单；
- 当前项高亮；
- 折叠入口。

侧栏不得展示“菜单来自服务端”“写操作由 Capability 校验”等技术说明。

### 页面头

- 返回入口（详情/流程页）；
- 页面标题；
- 状态；
- 一句话业务说明；
- 次要操作；
- 唯一主要操作。

### 内容区

根据页面类型承载查询、列表、详情、工作台、表单或专用流程，不允许每个页面自行创造完全不同的结构。

## 5. 导航迁移原则

现有页面逐项标记：

- 保留：业务目标正确，只需产品化重构；
- 合并：多个技术队列可合并为一个业务页面的预设视图；
- 重命名：功能保留，改为用户可理解名称；
- 移入配置中心：仅高级配置人员使用；
- 移入审计与监控：技术诊断和接口记录；
- 删除入口：仅开发验证且无正式用户任务，但底层能力可保留。

任何菜单迁移都不得删除真实后端能力或绕过服务端授权。
