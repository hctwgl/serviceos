# ServiceOS Admin 经典专业风视觉与页面基线

## 1. 决策结论

产品负责人已明确批准 ServiceOS Admin 采用：

> **方案 A｜经典专业风**

该方案作为 Admin 后续页面设计、前端实现、人工视觉审查和视觉回归的统一方向。

批准的是以下整体产品语言：

- 稳重、专业、可信赖的企业级后台；
- 浅色工作区、白色导航和标准企业蓝主色；
- 清晰的信息层级和较高但可读的信息密度；
- 以任务、责任、版本、风险、状态和下一步为中心；
- 复杂配置通过目录、画布、属性面板、版本和影响分析组织；
- 不使用营销插画、夸张渐变、巨型留白或花哨装饰；
- 不把数据库字段、JSON、UUID、API 和英文枚举直接暴露给运营人员。

本文件将已批准效果图的视觉和结构转化为可实施规范。概念效果图用于表达产品方向，不替代真实数据契约、Capability、allowed-actions、响应式和可访问性要求。实现时不得机械复制效果图中的示例数字、名称或可能不准确的文字，而应严格使用真实服务端数据和统一中文 Presenter。

## 2. 适用范围

本基线适用于：

- ServiceOS Admin 全部正式页面；
- Admin 中的项目履约配置中心；
- 工作流设计器；
- 任务模板中心；
- 工单中心和工单详情；
- 用户、组织、角色与授权；
- 审核、整改、SLA、异常和审计页面。

Network Portal 和 Technician 客户端共享品牌 Token、状态语义和基础组件，但不强制使用 Admin 的桌面布局。

## 3. 产品气质

### 3.1 必须呈现

- 企业软件的稳定感；
- 结构清楚、可长期使用；
- 复杂业务可解释；
- 信息密度高但不拥挤；
- 主操作明确，次操作克制；
- 关键状态、风险、责任和版本一眼可见；
- 页面像一个完整业务工作区，而不是多个接口卡片拼接。

### 3.2 禁止呈现

- 营销官网式大标题、大插画和渐变背景；
- 深色霓虹、玻璃拟态和过度投影；
- 大量彩色 KPI 卡片；
- 所有卡片同等强调；
- 过度圆角、胶囊化和移动端化；
- 为了“现代”牺牲表格密度和业务比较能力；
- 以英文技术名、原始枚举、JSON 或 UUID 作为主要内容；
- 用空白页面、开发按钮和原始表单代替产品设计。

## 4. Admin 全局视觉系统

### 4.1 推荐语义 Token

具体值在代码 Token 中统一维护，禁止页面自行写死。以下为已批准风格的目标值：

| Token | 建议值 | 用途 |
|---|---:|---|
| `color-brand-primary` | `#1677FF` | 主按钮、选中、链接、当前步骤 |
| `color-brand-primary-hover` | `#4096FF` | 主操作悬停 |
| `color-canvas` | `#F5F7FA` | 应用工作区背景 |
| `color-surface` | `#FFFFFF` | 导航、卡片、表格、面板 |
| `color-surface-subtle` | `#FAFAFA` | 次级区域、只读字段、分组背景 |
| `color-border` | `#E5E7EB` | 卡片、表格、分隔线 |
| `color-text-primary` | `#111827` | 标题和主要正文 |
| `color-text-secondary` | `#4B5563` | 说明、标签和值 |
| `color-text-tertiary` | `#6B7280` | 时间、辅助信息 |
| `color-success` | `#16A34A` | 服务端确认成功、已发布 |
| `color-warning` | `#D97706` | 草稿、风险、即将超时 |
| `color-critical` | `#DC2626` | 阻断、超时、失败、高风险 |
| `color-info` | `#1677FF` | 进行中、提示 |

品牌蓝不能替代状态语义；成功、风险和失败必须保留独立语义颜色、图标和文字。

### 4.2 字体

中文优先使用系统字体栈：

```text
-apple-system, BlinkMacSystemFont, "Segoe UI", "PingFang SC",
"Microsoft YaHei", "Noto Sans CJK SC", sans-serif
```

建议层级：

| 用途 | 字号 | 字重 |
|---|---:|---:|
| 页面标题 | 24px | 600 |
| 核心对象标题 | 20px | 600 |
| 分区标题 | 16px | 600 |
| 正文/表格 | 14px | 400/500 |
| 辅助信息 | 12px | 400 |
| 关键数字 | 20～28px | 600 |

禁止以超大字号制造层级；企业后台层级主要通过位置、间距、字重和分区表达。

### 4.3 圆角、边框和阴影

- 默认卡片圆角：6～8px；
- 输入框、按钮、标签遵循 Ant Design Vue 的统一半径；
- 大面板优先使用细边框，阴影只用于浮层和明确层级；
- 页面内不使用大面积悬浮卡片阴影；
- Drawer、Dropdown、Popover 使用统一 elevation Token；
- 不使用玻璃、发光和拟物效果。

### 4.4 密度

- Admin 默认使用中高密度；
- 标准表格行高：44～48px；
- 紧凑辅助表格可使用 40px，但必须保持可点击性；
- 表单控件高度：32～36px；
- 卡片内边距：16～20px；
- 页面主要区块间距：16px；
- 同一分组内字段间距：12px；
- 页面水平内边距：20～24px。

密度不能通过减小到难以阅读的字体实现。

## 5. 全局应用壳

### 5.1 左侧导航

- 使用白色或极浅色背景；
- 默认宽度 184～208px；
- Logo 位于顶部，展示 `ServiceOS`；
- 一级菜单使用统一线性图标和中文名称；
- 当前一级菜单以浅蓝背景、蓝色图标和文字高亮；
- 展开二级菜单时保持清晰缩进和当前项标记；
- 底部提供“收起菜单”；
- 不使用深色侧栏作为 Admin 默认方案。

一级菜单继续使用已冻结的九项：

1. 工作台；
2. 工单运营；
3. 服务履约；
4. 审核与整改；
5. 客户与项目；
6. 组织与资源；
7. 配置中心；
8. 系统管理；
9. 审计与监控。

### 5.2 顶部栏

高度建议 56px，内容包括：

- 当前项目/范围选择；
- 全局搜索；
- 通知和待办数量；
- 帮助入口；
- 当前用户头像、名称和菜单。

规则：

- 范围选择显示业务名称，不显示 Tenant ID 或 Project ID；
- 全局搜索说明可搜索的业务对象；
- 通知和待办数字必须可下钻；
- 用户菜单不得混入普通业务动作。

### 5.3 页面头

标准结构：

```text
Breadcrumb
PageTitle + Description
PrimaryActions
Optional Version / Scope Controls
```

页面头下方可以放 3～6 个概览块，但概览块必须服务当前任务，不得形成无意义 KPI 墙。

## 6. 通用页面骨架

### 6.1 标准列表页

```text
ApplicationShell
→ Breadcrumb + PageHeader
→ SummaryStrip（可选）
→ FilterBar + Toolbar
→ PrimaryTable
→ ContextDrawer / DetailPanel（可选）
→ Pagination
```

列表表格必须支持业务比较；固定主标识、状态、责任、SLA 和操作列。

### 6.2 标准详情工作区

```text
ApplicationShell
→ Breadcrumb + ObjectHeader + AllowedActionBar
→ SummaryFacts
→ BusinessProgress
→ MainTabs / Workspace
→ RightContextRail
```

首屏必须同时看到：对象、状态、项目、当前阶段、当前任务、责任、SLA、风险、下一步和主操作。

### 6.3 复杂配置工作区

```text
ApplicationShell
→ Project/Version Context
→ Configuration Summary
→ Left Asset Navigation
→ Center Editor / Table / Canvas
→ Right Explain / Version / Impact Panel
→ Validation / Publish Entry
```

复杂配置不得用一个长 JSON 表单解决。

## 7. 已批准核心页面一：项目履约配置中心

### 7.1 页面目标

让项目配置管理员进入一个项目后，在单一工作区中理解并维护：

- 当前项目；
- 当前已发布配置包；
- 草稿配置包；
- 关联工单类型；
- Workflow、Task、表单、资料、SLA、派单、通知和集成映射；
- 版本差异、影响范围、运行快照和发布记录。

### 7.2 页面结构

```text
PageHeader：项目履约配置中心
SummaryStrip：项目 / 已发布版本 / 草稿 / 工单类型数 / 最近发布 / 使用中工单
LeftSubNavigation
Main：配置包概览 + 工单类型配置表
RightRail：配置说明 / 影响分析 + 版本时间线
```

### 7.3 左侧二级导航

- 基础信息；
- 工单类型；
- 工作流；
- 任务模板；
- 表单模板；
- 资料要求；
- SLA 规则；
- 派单策略；
- 通知策略；
- 集成映射；
- 发布记录。

### 7.4 配置包概览

必须展示：

- 配置包名称；
- 适用品牌/客户；
- 适用区域；
- 当前生效版本和状态；
- 当前草稿版本和状态；
- 创建人和时间；
- 发布说明；
- 影响范围；
- “不影响历史已冻结工单”的明确说明。

主操作：

- 编辑草稿；
- 版本对比；
- 发布；
- 查看运行快照。

### 7.5 工单类型配置表

建议列：

- 工单类型；
- 流程模板；
- 表单模板；
- 资料模板；
- SLA；
- 派单策略；
- 通知；
- 状态；
- 最后更新人；
- 最后更新时间；
- 操作。

### 7.6 右侧说明区

`配置说明 / 影响分析` 必须表达：

- 草稿发布后影响的新工单范围；
- 存量工单继续使用冻结版本；
- 主要变更点；
- 配置缺口或风险；
- 当前版本时间线。

### 7.7 禁止

- 直接显示 Manifest JSON；
- 让用户输入 assetId、stageCode、slaRef；
- 只显示配置数量而不显示使用关系；
- 把发布做成一个无差异说明的确认弹窗；
- 将草稿、发布版本和运行快照混为一个状态。

## 8. 已批准核心页面二：工作流设计器

### 8.1 页面目标

以可视化方式定义工单的履约路径、人工任务、系统任务、条件分支、等待事件、异常出口和结束条件，并与 Task 模板、表单、资料、SLA 和责任规则建立清晰引用。

### 8.2 页面三栏结构

#### 左栏：流程目录与节点组件

上半部分：流程目录，例如安装、维修、移机流程及版本状态。

下半部分：可拖拽节点：

- 开始；
- 人工任务；
- 系统任务；
- 条件网关；
- 等待事件；
- 结束。

#### 中间：流程画布

画布必须支持：

- 节点拖拽和连线；
- 条件标签；
- 汇合与分支；
- 当前选中节点；
- 缩放、适配、锁定；
- 自动排版；
- 校验结果；
- 草稿/已发布版本切换；
- 版本对比；
- 试运行/预览。

标准示例：

```text
开始
→ 受理协调
→ 判断是否需要勘测
   ├─ 需要勘测 → 上门勘测 → 勘测审核 → 安装准备
   └─ 无需勘测 → 直接安装准备
→ 上门安装
→ 资料审核
→ 车企回传
→ 完成
```

#### 右栏：节点配置

选中节点后展示：

- 节点名称；
- 节点编码（普通模式只读或隐藏）；
- 节点类型；
- 任务模板；
- 执行角色；
- 分配策略；
- 表单模板；
- 资料要求；
- SLA；
- 前置条件；
- 完成动作；
- 异常出口；
- 业务说明。

底部主操作：保存节点、预览流程。

### 8.3 顶部操作

- 撤销；
- 重做；
- 自动排版；
- 校验；
- 版本对比；
- 发布。

发布只在服务端 allowed-action 允许且校验通过时显示或启用。

### 8.4 校验区

画布底部展示：

- 可达性；
- 缺少结束节点；
- 条件分支完整性；
- 循环/回退路径；
- Task 模板引用；
- 表单/资料绑定；
- SLA 覆盖；
- 未配置异常出口。

错误必须能定位到节点和字段。

### 8.5 禁止

- 以 JSON/脚本编辑器作为主编辑方式；
- 在画布直接展示大段技术编码；
- 允许未校验流程直接发布；
- 修改已发布不可变版本；
- 前端自行解释运行 Manifest；
- 用“任务节点 1、节点 2”代替业务名称。

## 9. 已批准核心页面三：任务模板中心

### 9.1 页面目标

统一管理人工、审核和系统任务的执行角色、分配规则、表单、资料、SLA、允许动作、提醒升级和流程引用关系。

### 9.2 页面布局

Toolbar + Search

RightDetailPanel

### 9.3 概览指标

- 任务模板总数；
- 已发布版本数；
- 草稿模板数；
- 正在被流程引用的模板数；
- 最近更新时间。

### 9.4 任务分类

- 受理类；
- 派单类；
- 预约类；
- 勘测类；
- 安装类；
- 审核类；
- 整改类；
- 回访类；
- 系统任务。

### 9.5 模板表格

建议列：

- 模板名称；
- 任务编码；
- 所属分类；
- 执行角色；
- 分配策略；
- 表单；
- 资料要求；
- SLA；
- 状态；
- 引用流程数；
- 最后更新人；
- 最后更新时间；
- 操作。

### 9.6 详情面板

选中模板后展示：

- 基础信息；
- 执行规则；
- 表单与资料；
- 允许动作；
- 提醒与升级；
- 版本说明；
- 被哪些 Workflow 引用。

“上门安装任务模板”的业务化示例：

```text
执行角色：安装师傅
分配策略：按网点师傅负载均衡
表单模板：安装工单表单
资料要求：安装照片、设备 SN、客户签字、现场环境照片
SLA：48 小时完成
允许动作：接单、开始作业、提交安装、异常上报、申请改约
升级策略：超时提醒网点负责人，持续超时升级总部调度
```

### 9.7 页面操作

- 新建模板；
- 复制模板；
- 版本对比；
- 批量发布；
- 查看引用关系。

版本发布必须校验所有引用关系和影响范围。

## 10. 已批准核心页面四：工单详情

### 10.1 页面目标

将 WorkOrder、Workflow、Task、Appointment、Evidence、Review、SLA 和外部集成组合成一个业务工作区，使运营人员不跨多个技术页面即可理解和处理一张工单。

### 10.2 页面头

展示：

- 工单业务编号；
- 客户和脱敏联系方式；
- 项目；
- 服务产品；
- 当前状态；
- 当前阶段；
- 当前任务；
- SLA 剩余时间；
- 当前责任人；
- 来源系统。

右上主操作由 allowed-actions 驱动，例如：

- 分配网点；
- 改派师傅；
- 发起整改；
- 提交车企审核；
- 更多操作。

### 10.3 主区

#### 履约进度

以冻结 Workflow 的业务阶段展示：

```text
接单成功
→ 受理协调
→ 网点分配
→ 上门勘测
→ 上门安装（当前）
→ 资料审核
→ 车企回传
→ 完成
```

阶段和 Task 必须分别表达；当前阶段下显示当前 Task 和责任。

#### 当前任务卡片

展示：

- 任务名称；
- 责任网点；
- 责任师傅；
- 预约时间；
- 任务说明；
- 服务地址；
- 允许动作；
- 阻塞原因。

#### Tabs

- 基本信息；
- 任务记录；
- 预约与上门；
- 表单资料；
- 审核整改；
- 外部回传；
- 操作日志；
- 配置来源（按需）。

### 10.4 右侧上下文区

- 风险与提醒；
- 当前责任链；
- 外部集成信息；
- 最近时间线。

右侧卡片必须是支持决策的上下文，不得重复主区信息。

### 10.5 表单资料区

必须支持：

- 表单完成度；
- 必填和选填统计；
- 资料缩略图；
- 资料版本和状态；
- 最近审核意见；
- 缺失或不合格项；
- 查看原图和完整审核记录。

## 11. 响应式与分辨率

### 11.1 1440×1024

作为 Admin 关键页面标准评审尺寸。必须完整呈现：

- 主导航；
- 页面标题和主操作；
- 核心工作区；
- 右侧上下文面板；
- 关键表格列。

### 11.2 1280px

- 左侧导航允许收起；
- 右侧详情/上下文可切换为 Drawer；
- 表格允许横向滚动并固定主标识和操作列；
- Workflow 属性面板不得遮挡主画布；
- 主操作始终可见。

### 11.3 小于 1024px

Admin 只保证查询和少量处理；Workflow 设计器、项目配置发布和高密度审核不承诺完整体验，应提示使用桌面端。

## 12. 空、错、只读与冲突状态

每个核心页面必须同时设计：

- 首次加载；
- 局部刷新；
- 空数据；
- 无权限；
- 只读已发布版本；
- 草稿未保存；
- 校验失败；
- 并发冲突；
- 数据投影滞后；
- 服务端失败；
- 发布进行中；
- 发布失败；
- 资源已被停用或删除。

空状态说明原因和下一步；错误说明数据是否保存和如何恢复；并发冲突不得静默覆盖。

## 14. 视觉验收标准

页面达到 `VISUAL_APPROVED` 必须满足：

1. 与本基线的导航、标题、密度、蓝色企业风和组件语言一致；
2. 信息优先级与已批准页面结构一致；
3. 主操作清晰，次操作不抢占；
4. 复杂配置采用目录/画布/属性/版本/影响结构；
5. 普通页面不暴露技术字段；
6. 1440×1024 和 1280px 可用；
7. 正常、空、错误、只读和冲突状态齐全；
8. 使用真实或脱敏业务数据；
9. 产品负责人对实际页面或截图明确批准；
10. 批准后才建立视觉回归金标。
