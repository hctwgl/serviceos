# ADR-021：领域事件发布、版本与消费边界

## Context

ServiceOS 已具备模块化单体、Outbox、Inbox、契约兼容 CI 和多个业务上下文。随着 WorkOrder、Task、Evidence、Review、Dispatch、SLA、Pricing 和 Integration 持续落地，如果没有统一事件边界，容易出现：

- 聚合事务提交后才临时发送消息，导致数据与消息不一致；
- 事件载荷直接暴露数据库实体或车企 DTO；
- 消费者重复处理；
- 同名事件原地修改，破坏历史消费者；
- 用事件代替同步查询或跨模块共享可变状态；
- 把“事件驱动”误解为默认事件溯源。

## Decision

### 1. 当前状态仍由领域表维护

ServiceOS 默认采用状态存储 + 领域事件，不采用事件溯源作为主存储。领域表是当前事实源，事件用于异步副作用、集成、审计、投影和可观测性。

### 2. 事件由事实源聚合产生

只有拥有事实的聚合可以产生对应事件。其他上下文不得伪造同名事件，不得直接修改事实源内部表。

### 3. 业务写入与 Outbox 原子提交

聚合变更和 Outbox 记录必须在同一数据库事务中提交。外部消息发布由独立 Worker 完成。

### 4. 消费者必须幂等

跨模块或跨进程消费者必须使用 Inbox、唯一消费键或等价机制防止重复副作用。重复事件不得重复创建任务、重复计费或重复回传。

### 5. 使用统一事件信封

跨模块事件必须包含：

- eventId；
- eventType；
- eventVersion；
- aggregateType；
- aggregateId；
- tenantId；
- projectId（适用时）；
- occurredAt；
- correlationId；
- causationId；
- traceParent；
- payload。

### 6. 已发布事件不可原地破坏性修改

允许向当前版本增加可选字段。删除字段、改变字段类型、含义或必填性必须发布新版本。事件 Schema 进入契约兼容门禁。

### 7. 不在事件中暴露内部对象图

事件只包含稳定标识和必要快照。禁止直接序列化 ORM Entity、完整 Aggregate、用户权限对象、原始车企 DTO 或敏感文件 URL。

### 8. 同步一致性不通过事件伪装

若命令必须在返回前知道结果，应使用明确的应用服务或同步端口。事件只表达已发生事实，不提供同步 RPC 的假象。

### 9. 外部协议事件先经过 ACL

车企回调、消息或 Webhook 必须先经过 Integration Anti-Corruption Layer，归一化后再转为内部命令或领域事件。外部枚举、字段名和状态不得直接渗透到核心领域。

## Consequences

### Positive

- 数据与事件保持原子一致；
- 消费者可安全重放；
- 事件契约可持续演进；
- 各上下文事实所有权清晰；
- 支持未来拆分进程而不提前微服务化；
- 对审计、BI、通知、SLA 和集成提供统一事实流。

### Negative

- 需要维护事件 Schema、版本和兼容门禁；
- 消费者必须实现幂等和失败恢复；
- 跨聚合最终一致性需要人工接管和补偿设计；
- Outbox 会增加存储、Worker 和监控成本。

## Rejected Alternatives

### A. 直接在事务中调用消息代理

拒绝。数据库提交与消息发送无法天然原子一致，失败窗口不可接受。

### B. 使用数据库实体作为消息体

拒绝。会暴露内部模型、导致循环引用、敏感信息泄露和版本强耦合。

### C. 所有聚合采用事件溯源

拒绝。当前团队、查询模型、迁移成本和业务复杂度不足以证明收益，且会显著提高研发门槛。

### D. 事件 Schema 不版本化，只保持向后兼容约定

拒绝。缺少机器门禁时无法保证长期消费者安全。

### E. 事件消费者直接更新其他上下文的表

拒绝。破坏事实所有权和模块边界。

## Implementation Requirements

- 事件目录以 `serviceos-contracts/src/main/resources/events/` 的已发布 Schema 为命名权威；
- 已发布事件必须有 JSON Schema 或等价机器契约；
- Outbox 写入必须有事务测试；
- 消费者必须有重复投递测试；
- 失败重试必须有最大次数和人工接管；
- 事件日志必须执行 PII 与密钥脱敏；
- 事件发布延迟、失败、积压和死信必须可观测。

## Review Triggers

以下情况需要新 ADR：

- 引入事件溯源；
- 更换 Outbox 存储或消息语义；
- 改变事件版本策略；
- 扩大跨上下文同步事务；
- 引入 exactly-once 平台承诺；
- 将模块内事件升级为公开外部事件。
