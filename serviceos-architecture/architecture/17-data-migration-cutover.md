# 历史数据迁移、双轨与切换设计

## 1. 目标

迁移的目标不是把旧库所有表原样复制到新库，而是把需要继续履约、审计、查询和结算的数据，按 ServiceOS 领域语义转换并保留来源血缘。

同一工单在任一时刻只能有一个写入事实源。双轨用于影子验证和结果核对，不允许两个系统同时对同一工单派单、通知、回传或结算。

## 2. 数据分类

| 类别 | 建议策略 |
|---|---|
| 配置与规则 | 重新建模、评审并发布，不直接抄旧配置表 |
| 组织、用户、网点、师傅 | 清洗主数据、建立外部 ID 映射 |
| 在途工单 | 选择性迁移到明确恢复点，逐单验证 |
| 已完成未结算 | 迁移履约事实、资料版本和结算资格 |
| 已锁定结算 | 只读归档或迁移锁定快照，不重算覆盖 |
| 已关闭历史工单 | 只读历史查询/归档，必要时迁移标准摘要 |
| 图片、视频、文件 | 校验 checksum 后复制/引用，保留来源路径 |
| 日志与审计 | 保留旧系统原始归档，不伪造为新领域事件 |

## 3. 核心对象

| 对象 | 职责 |
|---|---|
| `MigrationPlanVersion` | 范围、映射、批次、验证和切换策略 |
| `SourceSnapshot` | 旧系统某次一致性数据快照和水位 |
| `MigrationBatch` | 一批待迁移记录及执行状态 |
| `MigrationRecord` | 单个来源业务对象的转换结果 |
| `IdMapping` | 旧 ID 到新领域 ID 的稳定映射 |
| `MigrationLineage` | 新对象字段/版本到旧来源位置的血缘 |
| `ValidationResult` | 结构、关系、业务和金额核对结果 |
| `CutoverCohort` | 一组由新系统或旧系统权威处理的业务范围 |
| `CutoverDecision` | Go/No-go、扩大、暂停或回退决策 |
| `RollbackPlan` | 回退水位、副作用清单、反向同步、阻塞、审批和验证的可执行计划 |

## 4. 迁移流水线

```mermaid
flowchart LR
  A["源系统盘点"] --> B["一致性 SourceSnapshot"]
  B --> C["抽取到不可变暂存区"]
  C --> D["标准化/清洗/映射"]
  D --> E["领域校验"]
  E --> F["幂等写入目标模块"]
  F --> G["关系/数量/金额/文件核对"]
  G --> H["业务抽样签署"]
  H --> I["切换 cohort"]
```

源快照、暂存区和转换结果均保留摘要；转换脚本和映射版本进入代码/配置版本管理。

## 5. 源数据盘点

必须识别：

- 表、字段、主外键和枚举；
- 业务事实源与重复冗余字段；
- 时间、时区、金额精度和单位；
- 软删除、历史覆盖和脏数据；
- 文件路径、缺失文件和重复文件；
- 外部车企编号、幂等键和状态同步；
- 工单数量、状态分布、时间范围和峰值；
- 对账、核减、争议和已锁定金额；
- 个人信息、敏感数据和保留期限。

每个旧字段映射为：迁移到标准字段、提取为事实、保留在原始归档、转换为扩展字段，或经批准不迁移。

## 6. IdMapping

外部 ID 映射按源系统、实体类型和源 ID 唯一。重复执行返回相同目标 ID。映射不能依赖可变名称或手机号。

主数据先迁移并冻结映射，再迁移工单和业务对象；发现同一实体多个源 ID 时通过主数据合并记录处理，不随意挑一个覆盖。

## 7. 在途工单

在途迁移必须映射到受控 `MigrationRecoveryPoint`，例如：

- 待派网点；
- 待预约；
- 待勘测提交；
- 待安装提交；
- 待总部审核；
- 待车企回执；
- 待试算。

迁移创建工单、配置包、必要历史摘要、当前阶段/任务和引用对象。不能伪造旧系统从未记录的详细领域事件；使用 `MigratedStateAccepted` 记录来源状态、映射版本和业务签署。

难以安全映射的状态留在旧系统完成，不为追求迁移率强行猜测恢复点。

## 8. 配置锁定

迁移在途工单必须明确绑定哪一个发布配置包。旧项目规则不能仅引用“当前新系统版本”。必要时为历史在途建立迁移专用发布版本，并在完成后退休。

## 9. 资料迁移

- 每个文件保存源路径/对象 ID、源 checksum（如有）、目标 checksum、大小和 MIME；
- 复制后重新计算摘要并抽样打开；
- 缺失、损坏、格式错误和重复文件单独列出；
- 无法恢复完整拍摄元数据时明确标记 `MIGRATED_LEGACY`，不伪造 GPS/拍摄时间；
- 已审核资料迁移审核摘要和原始审计引用，不伪造新审核员决定；
- 敏感文件按新权限重新控制访问。

## 10. 历史结算

- 已锁定：迁移 Statement/Line 快照、财务外部号和来源证据，标记 `MIGRATED_LOCKED`，默认禁止重算替换；
- 未结算：优先生成标准履约事实并影子试算，与旧结果核对；
- 已争议：迁移争议摘要、当前责任和旧系统链接；
- 金额核对到方向、币种、税、明细和总额；
- 不把缺少事实的旧金额反推成虚假履约事实。

## 11. 验证

### 11.1 技术验证

- 记录数、唯一键、非空、枚举和关系完整；
- 时间、Decimal、单位和编码转换；
- 文件数量、总字节、checksum 和可读性；
- 批次幂等和失败重跑；
- 数据范围和脱敏。

### 11.2 业务验证

- 工单生命周期/恢复点；
- 当前网点、师傅、负责人和待办；
- 表单、资料、审核和车企回传版本；
- 履约事实；
- 对上/对下试算和历史金额；
- 关键报表与抽样用户查询。

## 12. 双轨模式

### 12.0 普通领域命令的权威门禁

单一写入权威不仅约束外部副作用。每个会改变工单及其任务、预约、Visit、表单、资料、审核、事实、派单或结算状态的普通领域命令，都必须携带路由层解析出的 `authorityAssignmentId + authorityVersion`，并在领域事务提交前再次校验：

- 当前系统是该工单的 authoritySystem；
- authorityVersion 与数据库当前版本一致；
- 命令模式允许写入，而非 SHADOW/READ_ONLY；
- cohort 和配置未被暂停。

校验失败返回 `WORK_ORDER_AUTHORITY_CHANGED` 或 `SHADOW_WRITE_FORBIDDEN`，不写业务表和 Outbox。后台 Task、批处理、连接器回放和管理员命令同样适用，不能因“内部调用”豁免。

### 12.1 影子读取/计算

旧系统保持权威写入，变更通过 CDC/事件/批次复制到 ServiceOS；ServiceOS 可以执行流程模拟、派单建议和试算，但禁止外部副作用。

### 12.2 单 cohort 切换

按明确项目、区域、外部工单白名单或创建时间选择 cohort。进入 cohort 后：

- ServiceOS 是工单唯一写入事实源；
- 旧系统只读展示或接收单向镜像；
- 只有 ServiceOS 可以通知、派单、回传和生成正式结算；
- 路由决定与时间写入 CutoverCohort。

禁止同一工单双主写入。

## 13. 副作用隔离

影子环境和双轨核对必须有 `SideEffectFence`：

- 禁止调用真实车企写接口；
- 禁止真实短信/微信/推送；
- 禁止激活正式 ServiceAssignment；
- 禁止创建 LOCKED Settlement 或财务交接；
- 输出标记 SHADOW；
- 使用独立凭据、队列、数据库标识和监控。

测试配置错误也不能越过 fence。

### 13.1 权威切换与在途副作用

Fence 的 ALLOW 结果不是可长期复用令牌。OutboundDelivery、NotificationDelivery、ServiceAssignment 激活和正式结算动作必须保存 fenceDecisionId、authorityAssignmentId、authorityVersion 与 fencePolicyVersion，并在真正提交外部请求/激活/锁定的最后一个本地事务中再次校验当前 authorityVersion。

切换操作必须先把旧 authorityVersion 标记为 `DRAINING`，停止产生新副作用，等待或取消尚未越过最终门禁的 delivery/notification/activation/settlement 动作；对已发送但结果未知的动作先完成对账，不能直接切换。排空清单和结果进入 CutoverDecision 证据。只有无未决副作用或经批准进入向前修复时，才发布新 authorityVersion。

执行器在最终校验后到外部调用之间仍可能失去权威，因此外部请求必须使用包含 authorityVersion 的稳定业务幂等键；回执若属于旧 authorityVersion，只能完成原 delivery 的结果对账，不能触发新的业务推进。

## 14. 切换步骤

1. 冻结目标 cohort 的配置和迁移映射；
2. 完成全量快照和增量追平；
3. 验证数据、文件、待办、金额和接口；
4. 停止旧系统接收该 cohort 新写入；
5. 记录水位并执行最终增量；
6. 切换入口路由和副作用所有权；
7. 运行 smoke test；
8. 观察门禁指标；
9. 扩大、暂停或回退。

## 15. 回退

回退条件和窗口在切换前定义。回退不是把数据库恢复到旧备份，而是：

- 停止 ServiceOS 对 cohort 接收新命令；
- 暂停未发送外部副作用并清点已发生副作用；
- 将 ServiceOS 期间产生的权威增量按受控反向映射同步到旧系统；
- 验证旧系统具备继续处理的完整状态；
- 变更路由并记录 CutoverDecision；
- 保留 ServiceOS 数据和审计，不删除失败现场。

如果某外部副作用无法安全反向同步，超过回退窗口后采用向前修复，不冒险双写。

### 15.1 RollbackPlan

PrepareRollback 创建可审计 `RollbackPlan`，状态为：

```text
DRAFT -> ANALYZING -> BLOCKED/READY -> APPROVED -> EXECUTING -> VERIFYING -> VERIFIED
                                      \-> CANCELLED                 \-> FAILED
```

计划必须保存：

- cohort、当前/目标 authority、关联 CutoverDecision；
- 目标 watermark、ServiceOS 权威增量起止水位和摘要；
- 已发生、发送中、结果未知和可取消的副作用逐项清单；
- 每类领域增量的反向映射/同步版本、目标业务键和幂等键；
- 无法反向同步的阻塞项、风险所有人和向前修复决定；
- 执行检查点、反向同步回执、数量/关系/金额/文件核对；
- 申请、审批、执行、验证和最终路由切换证据。

只有状态 `READY` 且审批完成的计划可执行。任一结果未知副作用或阻塞项未处置时保持 `BLOCKED`；执行失败不能自动恢复双主写入。

## 16. 权限与隐私

- 迁移暂存数据最小访问、加密和限期删除；
- 脱敏样本用于开发测试，生产迁移才访问原值；
- 导出/下载/重跑/删除均审计；
- 不迁移已超过合法保留期限且无业务依据的数据；
- 旧系统只读访问保留相同或更严格的数据范围。

## 17. MVP 验收

1. 同一批次重复运行不产生重复对象；
2. 每个目标对象可反查源系统、表/记录、快照和转换版本；
3. 在途工单只映射到已批准恢复点；
4. 文件数量、摘要和异常清单可核对；
5. 已锁定历史金额不被重新计算覆盖；
6. 影子模式无真实副作用；
7. cohort 内同一工单只有一个写入事实源；
8. 切换水位和增量无遗漏/重复；
9. 回退演练能恢复入口与权威状态；
10. 业务、财务和技术签署迁移验证报告。
