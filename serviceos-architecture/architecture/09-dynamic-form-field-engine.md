# 动态表单与字段引擎设计

## 1. 目标

字段引擎建立跨项目稳定的标准字段语义；表单引擎把字段组合到具体任务和页面，配置显隐、必填、校验、默认值和资料联动。

表单不是数据库设计器。客户、车辆、设备、预约、安装和结算等核心业务对象仍使用明确领域模型；动态表单负责项目扩展输入和任务提交快照。

## 2. 核心对象

| 对象 | 职责 |
|---|---|
| `FieldDefinition` | 标准字段编码、语义、类型、敏感级别和业务对象 |
| `FieldMapping` | 外部字段与标准字段的版本化映射 |
| `FormDefinition` | 稳定表单资产身份 |
| `FormVersion` | 已发布、不可变的完整表单 Schema |
| `FieldBinding` | 字段在表单中的标签、位置、权限和规则 |
| `FormRule` | 条件、计算、显隐、必填和校验 |
| `FormDraft` | 用户未提交的可变草稿 |
| `FormSubmission` | 一次不可变提交版本 |
| `SubmissionValidation` | 服务端校验结果 |

## 3. 标准字段目录

标准字段编码以业务语义命名：

```text
workOrder.externalNo
customer.name
customer.mobile
vehicle.vin
charger.serialNumber
survey.conclusion
installation.cableLength
installation.pillarInstalled
```

字段定义至少包含：

- 编码、名称、业务定义和所属对象；
- 类型、单位、精度、枚举目录和格式；
- 是否核心领域字段、项目扩展字段或计算字段；
- 敏感等级、脱敏和导出策略；
- 可用于派单、审核、事实提取、结算和外部回传的标记；
- 废弃和替代字段。

同义的外部名称通过 `FieldMapping` 映射到一个标准字段，不能复制出多个含义相同的内部字段。

## 4. 字段存储策略

采用混合模型：

- 业务身份、关联、生命周期、权限、派单和高频检索字段进入明确类型化模型；
- 项目特有、低频扩展字段保存在通过 Schema 校验的提交文档中；
- 报表、派单和结算需要的扩展值提取为受控投影或履约事实；
- 不采用“所有字段一张 EAV 表”；
- 不为每家车企创建专属宽表。

字段目录决定值类型和语义，JSON/JSONB 只是扩展提交的物理载体，不是无约束数据仓库。

## 5. 表单版本

FormVersion 包含：

```text
formVersionId
formCode
configurationVersion
sections[]
fieldBindings[]
rules[]
dataSchema
uiSchema
requiredEvidenceLinks[]
contentDigest
```

工单任务创建时锁定 `formVersionId`。草稿和提交均保存该版本；发布新表单不改变在途任务。

## 6. FieldBinding

一个标准字段可在不同表单使用不同展示配置：

- 标签、说明、占位、分组和排序；
- 默认值和预填来源；
- 只读、隐藏、可编辑角色；
- 必填、最小/最大、正则、精度；
- 枚举和远程数据源引用；
- 离线是否可用；
- 是否触发资料要求；
- 提交后映射到哪个领域命令。

显示标签不同不改变标准字段语义。

## 7. 规则模型

首期支持受控规则类型：

| 类型 | 示例 |
|---|---|
| 显隐 | 安装立柱=是时显示立柱数量 |
| 条件必填 | 产生增项时收费确认单编号必填 |
| 值校验 | 线缆长度必须 ≥ 0 |
| 跨字段校验 | 勘测结论不通过时原因必填 |
| 计算 | 超出免费长度的提示值 |
| 默认值 | 从工单地址预填区域 |
| 资料联动 | 安装立柱=是时要求立柱照片 |

规则使用声明式表达式和已注册函数，不允许任意浏览器 JavaScript 或直接数据库查询。

## 8. 确定性与依赖图

发布前将规则编译为字段依赖图：

- 禁止循环计算；
- 被引用字段必须在当前或前序上下文可用；
- 条件结果必须类型确定；
- 服务端与客户端使用同一规范和测试样例；
- 服务端校验是权威结果；
- 时间、随机数和外部实时数据不得作为无快照的计算输入。

远程数据源选择保存选中项标识和必要快照，避免后续字典变化改变历史提交含义。

## 9. 预填

预填来源按优先级显式配置：

```text
外部接入标准字段
→ 已确认领域对象
→ 前序任务提交
→ 项目默认值
→ 用户输入
```

预填生成 `prefillSnapshot`，记录来源对象和版本。提交时如果权威来源已经变化，按字段策略：自动刷新、阻止并提示、允许保留旧值或要求人工确认。

## 10. 草稿与提交

### 10.1 FormDraft

草稿可覆盖保存，按用户、任务和表单版本唯一。草稿不是业务事实，不触发流程和结算，可设置保留期限。

### 10.2 FormSubmission

提交后不可变。整改或更正创建新 submission version，并关联替代的旧版本和原因。

```text
DRAFT -> SUBMITTED -> VALIDATED
                    -> INVALID
VALIDATED -> SUPERSEDED（出现新版本）
```

`VALIDATED` 只表示结构和业务校验通过，不等同审核通过。

## 11. 提交事务

表单提交与任务完成分成两个明确动作：

1. `SubmitForm` 创建不可变提交并完成服务端校验；
2. `CompleteTask` 引用已验证 submission 和资料集合，校验任务完成条件。

这样上传大文件或反复保存草稿不会误完成任务。也允许任务完成同时以编排式 API 调用两个命令，但底层事实仍分离。

## 12. 领域映射

表单提交本身不是所有领域对象的事实源。提交成功后由应用服务执行受控映射：

- 用户联系信息更正进入 Customer 命令；
- 实际安装日期和作业结果进入 FieldOperationSubmission；
- 设备 SN 扫码进入资产绑定命令；
- 可计价字段在审核后提取为 FulfillmentFact。

映射保存来源 submission 和字段版本，避免无法追溯。

## 13. 表格、重复组和明细

重复组中的每一行使用稳定 `rowId`，增删改产生明确差异。物料和费用等具有独立生命周期的明细不能只保存在重复组中，表单仅用于采集并映射到对应领域对象。

## 14. 文件字段

文件、照片、视频和签名字段保存 `EvidenceItem`/`EvidenceRevision` 引用，不把对象存储 URL 直接写入表单值。资料的采集元数据、版本和审核由资料域管理。

## 15. 扫码、OCR 和计算结果

- 扫码保存原始值、解析值、设备和时间；
- OCR 结果是机器建议，保存置信度和模型版本；
- 用户确认后的值与 OCR 原始结果分开；
- 计算字段保存表达式版本和输入摘要；
- 关键 SN/VIN 一致性必须由服务端再次校验。

## 16. 离线表单

移动端下载已发布表单包、规则、枚举、示例和摘要。离线草稿保存：

```text
taskId / formVersionId
prefillSnapshotVersion
field values
local revision
deviceId
capturedAt
```

上传时使用设备命令 ID 幂等。若表单版本不再允许提交、任务已改派或预填权威值冲突，服务器返回结构化冲突，不做最后写入覆盖。

离线期间不允许依赖未缓存的远程选项或实时公式；表单发布校验必须标记离线不兼容字段。

## 17. 表单发布校验

- 标准字段存在且类型匹配；
- 规则依赖无循环；
- 所有必填字段在允许角色下可见且可编辑；
- 隐藏字段不能在同条件下必填；
- 离线表单不引用未缓存依赖；
- 资料联动引用有效资料要求；
- 领域映射目标和命令存在；
- 历史样本可通过或差异有明确说明；
- 敏感字段权限和脱敏策略完整。

## 18. MVP 边界

MVP 支持文本、数字、日期、单多选、地址、扫码、重复组、签名引用、资料引用、条件显隐/必填、校验、预填、草稿、不可变提交和基础离线。

通用页面设计器、任意组件脚本、复杂报表设计、跨表事务编排和业务人员自定义数据库延后。
