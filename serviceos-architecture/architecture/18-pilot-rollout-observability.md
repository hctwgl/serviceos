# 试点、灰度发布与生产可观测性

## 1. 目标

试点不是“把系统上线给少数人”，而是用明确 cohort、唯一副作用所有权、可量化门禁和可演练回退，逐步证明 ServiceOS 能承接真实业务。

## 2. 试点选择

建议选择：

- 一个复杂项目验证配置能力；
- 一个高业务量但流程相对稳定的对照项目；
- 具备积极业务负责人、真实接口样本和可核对历史数据；
- 区域/网点规模可控；
- 允许并行影子和灰度；
- 财务可提供对上/对下历史结果。

最终项目由业务负责人决定并登记，不在架构文档中假定具体车企。

## 3. 阶段

```text
P0 实验室：配置样例和自动化测试
P1 历史回放：旧数据事实提取与试算
P2 影子运行：实时复制但无副作用
P3 内部试用：运营/客服处理受控测试工单
P4 小 cohort：真实网点/师傅和真实副作用
P5 扩大 cohort：按门禁逐步增加区域/比例
P6 项目切换：完成权威系统迁移
```

阶段前进需要 Go 决策；发现严重问题可以 Hold、缩小或回退。

## 4. CutoverCohort

cohort 使用确定性规则：

- 项目/品牌；
- 省市区；
- 指定网点/师傅；
- 外部工单白名单；
- 创建时间；
- 稳定哈希百分比。

规则版本发布并保存命中解释。工单创建时锁定 cohort 和权威系统；运行中不因灰度比例变化自动迁移。

## 5. SideEffectFence

环境/运行模式：

- `SHADOW`：禁止真实派单激活、通知、车企写回、正式结算和财务交接；
- `PILOT_ACTIVE`：只有命中 cohort 的 ServiceOS 工单允许副作用；
- `LEGACY_ACTIVE`：旧系统拥有副作用；
- `READ_ONLY`：只读查询。

每个副作用执行前校验：environment、cohort、workOrder authority、operation type 和 fence policy version。拒绝记录审计和指标。

## 6. 发布门禁

### 6.1 功能门禁

- M2–M5 P0 自动化通过；
- 试点配置发布校验和样本回放通过；
- 正常、驳回、改派、超时、回传失败和恢复链路通过；
- 师傅弱网/离线完成演练；
- 权限和敏感资料验证通过。

### 6.2 数据门禁

- 迁移记录数与关系完整；
- 附件 checksum/缺失率在接受范围；
- 在途恢复点逐单或高比例核对；
- 影子事实/金额差异完成分类；
- 未解释金额差异低于业务签署阈值；
- cohort 路由无双写。

阈值必须在试点计划中给出真实数字和签署人，不能只写“基本一致”。

### 6.3 运行门禁

- 接口成功率和延迟；
- 自动派单成功率；
- 任务/Outbox/Delivery/SLA 调度积压；
- 重复副作用为零；
- 严重异常未处理数量；
- 移动端崩溃、同步冲突和上传失败；
- RPO/RTO 演练通过。

## 7. 服务级目标

在掌握真实峰值和业务影响后，试点计划定义 SLI/SLO：

- 工单创建可用性和延迟；
- 核心命令成功率；
- 外部接入/回传处理延迟；
- 自动任务按期执行率；
- 文件 finalize 成功率；
- 数据投影新鲜度；
- 恢复时间和数据丢失目标。

本架构不虚构具体百分比。所有 SLO 有窗口、数据源、排除条件和负责人。

## 8. 可观测性

### 8.1 业务追踪

从 WorkOrder 可追踪：

- 配置包和 cohort；
- 阶段、Task、责任和 SLA；
- 预约、Visit、表单、资料、审核、整改；
- 派单决策和 ServiceAssignment；
- 外部 delivery/attempt/ack；
- 事实、试算、Statement；
- OperationalException 和恢复证据。

### 8.2 技术信号

- structured logs 带 tenant/workOrder/task/correlation/causation；
- metrics 按模块、项目和错误分类控制基数；
- traces 跨 API、消息、流程适配器和外部连接器；
- 审计与技术日志分离；
- 敏感数据不进入日志标签。

## 9. 核心看板

| 看板 | 指标 |
|---|---|
| 接入 | 收单量、重复、冲突、映射错误、延迟 |
| 履约 | 在途、阶段积压、Task 时长、改派、取消 |
| 移动端 | 登录、工作包、离线同步、上传、崩溃 |
| 资料审核 | 提交量、机器校验、驳回、补传轮次、审核 SLA |
| 派单 | 自动成功、无候选、容量冲突、人工覆盖 |
| SLA | 预警、超时、暂停、升级和恢复 |
| 集成 | delivery 积压、attempt、业务拒绝、回执延迟 |
| 试算 | 可计算率、差异率、缺失事实、方向金额 |
| 异常 | P0/P1、新增、最老、处理时间、复发 |

## 10. 告警

告警必须可操作并指向 runbook：

- 工单创建连续失败；
- 无可派比例异常；
- Outbox/Task/Delivery 积压超过门限；
- 同幂等键冲突上升；
- SLA 调度延迟；
- 文件上传/校验异常；
- 车企连接器认证失败；
- 影子环境触发副作用 fence；
- 金额差异或不可计算率异常；
- 权限/敏感导出异常。

告警按严重度去重、抑制和升级，避免通知风暴。

## 11. Runbook

首批 runbook：

- 工单无法创建/配置多命中；
- 自动派单无候选；
- Task/流程不同步；
- 师傅改派 saga 卡住；
- SLA milestone 漏触发；
- 资料上传积压；
- 车企回传持续失败/结果未知；
- 事实冲突/试算不可计算；
- 重复通知/外部副作用疑似发生；
- cohort 路由错误和紧急停写。

每份 runbook 包含检测、影响、权限、诊断、止损、修复、验证、升级和事后记录。

## 12. 试点组织

明确：

- 业务总负责人；
- 品牌/项目经理；
- 客服和审核负责人；
- 网点试点负责人和师傅名单；
- 产品、架构、后端、前端、移动端、测试；
- 数据迁移和财务核对负责人；
- SRE/运维和安全；
- 车企接口联系人；
- Go/No-go 决策人。

上线窗口建立 war room、值班表、升级通讯录和决策日志。

## 13. 灰度扩大

每次扩大记录：当前 cohort、目标 cohort、门禁报告、风险、负责人、开始时间和观察窗口。只对新工单扩大；在途迁移另走 MigrationPlan。

扩大量由业务和容量证据决定，不因时间计划自动推进。

## 14. 暂停与回退触发

示例触发类别：

- 数据丢失或不可恢复不一致；
- 重复派单、回传、结算等外部副作用；
- 权限越权或敏感数据泄露；
- 核心链路不可用超过门限；
- 金额差异无法解释；
- cohort 双写或路由错误；
- 网点/师傅无法完成现场闭环。

具体数值在试点计划中签署。安全和资金类严重事件默认立即停止扩大并评估停写。

## 15. 上线检查表

- 配置/价格/映射版本和审批；
- 数据迁移和增量水位；
- 账号、角色、数据范围和离职清单；
- 车企凭据、IP、签名、限流和回执；
- 对象存储、短信、地图、OCR 等依赖；
- 队列、定时器、Outbox 和死信；
- 备份、恢复、RPO/RTO；
- 看板、告警、runbook 和值班；
- SideEffectFence 和 cohort 路由；
- smoke test、回退演练和签署。

## 16. 试点完成定义

试点不是以“运行了若干天”结束，而是：

1. 目标业务量和典型异常覆盖；
2. M2–M5 P0 场景生产证据或等价演练通过；
3. 无未解释的数据和金额差异；
4. 无重复外部副作用；
5. 业务 SLA 和系统 SLO 达到签署目标；
6. 网点、师傅、客服和财务完成验收；
7. 迁移、运行和安全问题已闭环；
8. 回退/灾备演练通过；
9. 决策人签署扩大或正式切换；
10. 形成试点复盘、遗留债务和下一阶段计划。
