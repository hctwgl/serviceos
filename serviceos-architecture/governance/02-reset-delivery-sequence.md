# ServiceOS 产品重置交付顺序

状态：Accepted  
更新日期：2026-07-21

## 1. 强制顺序

产品重置按以下顺序执行：

1. 冻结旧开发和登记待替换范围；
2. 固化中文友好工程规范、日志异常规范、数据库设计规范和禁止兜底规范；
3. 完成 A+ 产品设计基线、高保真原型和页面状态规格；
4. 从页面反推读模型、命令契约和错误模型；
5. 建立新的前端 Workspace、设计系统和产品语言；
6. 重建 Admin 核心链路；
7. 重建 Network 核心链路；
8. 重建原生 iOS 师傅端工作闭环；
9. 建立新数据库基线和业务场景数据；
10. 删除旧前端、旧 DTO、旧路由、旧数据、旧文档和旧测试夹具。

前一步未形成 Accepted 事实源和验收结论时，后一步不得宣称产品完成。

## 2. 当前 PR 关系

- PR1：产品重置与旧开发冻结；
- PR2：中文友好工程与数据库规范；
- PR3：A+ 产品设计基线与首批核心页面规格；
- PR4：新前端 Workspace 和共享基础；
- PR5：页面读模型、命令契约和错误模型；
- 后续 PR：按完整用户链路交付 Admin、Network 和 iOS。

PR3 可以提前作为 Draft 探索，但在 PR1、PR2 合并前不得转为 Ready，也不得据此开始运行时实现。

## 3. PR 拆分规则

每个实现 PR 必须至少满足一项：

- 提供一个完整、可操作、可验收的用户任务；
- 提供一个完整且可被后续功能直接使用的工程基础；
- 删除一组已经被新实现完全替代的旧能力。

禁止按以下方式拆分产品里程碑：

- 增加一个字段；
- 增加一个筛选项；
- 接通一个接口；
- 新增一个技术对象详情页；
- 只增加截图测试；
- 只把路由注册进导航。

## 4. 产品完成定义

只有同时满足以下条件，功能才可标记为完成：

- 目标角色和完整用户任务明确；
- 页面原型和产品规格已接受；
- 后端契约和数据库模型符合新规范；
- 正常、空、错误、无权限、只读和冲突状态完整；
- 中文文案、日志和数据库描述完整；
- 不存在旧接口、旧路由或静默兜底；
- 功能、视觉、无障碍和真实认证验证通过；
- 产品负责人明确批准。

接口可用、测试通过或页面能打开都不能单独构成产品完成。

## 5. 变更控制

当前仍处于探索期，允许推翻不合理决定，但修改已接受决策必须：

1. 明确指出被替代的事实源；
2. 直接更新或删除旧文档；
3. 同步修改所有调用方和测试；
4. 不保留旧行为兼容；
5. 在 PR 中说明为何新设计更符合产品目标。
