# ServiceOS 产品重置与旧开发冻结决定

状态：Accepted  
生效日期：2026-07-21

## 1. 决定

ServiceOS 当前仍处于产品探索和开发阶段，尚未形成必须兼容的正式生产用户、正式业务数据和稳定外部契约。

从本决定生效起，系统按新的产品目标进行破坏性重构：

- 不兼容旧前端页面、路由和交互；
- 不兼容旧开发数据、演示数据和测试租户；
- 不兼容仅为旧页面存在的接口展示模型；
- 不兼容旧数据库结构和探索期迁移链；
- 不保留旧文档口径；
- 不建立新旧双轨；
- 不编写静默兜底或旧版本适配代码。

Git 历史负责保存过去，当前代码库只表达当前有效的产品和技术事实。

## 2. 当前事实纠正

PR #211（`M385–M452 Admin productization`）已于 2026-07-21 合并进入 `master`。

该批代码可以作为领域能力、接口行为和问题样本的参考，但不构成新产品前端的设计基线。其页面、路由、演示入口、技术导航和字段导向实现统一标记为：

`TO_BE_REPLACED`

不得继续在该实现上追加新的页面字段、筛选项、技术对象入口或逐字段里程碑。

## 3. 立即冻结范围

以下工程停止功能扩展：

- `serviceos-admin-web`；
- `serviceos-network-web`；
- `serviceos-technician-web`；
- 仅为上述旧页面服务的前端适配和展示 DTO；
- 旧的 Demo Data、Golden Path、Portal Stub、Token 登录和技术上下文入口。

允许的修改仅限：

- 阻断安全漏洞；
- 修复阻止主干构建的错误；
- 为新重构提供必要的事实审计；
- 删除旧实现。

任何例外必须在 PR 中写明原因、影响范围和删除时间，不得以“暂时兼容”为理由长期保留。

## 4. 新开发入口

新产品开发必须从最新 `master` 创建独立分支，并按以下顺序推进：

1. 产品重置与冻结；
2. 工程规范和自动化门禁；
3. 产品设计基线和高保真原型；
4. 页面读模型与命令契约；
5. 新前端 Workspace；
6. Admin、Network 和原生 iOS 核心链路；
7. 新数据库基线和场景数据；
8. 删除旧实现。

不得再使用按单个字段、筛选项或接口编号推进的 Mxxx 产品化里程碑。

## 5. 师傅端决定

正式师傅端采用原生 iOS：

- SwiftUI 页面和导航；
- Swift Concurrency；
- 本地数据库；
- 离线命令队列；
- 后台上传；
- 相机、定位和设备安全；
- 杀进程恢复与同步冲突。

Technician H5 只作为在线参考、联调和有限应急入口，不与原生 iOS 承诺功能等价。Android 当前不建设，也不预留空工程或兼容层。

## 6. 禁止事项

新代码中禁止出现以下设计意图：

- `legacy`；
- `compat`；
- `fallback`；
- `deprecated`；
- 旧字段别名；
- 新旧接口双读或双写；
- 新接口失败后调用旧接口；
- 名称缺失时显示内部 ID；
- 中文标签缺失时显示英文枚举；
- 捕获异常后返回空集合或默认成功；
- 把演示和调试入口放入正式产品导航。

合理的空状态、错误状态、无权限状态、只读状态和并发冲突状态不属于兜底，必须明确设计并真实呈现。

## 7. 完成条件

本冻结决定完成落地后：

- 旧前端不再新增产品功能；
- 新 PR 必须声明是否触及待替换区域；
- PR #214 及后续产品设计只以新的 A+ 产品基线为目标；
- 所有实现 PR 必须能够说明其最终用户、完整任务和产品验收方式；
- 未经产品设计批准，不进入页面实现。
