# 领域策略目录

领域策略承载跨实体、跨配置或依赖上下文判断的业务规则。策略不是工具类，不替代聚合不变量，也不允许成为可以随意调用数据库和网络的“万能服务”。

## 1. 分类

### 1.1 聚合内不变量

直接写在聚合中，例如：

- 已取消工单不能继续完成普通任务；
- 已锁定资料不能被普通角色覆盖；
- 审核决定必须引用明确的资料版本。

这类规则不单独抽成 Policy。

### 1.2 领域策略

当规则依赖多个输入、配置版本或可替换算法时建模为 Policy，例如：

- 重复外部订单判断；
- 配置包选择；
- 派单候选评分；
- SLA 暂停资格；
- 资料动态必传；
- 计价事实资格。

### 1.3 应用编排策略

决定调用顺序、重试和人工接管，不拥有核心业务事实。例如：

- 外部接口临时错误重试；
- 自动任务失败后创建人工任务；
- 回调重复的幂等处理。

## 2. WorkOrder 策略

### 2.1 DuplicateExternalOrderPolicy

输入：客户端、外部订单号、请求摘要、配置包引用。

结果：

- `NEW`：首次接收；
- `REPLAY`：同摘要、同配置版本；
- `PAYLOAD_CONFLICT`：同外部订单不同摘要；
- `BUNDLE_CONFLICT`：同外部订单不同配置版本。

策略不得仅通过“数据库是否存在”返回布尔值，必须给出可审计判定原因。

### 2.2 ConfigurationBundleSelectionPolicy

按以下维度解析唯一可用配置包：

```text
client + brand + project + serviceProduct + region + effectiveTime
```

不变量：

- 必须解析到已发布、在生效期内的精确版本；
- 多个同优先级配置同时命中时失败关闭；
- 不允许自动选择 `latest`；
- 工单创建后保存精确引用和 checksum。

### 2.3 WorkOrderClosurePolicy

关闭前检查：

- 所有必需阶段已完成或被有权取消；
- 不存在未决强制整改；
- 必需回传已成功或有正式豁免；
- 结算前置条件按项目配置满足；
- 关闭类型与操作者权限匹配。

## 3. Task 与 Workflow 策略

### 3.1 TaskReadinessPolicy

根据前置任务、网关条件、必要资料和外部回执判断任务是否可进入 READY。

### 3.2 AssigneeResolutionPolicy

解析：

- 责任角色；
- 品牌与区域作用域；
- 候选人；
- 自动分配或人工领取模式；
- 无候选人时的兜底处理。

解析结果必须快照化，避免组织数据变化后历史任务责任不可追溯。

### 3.3 RetryAndInterventionPolicy

对自动任务定义：

- 可重试错误码；
- 最大次数；
- 退避策略；
- 超限后人工任务类型；
- 升级对象；
- 是否阻断主流程。

## 4. Evidence 与 Review 策略

### 4.1 EvidenceRequirementPolicy

根据品牌、桩型、勘测结论、安装字段和增项信息动态生成资料要求。

输出必须包含：

- 资料编码；
- 是否必传；
- 数量约束；
- 采集方式；
- 水印/GPS/OCR/视频规则；
- 规则版本与命中解释。

### 4.2 EvidenceAcceptancePolicy

综合自动校验与人工审核决定资料是否可进入快照。自动校验通过不等于人工审核通过。

### 4.3 ReviewRoutingPolicy

决定审核任务由谁处理，以及是否需要总部、车企或多级审核。首期支持单审核人，但模型不锁死未来复审。

### 4.4 ForceApprovalPolicy

强制通过必须同时满足：

- 操作者具备显式能力；
- 提供标准原因和补充说明；
- 保存授权依据；
- 产生独立事件；
- 不伪装成普通通过。

## 5. Dispatch 策略

### 5.1 CandidateEligibilityPolicy

硬过滤顺序：

1. 指定网点与人工覆盖合法性；
2. 黑名单；
3. 停派状态；
4. 品牌、区域和业务能力；
5. 必需资质；
6. 在途容量；
7. 其他项目级硬约束。

任一硬过滤失败必须留下原因代码。

### 5.2 CandidateScoringPolicy

可配置因子包括：

- 签约比例偏差；
- 当前在途量；
- 剩余产能；
- 历史履约率；
- SLA 表现；
- 网点评分；
- 临时额度调整。

评分结果必须保存因子值、权重、总分和策略版本。

### 5.3 CapacityReservationPolicy

派单决定成功前先原子预占容量；竞争失败时重新计算，不允许仅靠最终写入 `network_id` 解决并发。

## 6. SLA 策略

### 6.1 SlaDefinitionSelectionPolicy

按车企、业务类型、风险等级、节点和配置版本选择 SLA 定义。

### 6.2 SlaPausePolicy

判断用户原因、物业、电力报装、缺货等是否可暂停，以及是否需要证明。暂停必须记录原因、开始时间和操作者。

### 6.3 SlaEscalationPolicy

根据预警阈值、超时级别和作用域解析升级角色。当前默认升级品牌经理，但必须配置化。

## 7. Pricing 与 Settlement 策略

### 7.1 BillableFactPolicy

判断履约事实是否可计价：

- 来源资料或任务是否有效；
- 必要审核是否通过；
- 车企驳回是否冻结资格；
- 是否已被新版本事实替代。

### 7.2 PricingPlanSelectionPolicy

按方向、车企、项目、区域、业务类型和发生时间选择精确价格版本。

### 7.3 SettlementAdjustmentPolicy

调整必须引用原费用项、调整原因、授权依据和金额差异，不允许覆盖原始计算结果。

## 8. Integration 策略

### 8.1 ErrorNormalizationPolicy

把车企协议中的 HTTP、业务码、部分成功和回调状态归一为：

- SUCCESS；
- RETRYABLE_FAILURE；
- BUSINESS_REJECTION；
- PARTIAL_SUCCESS；
- SECURITY_REJECTION。

### 8.2 DeliveryRetryPolicy

只重试明确的临时错误；业务拒绝直接转人工，不无限重试。

### 8.3 CallbackConflictPolicy

重复回调相同载荷安全重放；同业务键不同载荷进入冲突处理，不静默覆盖。

## 9. 策略实现边界

策略可以：

- 接收领域对象快照和值对象；
- 读取已发布配置；
- 返回明确决策和解释；
- 通过接口依赖必要的只读能力。

策略禁止：

- 直接提交事务；
- 随意修改多个聚合；
- 直接调用车企网络接口；
- 返回无法解释的裸布尔值；
- 读取未版本化的“当前规则”影响历史工单。

## 10. 测试要求

每个策略至少覆盖：

1. 正常命中；
2. 边界条件；
3. 冲突和无匹配；
4. 规则版本锁定；
5. 决策解释完整性；
6. 并发场景（涉及容量或唯一性时）；
7. 不同车企配置隔离。
