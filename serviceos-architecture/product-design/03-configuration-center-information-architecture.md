# 配置中心信息架构与页面关系

状态：Draft for Product Review  
适用范围：ServiceOS Admin 配置中心  
视觉基线：A+ 经典专业任务型

## 1. 重整目的

配置中心不是若干技术对象页面的集合，也不是把流程、节点、规则、表单和版本全部平铺在左侧菜单中。

配置中心应帮助项目配置人员完成一个明确结果：

> 为某个项目建立完整、可校验、可预览、可发布、可追溯的履约配置。

本文件重新定义：

- 配置中心入口；
- 项目配置工作区；
- 左侧业务导航；
- 各模块默认首页；
- 列表、详情和设计器之间的关系；
- 模块版本与项目发布版本的边界；
- 已废弃的重复原型和菜单。

## 2. 总体层级

配置中心采用三级结构。

### 2.1 第一级：Admin 主导航

Admin 左侧主导航中只保留一个入口：

- 配置中心

配置中心不在 Admin 主导航中继续展开流程、表单、SLA、节点或版本等技术对象。

### 2.2 第二级：配置中心首页

进入配置中心后，默认展示“项目配置列表”。

配置中心首页负责回答：

- 哪些项目尚未配置；
- 哪些项目存在未发布草稿；
- 哪些项目有阻塞发布的问题；
- 哪些项目最近完成发布；
- 当前用户可以配置哪些项目；
- 下一步最需要处理什么。

首页主要页面：

1. 项目配置列表；
2. 待处理问题视图；
3. 最近发布记录；
4. 配置健康概览。

点击项目后进入项目配置工作区。

### 2.3 第三级：项目配置工作区

项目配置工作区始终具有明确项目上下文。

页面顶部固定展示：

- 当前项目；
- 项目状态；
- 当前生效版本；
- 当前草稿版本；
- 配置完整度；
- 校验状态；
- 最近保存时间；
- 当前首要动作。

用户可以切换项目，但切换前必须处理未保存修改。

## 3. 项目配置工作区最终左侧菜单

项目配置工作区的左侧菜单统一采用浅色样式，仅保留以下业务模块：

1. 配置概览；
2. 服务范围；
3. 流程设计；
4. 表单与资料；
5. SLA 与预约；
6. 角色与责任；
7. 网点与资源；
8. 审核与整改；
9. 集成与回传；
10. 版本与发布。

不得在左侧菜单中出现：

- 流程列表；
- 流程版本；
- 节点设计；
- 流转规则；
- 表单列表；
- 表单设计器；
- 资料要求详情；
- SLA 指标详情；
- 校验结果；
- 影响分析；
- 发布确认。

这些内容属于模块内部页面、页签、抽屉或流程步骤，不是左侧一级业务模块。

## 4. 配置中心首页

### 4.1 项目配置列表

默认首页采用标准列表模式。

列表至少展示：

- 项目名称；
- 客户；
- 项目状态；
- 配置完整度；
- 当前生效版本；
- 草稿状态；
- 阻塞问题数量；
- 最近修改人；
- 最近修改时间；
- 最近发布时间；
- 当前首要动作。

主要动作：

- 开始配置；
- 继续配置；
- 处理问题；
- 查看生效配置；
- 查看发布记录。

不得从首页直接编辑流程节点、表单字段或 SLA 规则。

### 4.2 首页辅助视图

项目配置列表可切换以下业务视图：

- 全部项目；
- 待首次配置；
- 有未发布草稿；
- 校验未通过；
- 等待发布；
- 最近已发布。

这些是同一列表的预设视图，不建立独立菜单页面。

## 5. 配置概览

配置概览是进入项目配置工作区后的默认页面。

必须展示：

- 配置完整度；
- 十个模块的完成状态；
- 当前生效版本；
- 草稿摘要；
- 阻塞错误；
- 重要警告；
- 变更摘要；
- 最近操作记录；
- 推荐下一步。

配置概览不直接承载复杂编辑。

点击模块卡片进入对应模块，点击错误进入具体问题位置。

## 6. 服务范围

服务范围负责定义项目提供什么服务，以及服务适用的业务范围。

内部页面：

1. 服务项目与业务类型；
2. 服务区域与覆盖要求；
3. 服务适用条件；
4. 服务范围校验；
5. 变更影响。

服务区域采用已确认的高德行政区规则和地图交互，但不把行政区数据结构暴露给普通运营用户。

## 7. 流程设计

### 7.1 默认首页：流程列表

点击“流程设计”后默认进入流程列表。

流程列表展示：

- 流程名称；
- 业务场景；
- 状态；
- 当前生效版本；
- 草稿版本；
- 使用中的工单数量；
- 校验问题；
- 最近发布人和时间；
- 最近修改人和时间。

主要动作：

- 新建流程；
- 复制流程；
- 进入设计；
- 查看生效版本；
- 停用流程；
- 查看引用关系。

### 7.2 流程设计工作区

选择流程后进入统一流程设计工作区。

工作区内部包含：

1. 流程画布；
2. 流转规则；
3. 节点配置；
4. 关联表单；
5. 权限与处理人；
6. SLA 规则；
7. 校验与影响分析；
8. 版本历史。

它们通过页签、右侧属性面板、底部面板或抽屉组织，不进入左侧菜单。

### 7.3 节点配置方式

点击画布节点后，在右侧属性面板编辑：

- 节点名称；
- 节点类型；
- 责任角色；
- 完成条件；
- 关联表单；
- 必要资料；
- SLA；
- 可执行动作；
- 跳过和回退规则；
- 业务说明。

不再建立独立“节点设计”菜单或多张重复节点页面。

### 7.4 流转规则

点击连线后，在右侧属性面板编辑流转规则。

每条流转规则必须包含：

- 来源节点；
- 目标节点；
- 触发动作；
- 进入条件；
- 条件组合关系；
- 表单完整性要求；
- 资料完整性要求；
- 可执行角色；
- 状态变化；
- 自动分配；
- 通知动作；
- 超时处理；
- 是否允许撤回或回退；
- 多规则命中优先级；
- 无规则命中时的明确错误。

“流线规则”不是正式术语，统一使用“流转规则”。

## 8. 表单与资料

### 8.1 默认首页

点击“表单与资料”后默认进入管理首页，内部提供两个一级页签：

- 表单模板；
- 资料要求。

不得默认打开字段编辑器。

### 8.2 表单模板列表

表单模板列表展示：

- 表单名称；
- 适用阶段；
- 状态；
- 当前生效版本；
- 草稿版本；
- 字段数量；
- 资料要求数量；
- 被引用流程；
- 校验问题；
- 最近修改人和时间。

点击表单名称先进入表单详情。

### 8.3 表单详情

表单详情默认只读，展示：

- 基本信息；
- 当前版本；
- 字段摘要；
- 资料要求摘要；
- 引用关系；
- 校验结果；
- 版本历史；
- 操作记录。

点击“编辑草稿”或“新建表单”后才进入表单设计器。

### 8.4 表单设计器

表单设计器采用独立工作区：

- 左侧：字段组件库；
- 中间：表单结构与实时预览；
- 右侧：字段属性；
- 顶部：保存草稿、预览、校验、影响分析；
- 抽屉：显示条件、校验规则、字段权限和资料要求。

### 8.5 资料要求

资料要求具有独立列表和详情，管理：

- 资料名称；
- 资料类型；
- 必填条件；
- 数量限制；
- 文件大小限制；
- 拍摄或上传要求；
- 示例和说明；
- 适用表单；
- 适用流程节点；
- 脱敏要求；
- 保留期限；
- 访问权限。

## 9. SLA 与预约

点击后进入统一规则中心，内部页签：

1. SLA 指标；
2. 时间规则；
3. 触发条件；
4. 通知与升级；
5. 预约规则；
6. 校验与影响。

SLA 指标列表是默认页，不直接打开指标编辑器。

模块版本历史放在模块内部，不成为左侧菜单。

## 10. 角色与责任

本模块只管理项目履约中的业务责任，不替代系统级用户和 RBAC 管理。

内部页面：

1. 项目参与角色；
2. 责任矩阵；
3. 节点处理人规则；
4. 可执行动作；
5. 数据可见范围；
6. 责任冲突校验。

用户账号、系统角色和租户级授权仍属于系统管理。

## 11. 网点与资源

内部页面：

1. 网点覆盖配置；
2. 网点准入条件；
3. 师傅能力要求；
4. 服务资源要求；
5. 产能约束；
6. 覆盖与资源校验；
7. 变更影响。

项目默认全国服务，具体覆盖由网点按项目和业务类型配置省级或地级行政区域。

## 12. 审核与整改

内部页面：

1. 审核策略；
2. 审核任务规则；
3. 审核结论；
4. 整改类型；
5. 整改时限；
6. 整改资料要求；
7. 复核规则；
8. 通知和升级。

审核与整改规则可以被流程节点引用，但不在流程画布中重复维护完整规则内容。

## 13. 集成与回传

本模块使用业务协同语言，不把 Envelope、Inbox、Outbox 等技术对象作为普通导航。

内部页面：

1. 入站工单规则；
2. 外部字段映射；
3. 结果回传规则；
4. 状态同步；
5. 失败处理；
6. 对接校验；
7. 最近协同记录。

技术诊断进入独立受控诊断工具，不进入普通项目配置页面。

## 14. 版本与发布

版本与发布负责项目级完整配置 Bundle，不替代各模块内部的版本历史。

### 14.1 模块版本

流程、表单、SLA 等模块内部可以查看自身版本历史和差异。

模块版本用于回答：

- 这个流程或表单发生了什么变化；
- 哪个版本正在被草稿引用；
- 哪些版本已经失效。

### 14.2 项目版本

项目级“版本与发布”负责：

1. 草稿总览；
2. 跨模块完整性校验；
3. 版本对比；
4. 影响分析；
5. 发布确认；
6. 发布进度；
7. 发布结果；
8. 历史发布记录。

项目发布必须形成不可变配置版本，并冻结其引用的流程、表单、SLA、角色、资源和审核规则版本。

不得只发布单个流程后让项目进入引用不一致状态。

## 15. 页面层级规则

统一遵守以下规则：

- 左侧菜单只放业务模块；
- 模块点击后先进入列表或概览；
- 详情页通过列表进入；
- 编辑器通过“新建”或“编辑草稿”进入；
- 节点和连线属性通过面板编辑；
- 校验、影响分析和发布采用明确步骤；
- 版本历史优先作为模块页签或项目发布页面；
- 不把列表、详情、编辑器和版本页同时平铺为菜单；
- 面包屑必须反映当前位置；
- 返回时保留用户原有筛选、分页和滚动位置。

## 16. 废弃的原型与菜单

以下内容统一标记为废弃，不进入正式规格或实现：

- 深色配置中心左侧导航；
- `流程设计 / 流程列表 / 流程版本 / 节点设计 / 流线规则` 同级菜单；
- 独立节点设计首页；
- 多张重复的节点详情和节点编辑页面；
- 配置中心首页直接展开表单字段编辑器；
- 表单列表和表单编辑器同时塞入同一首页；
- 单个模块页面直接承担整个项目发布；
- 把技术配置 JSON、内部 ID 或英文枚举作为运营配置内容。

已生成的相关图片仅作为探索过程记录，不构成产品事实源。

## 17. 后续设计顺序

配置中心后续设计按以下顺序推进：

1. 配置中心首页和项目配置列表；
2. 项目配置概览；
3. 流程列表和统一流程设计工作区；
4. 表单模板列表、表单详情和表单设计器；
5. 资料要求列表和详情；
6. SLA 与预约；
7. 角色与责任；
8. 网点与资源；
9. 审核与整改；
10. 集成与回传；
11. 版本与发布。

在信息架构和页面规格未确认前，不继续批量生成高保真图片。

## 18. 验收标准

- 用户可以从项目配置列表进入明确项目上下文；
- 左侧菜单没有列表、版本、节点和规则等混合层级；
- 每个模块默认先展示列表或概览，不直接进入编辑器；
- 流程节点和流转规则在同一流程工作区维护；
- 表单列表、详情和设计器职责明确分离；
- 模块版本和项目发布版本职责清楚；
- 所有页面使用统一浅色导航和 A+ 视觉语言；
- 不再根据图片临时增加菜单和页面；
- 后续页面规格、读模型和实现均以本文件为配置中心结构基线。
