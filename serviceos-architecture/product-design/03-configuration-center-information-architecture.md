# 项目管理与履约配置的信息架构

状态：Accepted  
适用范围：ServiceOS Admin 项目管理与履约配置  
视觉基线：A+ 经典专业任务型

> 说明：文件名保留历史名称，但“配置中心”不再是正式产品概念，也不作为 Admin 一级菜单。

## 1. 已确认领域关系

ServiceOS 当前阶段采用以下关系：

```text
客户
└── 项目
    ├── 项目团队与区域分工
    ├── 履约配置
    │   └── 一个履约工作流
    └── 项目工单
```

关键边界：

- 项目是实际履约的业务对象；
- 项目团队与区域分工是项目运营主数据；
- 履约配置是项目的规则版本；
- 一个项目当前只有一个履约工作流；
- 工单创建时绑定履约配置版本，并保存项目岗位人员快照；
- 项目团队调整不需要重新发布履约配置；
- 区域分工变化不能静默修改历史工单人员。

## 2. 取消“配置中心”一级菜单

Admin 主导航不设置“配置中心”。

项目的权威入口为：

```text
客户与项目
└── 项目列表
    └── 项目详情
```

项目列表是项目主数据的唯一列表，不另建“项目配置列表”。

跨项目配置待办放入 Admin 工作台，例如：

- 待首次配置；
- 存在未发布草稿；
- 校验失败；
- 等待发布；
- 发布失败；
- 最近完成发布。

点击待办后直接进入对应项目详情的“履约配置”。

## 3. 项目详情信息架构

项目详情至少包含：

1. 项目概览；
2. 服务与合同；
3. 项目团队与区域分工；
4. 履约配置；
5. 项目工单；
6. 操作记录。

页面顶部持续展示项目名称、客户和项目状态。

### 3.1 项目团队与区域分工

该页签用于管理：

- 项目成员；
- 客服经理、项目经理、项目助理三个固定岗位；
- 项目内行政区域与岗位人员的对应关系；
- 人员匹配预览；
- 分工变更记录。

该模块不属于履约配置，不使用履约配置草稿和发布流程。

### 3.2 履约配置

该页签用于管理可版本化、可校验、可整体发布的项目履约规则。

页面顶部必须持续展示：

- 当前生效配置版本；
- 当前草稿版本；
- 配置完整度；
- 校验状态；
- 最近保存时间；
- 当前首要动作。

## 4. 履约配置工作区

工作区左侧采用统一浅色导航，只保留以下业务模块：

1. 配置概览；
2. 服务范围；
3. 流程设计；
4. 表单与资料；
5. SLA 与预约；
6. 网点与资源；
7. 审核与整改；
8. 集成与回传；
9. 版本与发布。

“角色与责任”不再是履约配置模块。

项目人员调整属于“项目团队与区域分工”，不能混入履约配置版本。

## 5. 配置概览

配置概览负责回答：

- 当前项目能否开始接单和履约；
- 哪些规则尚未完成；
- 当前生效配置是什么；
- 是否存在未发布草稿；
- 哪些错误阻止发布；
- 发布后会影响什么；
- 用户下一步应处理什么。

必须展示：

- 配置完整度；
- 八个规则模块的完成状态；
- 生效版本摘要；
- 草稿变更摘要；
- 阻塞错误；
- 重要警告；
- 最近操作；
- 推荐下一步。

配置概览不直接承载复杂编辑。

## 6. 服务范围

服务范围定义项目提供什么服务、服务适用于什么区域和业务条件。

内部内容包括：

- 服务项目和业务类型；
- 服务行政区域；
- 适用条件；
- 范围校验；
- 变更影响。

服务范围回答“项目在哪里提供什么服务”。

项目团队的区域分工回答“这些区域分别由谁负责”。

两者不得合并成一张表，也不得各自维护第二套行政区数据。

## 7. 流程设计

### 7.1 一个项目一个工作流

点击“流程设计”后直接进入该项目唯一履约流程工作区。

当前阶段不提供：

- 流程列表；
- 项目内多个流程；
- 复制流程；
- 独立流程版本；
- 独立流程发布；
- 公共流程模板。

### 7.2 统一流程设计工作区

工作区包含：

1. 流程画布；
2. 节点配置；
3. 流转规则；
4. 关联表单和资料；
5. 工单处理对象；
6. SLA 规则；
7. 校验与影响提示。

点击节点编辑节点属性，点击连线编辑流转规则，不建立独立节点页或规则页。

### 7.3 工单处理对象

节点和流转动作只允许引用当前业务已经存在的处理对象：

- 当前工单客服经理；
- 当前工单项目经理；
- 当前工单项目助理；
- 当前工单所属网点；
- 当前工单已分配师傅；
- 明确人工选择的处理人；
- 系统任务。

流程不创建项目角色，也不实现行政区域人员匹配。

客服经理、项目经理和项目助理由“项目团队与区域分工”在工单创建或地址确认时匹配，并写入工单人员快照。

### 7.4 节点配置

节点至少包含：

- 节点名称；
- 节点类型；
- 处理对象；
- 业务目的；
- 进入前提；
- 完成条件；
- 关联表单；
- 必要资料；
- SLA 或时限；
- 可执行动作；
- 跳过、撤回和回退规则；
- 外部任务失败处理。

### 7.5 流转规则

每条连线必须明确：

- 来源节点；
- 目标节点；
- 触发动作；
- 进入条件；
- 条件组合关系；
- 表单和资料完整性要求；
- 可执行对象；
- 状态变化；
- 通知动作；
- 超时处理；
- 撤回和回退规则；
- 优先级；
- 无规则命中时的中文错误。

“流线规则”不是正式术语，统一使用“流转规则”。

## 8. 表单与资料

管理当前项目履约过程中使用的表单和资料要求。

页面关系：

```text
表单列表
→ 表单详情（默认只读）
→ 新建表单或编辑草稿
→ 表单设计器
```

同时提供资料要求列表和详情。

表单和资料属于当前项目履约配置，不建设跨项目公共模板库。

## 9. SLA 与预约

一个项目当前可以拥有：

- 多个 SLA 指标；
- 一套预约政策；
- 一套服务日历。

SLA 通知和升级对象可以引用当前工单客服经理、项目经理和项目助理，但不能自行实现区域人员匹配。

所有 SLA、预约和日历规则随履约配置整体发布。

## 10. 网点与资源

内部内容包括：

- 项目服务覆盖；
- 网点准入条件；
- 师傅能力要求；
- 服务资源要求；
- 班次与产能约束；
- 覆盖与资源校验；
- 变更影响。

“网点与资源”负责履约执行资源，不负责客服经理、项目经理和项目助理的区域分工。

## 11. 审核与整改

内部内容包括：

- 审核策略；
- 审核任务规则；
- 审核结论；
- 整改类型；
- 整改时限；
- 整改资料要求；
- 复核规则；
- 通知和升级。

审核人、整改负责人或通知对象可以引用工单人员快照、所属网点或已分配师傅，不能创建通用项目角色。

## 12. 集成与回传

使用业务协同语言，不把 Inbox、Outbox、Envelope 等技术对象作为普通导航。

内部内容包括：

- 入站工单规则；
- 外部字段映射；
- 结果回传规则；
- 状态同步；
- 失败处理；
- 对接校验；
- 最近协同记录。

技术诊断进入独立受控诊断工具。

## 13. 版本与发布

### 13.1 履约配置整体版本

一个履约配置版本整体包含：

- 服务范围；
- 唯一工作流；
- 表单与资料；
- SLA 与预约；
- 网点与资源规则；
- 审核与整改规则；
- 集成与回传规则。

不包含：

- 项目成员；
- 区域人员分工；
- 工单人员快照；
- 实时网点班次和师傅可用性。

### 13.2 草稿与生效版本

每个项目最多存在：

- 一个当前生效履约配置版本；
- 一个正在编辑的履约配置草稿。

生效版本不可修改。发布后形成新的不可变版本。

### 13.3 发布步骤

1. 保存草稿；
2. 跨模块完整性校验；
3. 业务预览；
4. 新旧版本业务差异；
5. 影响分析；
6. 发布说明；
7. 最终确认；
8. 发布；
9. 发布结果。

不得跳过校验和影响分析直接发布。

### 13.4 工单绑定

工单创建时：

- 绑定当时的履约配置版本；
- 根据项目和地址行政区匹配岗位人员；
- 保存客服经理、项目经理和项目助理快照。

项目发布新履约配置后，新工单使用新版本，进行中工单继续使用原版本。

区域人员分工变化只影响后续匹配，不静默覆盖已有工单人员。

## 14. 页面层级规则

统一遵守：

- 项目列表是项目唯一权威列表；
- 项目团队与区域分工从项目详情进入；
- 履约配置从项目详情进入；
- 模块默认先展示概览或列表；
- 详情通过列表进入；
- 编辑器通过“新建”或“编辑”进入；
- 节点和连线在同一流程工作区编辑；
- 面包屑必须反映项目上下文；
- 返回时保留筛选、分页和滚动位置。

## 15. 当前阶段明确废弃

- Admin 一级菜单“配置中心”；
- 第二套项目配置列表；
- 项目内多个工作流；
- 流程列表和独立流程发布；
- 独立节点设计首页；
- “角色与责任”履约配置模块；
- 项目自定义业务角色；
- 通用责任矩阵；
- 通用处理人规则；
- 根据区域分工变化静默改派历史工单；
- 深色履约配置导航；
- 列表和编辑器混在同一首页；
- 技术 JSON、内部 ID 和英文枚举直接展示。

此前生成的相关图片仅作为探索记录，不构成产品事实源。

## 16. 下一期候选：公共模板

公共模板不进入当前阶段范围。

下一期在出现明确跨项目复用需求后，可评估：

- 流程模板；
- 表单模板；
- 资料要求模板；
- SLA 模板；
- 通知模板；
- 公共业务字典。

公共模板应放在“系统管理 / 公共模板”等明确入口中，不恢复含义模糊的“配置中心”。

## 17. 后续设计顺序

1. 项目团队与区域分工；
2. 网点与资源；
3. 审核与整改；
4. 集成与回传；
5. 版本与发布；
6. Admin 工作台配置待办；
7. Admin 工单中心和派单链路收口。

## 18. 验收标准

- Admin 主导航中不存在“配置中心”；
- 项目只有一个权威列表；
- 项目团队与区域分工和履约配置职责分离；
- 客服经理、项目经理和项目助理按项目与行政区域匹配；
- 工单保存匹配时的人员快照；
- 区域分工变化不静默修改历史工单；
- 履约配置中不存在通用“角色与责任”模块；
- 一个项目当前只支持一个工作流；
- 履约配置作为完整不可变版本发布；
- 公共模板明确延期到下一期；
- 所有页面使用统一浅色导航和 A+ 产品语言；
- 后续规格、读模型和实现均以本文件为事实源。
