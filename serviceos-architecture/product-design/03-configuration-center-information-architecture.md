# 项目管理与履约配置的信息架构

状态：Accepted  
适用范围：ServiceOS Admin 项目管理、项目运营主数据与履约配置  
视觉基线：A+ 经典专业任务型

> 说明：文件名保留历史名称，但“配置中心”不再是正式产品概念，也不作为 Admin 一级菜单。

## 1. 已确认领域关系

```text
客户
└── 项目
    ├── 项目团队与区域分工
    ├── 履约配置
    │   └── 一个履约工作流
    └── 项目工单
        ├── 履约配置版本
        ├── 项目岗位人员快照
        ├── 责任网点分配记录
        ├── 责任师傅分配记录
        └── 审核、整改与复核记录
```

关键边界：

- 项目是实际履约的业务对象；
- 项目团队与区域分工是项目运营主数据；
- 履约配置是可校验、可整体发布的项目规则版本；
- 一个项目当前只有一个履约工作流；
- 工单创建时绑定履约配置版本；
- 工单按项目和地址行政区域匹配客服经理、项目经理和项目助理，并保存人员快照；
- 项目团队调整不需要重新发布履约配置；
- 区域分工变化不能静默修改历史工单人员；
- 网点和师傅的实时状态、排班、负载和产能是运营数据，不进入履约配置版本；
- 实际审核任务、整改记录和复核记录是工单运行数据，不进入履约配置版本；
- 新配置发布不能静默改派工单，也不能改写运行中的审核与整改任务。

## 2. 取消“配置中心”一级菜单

Admin 主导航不设置“配置中心”。

项目的权威入口为：

```text
客户与项目
└── 项目列表
    └── 项目详情
```

项目列表是项目主数据的唯一列表，不另建“项目配置列表”。

跨项目配置待办放入 Admin 工作台，例如：

- 待首次配置；
- 存在未发布草稿；
- 校验失败；
- 等待发布；
- 发布失败；
- 最近完成发布。

点击待办后直接进入对应项目详情的“履约配置”。

## 3. 项目详情信息架构

项目详情至少包含：

1. 项目概览；
2. 服务与合同；
3. 项目团队与区域分工；
4. 履约配置；
5. 项目工单；
6. 操作记录。

页面顶部持续展示项目名称、客户名称、项目状态、服务周期和当前首要风险或动作。

### 3.1 项目团队与区域分工

该页签用于管理：

- 项目成员；
- 客服经理、项目经理和项目助理三个固定岗位；
- 行政区域与岗位人员的对应关系；
- 人员匹配预览；
- 分工变更记录。

该模块不属于履约配置，不使用履约配置草稿和发布流程。它使用独立生效时间，只影响后续新工单或明确重新匹配的工单。

### 3.2 履约配置

该页签用于管理可版本化、可校验、可整体发布的项目履约规则。

页面顶部持续展示：

- 当前生效配置版本；
- 当前草稿版本；
- 配置完整度；
- 校验状态；
- 最近保存时间；
- 当前首要动作。

## 4. 履约配置工作区

左侧使用统一浅色导航：

1. 配置概览；
2. 服务范围；
3. 流程设计；
4. 表单与资料；
5. SLA 与预约；
6. 网点与资源；
7. 审核与整改；
8. 集成与回传；
9. 版本与发布。

“角色与责任”不再是履约配置模块。项目人员和区域分工属于项目详情；流程、SLA、审核、整改和通知只引用工单岗位人员快照。

不得在履约配置左侧菜单中出现流程列表、流程版本、节点设计、流转规则、表单设计器、校验结果、影响分析或发布确认。这些内容属于模块内部列表、详情、属性面板、抽屉或发布步骤。

## 5. 配置概览

配置概览回答：

- 当前项目能否开始接单和履约；
- 哪些规则尚未完成；
- 当前生效配置是什么；
- 是否存在未发布草稿；
- 哪些错误阻止发布；
- 发布后会影响什么；
- 用户下一步应处理什么。

必须展示七个规则模块的完成状态：服务范围、流程设计、表单与资料、SLA 与预约、网点与资源、审核与整改、集成与回传。

配置概览不直接承载复杂编辑。

## 6. 服务范围

服务范围定义项目在哪里提供哪些服务，包括服务类型、服务行政区域、适用条件、范围校验和变更影响。

三个区域相关模块职责不同：

```text
服务范围：项目在哪些区域提供哪些服务
项目团队与区域分工：这些区域由哪些项目管理人员负责
网点与资源：哪些网点和师傅具备承接资格
```

三个模块共享标准行政区目录，但不得合并成一张表，也不得复制维护行政区主数据。

## 7. 流程设计

一个项目当前只有一个履约工作流。点击“流程设计”后直接进入统一流程工作区，不提供项目内流程列表、多个流程、独立流程版本或独立流程发布。

流程工作区包含流程画布、节点配置、流转规则、关联表单和资料、工单处理对象、SLA 规则、审核规则以及校验与影响提示。

流程可以引用：

- 当前工单客服经理；
- 当前工单项目经理；
- 当前工单项目助理；
- 当前工单责任网点；
- 当前工单已分配师傅；
- 当前节点实际处理人；
- 由有权限人员人工选择；
- 系统自动执行。

三个项目岗位人员来自工单快照，流程不得重新执行区域匹配。

审核节点只选择审核规则，并为通过、退回补充、发起整改和异常处理配置明确出口。流程节点不得复制完整检查项或整改规则。

## 8. 表单与资料

“表单与资料”管理当前项目履约过程中使用的表单和资料要求。

默认管理首页包含“表单”和“资料要求”两个页签。页面关系为：

```text
表单列表
→ 表单详情（默认只读）
→ 新建表单或编辑草稿
→ 表单设计器
```

审核和整改规则只能引用表单、字段和资料要求，不能复制字段结构、文件格式、数量或采集方式。

删除被流程、审核或整改规则引用的表单、字段或资料要求时必须阻止，并列出引用位置。

当前阶段不建设跨项目公共模板库。

## 9. SLA 与预约

本模块管理 SLA 指标、预约政策、服务日历、暂停恢复、豁免、预警升级和时间场景预览。

审核、补充、整改和复核时限引用已定义的 SLA 指标，不在“审核与整改”中重复实现时间计算和服务日历。

SLA 预警、升级和预约提醒可以引用工单项目岗位人员快照。本模块不得根据行政区域重新匹配项目人员，也不得维护第二套网点、师傅排班和实时产能。

## 10. 网点与资源

本模块定义：

- 哪些网点可以参与当前项目；
- 每个网点覆盖哪些项目服务区域；
- 每个网点可承接哪些服务类型；
- 网点必须满足的准入条件；
- 师傅必须满足的技能或资质要求；
- 多候选网点的排序和人工选择规则；
- 无符合条件网点时的处理方式。

项目配置只保存资格和匹配规则。网点营业状态、师傅所属网点、师傅资质真实状态、班次排班、任务量、预约占用和实时产能属于运营数据。

实际派单和改派通过工单上下文操作完成，不在本模块直接执行。

## 11. 审核与整改

本模块配置：

- 哪些流程节点或提交动作需要审核；
- 审核对象；
- 审核人来源；
- 审核检查项；
- 审核结论及后续业务效果；
- 退回补充要求；
- 整改触发条件；
- 整改责任对象；
- 整改重新提交要求；
- 整改与复核 SLA；
- 复核规则；
- 达到最大整改轮次后的异常处理；
- 通知和升级。

当前阶段支持的审核结论为：通过、退回补充、发起整改、转异常处理。每个启用结论都必须存在明确流程出口。

审核和整改可以引用：

- 当前工单客服经理；
- 当前工单项目经理；
- 当前工单项目助理；
- 当前责任网点及其指定审核人；
- 当前已分配师傅；
- 原提交人；
- 由具有权限的人员明确指定。

该模块不得重新实现项目人员区域匹配，不得自行创建项目岗位，也不得建设第二套流程任务系统。

运行时审核任务、整改记录、整改任务和复核任务统一归属于工单，并在 Admin、网点端和 iOS 端共享同一权威状态。

## 12. 集成与回传

本模块使用业务协同语言，不把 Inbox、Outbox、Envelope 等技术对象作为普通导航。

内部内容包括入站工单规则、外部字段映射、结果回传、状态同步、通知渠道、失败处理、对接校验和最近协同记录。

业务通知可以引用工单岗位人员快照。外部系统需要负责人信息时，回传工单快照中的业务人员信息，不回传项目区域分工内部结构。

技术诊断进入独立受控诊断工具，不进入履约配置页面。

## 13. 工单岗位人员快照统一规则

流程、SLA、审核、整改、通知和工单页面统一消费当前工单客服经理、项目经理和项目助理。

统一规则：

- 页面打开时不重新计算；
- 流程运行时不重新计算；
- SLA 触发时不重新计算；
- 审核和通知不重新计算；
- 区域分工变化不静默修改历史工单；
- 人员缺失时进入“项目人员待确认”；
- 调整已有工单人员必须使用明确操作并保留审计记录。

详细决策见 `decisions/DEC-001-work-order-project-personnel-snapshot.md`。

## 14. 版本与发布

项目履约配置版本整体包含：

- 服务范围；
- 唯一履约工作流；
- 表单与资料；
- SLA、预约和服务日历；
- 项目网点准入、覆盖和师傅能力规则；
- 审核与整改规则；
- 集成、通知和回传规则。

履约配置版本不包含：

- 项目成员；
- 项目区域人员分工；
- 工单岗位人员快照；
- 网点和师傅主数据；
- 实时排班、负载和产能；
- 已有工单责任网点和师傅分配记录；
- 实际审核任务、整改记录和复核记录。

每个项目最多存在一个当前生效版本和一个正在编辑的草稿版本。生效版本不可修改。

发布步骤为：保存草稿、完整性校验、业务预览、版本对比、影响分析、发布说明、最终确认、发布和结果展示。

项目发布新版本后：

- 新工单使用新版本；
- 进行中工单继续使用创建时绑定的版本；
- 新配置不静默修改已有工单人员、网点、师傅、预约、审核或整改任务；
- 当前阶段不提供自动迁移存量工单。

## 15. 页面层级规则

统一遵守：

- 项目列表是项目的唯一权威列表；
- 项目团队和履约配置都从项目详情进入；
- Admin 工作台只提供跨项目待办和深链接；
- 模块默认先展示概览或列表；
- 详情通过列表进入；
- 编辑器通过“新建”或“编辑草稿”进入；
- 节点和连线通过同一流程工作区属性面板编辑；
- 版本对比、影响分析和发布采用明确步骤；
- 面包屑必须反映项目上下文；
- 返回时保留筛选、分页和滚动位置。

## 16. 当前阶段明确废弃

- Admin 一级菜单“配置中心”；
- 第二套项目配置列表；
- 项目内流程列表和一个项目多个工作流；
- 独立流程、表单、SLA、审核或整改规则发布；
- 履约配置中的“角色与责任”；
- 项目自定义业务角色和通用责任矩阵；
- 通用办公审批引擎；
- 会签、或签、加签和转签；
- 各模块自行按区域匹配项目人员；
- 在项目配置中复制网点、师傅、排班或实时产能；
- 技术 JSON、内部 ID 和英文枚举直接展示给运营人员；
- 根据新配置自动迁移运行中的工单和任务。

此前生成的相关图片仅作为探索记录，不构成产品事实源。

## 17. 下一期候选：公共模板

公共模板不进入当前阶段范围。下一期在出现明确跨项目复用需求后，可评估流程、表单、资料、SLA、审核整改、通知和业务字典模板。

公共模板应放在“系统管理 / 公共模板”等明确入口中，不恢复含义模糊的“配置中心”。

## 18. 后续设计顺序

1. 集成与回传；
2. 版本与发布细化；
3. Admin 工作台配置待办；
4. Admin 工单中心和派单链路收口；
5. 网点端履约链路；
6. 原生 iOS 师傅端。

在页面规格和领域关系未确认前，不继续批量生成高保真图片。

## 19. 验收标准

- Admin 主导航中不存在“配置中心”；
- 项目只有一个权威列表；
- 项目团队与区域分工和履约配置职责明确；
- 一个项目当前只支持一个工作流；
- 项目岗位人员通过工单快照统一提供给流程、SLA、审核和通知；
- 项目规则和工单运行数据边界明确；
- 审核结论具有明确流程出口；
- 整改形成问题、责任、时限、提交和复核闭环；
- 工作流、表单、SLA、审核和整改规则不独立发布；
- 项目履约配置作为完整不可变版本发布；
- 工单绑定创建时的履约配置版本；
- 新配置不静默修改已有工单人员、分配结果和运行任务；
- 公共模板明确延期到下一期；
- 所有页面使用统一浅色导航和 A+ 产品语言；
- 后续规格、读模型和实现均以本文件和已接受决策为事实源。
