# 项目管理与履约配置的信息架构

状态：Accepted  
适用范围：ServiceOS Admin 项目管理、项目运营主数据与履约配置  
视觉基线：A+ 经典专业任务型

> 说明：文件名保留历史名称，但“配置中心”不再是正式产品概念，也不作为 Admin 一级菜单。

## 1. 已确认领域关系

```text
客户
└── 项目
    ├── 项目团队与区域分工
    ├── 履约配置
    │   └── 一个履约工作流
    └── 项目工单
        ├── 履约配置版本
        ├── 项目岗位人员快照
        ├── 责任网点分配记录
        └── 责任师傅分配记录
```

关键边界：

- 项目是实际履约的业务对象；
- 项目团队与区域分工是项目运营主数据；
- 履约配置是可校验、可整体发布的项目规则版本；
- 一个项目当前只有一个履约工作流；
- 工单创建时绑定履约配置版本；
- 工单按项目和地址行政区域匹配客服经理、项目经理和项目助理，并保存人员快照；
- 项目团队调整不需要重新发布履约配置；
- 区域分工变化不能静默修改历史工单人员；
- 网点和师傅的实时状态、排班、负载和产能是运营数据，不进入履约配置版本；
- 已分配工单不会因新配置发布而静默改派。

## 2. 取消“配置中心”一级菜单

Admin 主导航不设置“配置中心”。

项目的权威入口为：

```text
客户与项目
└── 项目列表
    └── 项目详情
```

项目列表是项目主数据的唯一列表，不另建“项目配置列表”。

跨项目配置待办放入 Admin 工作台，例如：

- 待首次配置；
- 存在未发布草稿；
- 校验失败；
- 等待发布；
- 发布失败；
- 最近完成发布。

点击待办后直接进入对应项目详情的“履约配置”。

## 3. 项目详情信息架构

项目详情至少包含：

1. 项目概览；
2. 服务与合同；
3. 项目团队与区域分工；
4. 履约配置；
5. 项目工单；
6. 操作记录。

页面顶部持续展示：

- 项目名称；
- 客户名称；
- 项目状态；
- 服务周期；
- 当前首要风险或动作。

### 3.1 项目团队与区域分工

该页签用于管理：

- 项目成员；
- 客服经理、项目经理和项目助理三个固定岗位；
- 行政区域与岗位人员的对应关系；
- 人员匹配预览；
- 分工变更记录。

该模块：

- 不属于履约配置；
- 不使用履约配置草稿和发布流程；
- 使用独立生效时间；
- 只影响后续新工单或明确重新匹配的工单；
- 不静默改写历史工单人员。

### 3.2 履约配置

该页签用于管理可版本化、可校验、可整体发布的项目履约规则。

页面顶部持续展示：

- 当前生效配置版本；
- 当前草稿版本；
- 配置完整度；
- 校验状态；
- 最近保存时间；
- 当前首要动作。

## 4. 履约配置工作区

履约配置工作区左侧使用统一浅色导航：

1. 配置概览；
2. 服务范围；
3. 流程设计；
4. 表单与资料；
5. SLA 与预约；
6. 网点与资源；
7. 审核与整改；
8. 集成与回传；
9. 版本与发布。

“角色与责任”不再是履约配置模块。

项目人员和区域分工属于项目详情；流程、SLA、审核和通知只引用工单岗位人员快照。

不得在履约配置左侧菜单中出现：

- 流程列表；
- 流程版本；
- 节点设计；
- 流转规则；
- 表单设计器；
- 校验结果；
- 影响分析；
- 发布确认。

这些内容属于模块内部列表、详情、属性面板、抽屉或发布步骤。

## 5. 配置概览

配置概览负责回答：

- 当前项目能否开始接单和履约；
- 哪些规则尚未完成；
- 当前生效配置是什么；
- 是否存在未发布草稿；
- 哪些错误阻止发布；
- 发布后会影响什么；
- 用户下一步应处理什么。

必须展示：

- 配置完整度；
- 七个规则模块的完成状态；
- 生效版本摘要；
- 草稿变更摘要；
- 阻塞错误；
- 重要警告；
- 最近操作；
- 推荐下一步。

七个规则模块为：服务范围、流程设计、表单与资料、SLA 与预约、网点与资源、审核与整改、集成与回传。

配置概览不直接承载复杂编辑。

## 6. 服务范围

服务范围定义项目在哪里提供哪些服务。

内部内容包括：

- 服务项目和业务类型；
- 服务行政区域；
- 适用条件；
- 范围校验；
- 变更影响。

服务范围回答：

> 当前项目在哪些区域提供哪些服务。

项目团队与区域分工回答：

> 这些区域由哪些客服经理、项目经理和项目助理负责。

网点与资源回答：

> 哪些网点和师傅具备承接这些服务的资格。

三个模块共享标准行政区目录，但不得合并成一张表，也不得复制维护行政区主数据。

## 7. 流程设计

### 7.1 一个项目一个工作流

点击“流程设计”后直接进入该项目唯一履约流程工作区。

当前阶段不提供：

- 流程列表；
- 项目内多个流程；
- 复制流程；
- 独立流程版本；
- 独立流程发布；
- 公共流程模板。

### 7.2 统一流程设计工作区

工作区包含：

1. 流程画布；
2. 节点配置；
3. 流转规则；
4. 关联表单和资料；
5. 工单处理对象；
6. SLA 规则；
7. 校验与影响提示。

点击节点后在右侧编辑节点；点击连线后在右侧编辑流转规则。它们不是独立菜单或独立首页。

### 7.3 工单处理对象

当前阶段流程可以引用：

- 当前工单客服经理；
- 当前工单项目经理；
- 当前工单项目助理；
- 当前工单责任网点；
- 当前工单已分配师傅；
- 当前节点实际处理人；
- 由有权限人员人工选择；
- 系统自动执行。

其中三个项目岗位人员来自工单快照，流程不得重新执行区域匹配。

## 8. 表单与资料

“表单与资料”管理当前项目履约过程中使用的表单和资料要求。

默认管理首页包含：

- 表单；
- 资料要求。

页面关系：

```text
表单列表
→ 表单详情（默认只读）
→ 新建表单或编辑草稿
→ 表单设计器
```

不得进入模块后直接打开字段编辑器，也不得把列表和编辑器挤在同一页面。

当前阶段的表单和资料要求归属于项目履约配置，不建设跨项目公共模板库。

## 9. SLA 与预约

内部内容包括：

- SLA 指标；
- 预约政策；
- 服务日历；
- 暂停、恢复和豁免；
- 预警和超时升级；
- 时间场景预览。

SLA 预警、升级和预约提醒可以引用工单项目岗位人员快照。

本模块不得：

- 根据行政区域重新匹配项目人员；
- 维护网点或师傅排班；
- 维护第二套实时产能；
- 独立发布 SLA 或预约规则。

## 10. 网点与资源

本模块定义：

- 哪些网点可以参与当前项目；
- 每个网点覆盖哪些项目服务区域；
- 每个网点可承接哪些服务类型；
- 网点必须满足的准入条件；
- 师傅必须满足的技能或资质要求；
- 多候选网点的排序和人工选择规则；
- 无符合条件网点时的处理方式。

项目配置只保存资格和匹配规则。

以下内容属于实时运营数据，不进入配置版本：

- 网点当前营业状态；
- 师傅所属网点；
- 师傅资质真实状态；
- 班次和排班；
- 当前任务量；
- 已占用预约时段；
- 实时可用产能；
- 临时停服和请假。

实际派单和改派通过工单上下文操作完成，不在本模块直接执行。

## 11. 审核与整改

本模块定义：

- 哪些节点或提交需要审核；
- 审核条件；
- 审核结论；
- 整改类型；
- 整改时限；
- 整改资料要求；
- 复核规则；
- 通知和升级。

审核、整改和复核规则可以引用：

- 当前工单项目岗位人员快照；
- 当前责任网点；
- 当前已分配师傅；
- 当前流程任务处理人；
- 明确的审核人员来源。

该模块不得重新实现项目人员区域匹配，也不得自行创建项目岗位。

## 12. 集成与回传

本模块使用业务协同语言，不把 Inbox、Outbox、Envelope 等技术对象作为普通导航。

内部内容包括：

- 入站工单规则；
- 外部字段映射；
- 结果回传规则；
- 状态同步；
- 通知渠道；
- 失败处理；
- 对接校验；
- 最近协同记录。

业务通知可以引用工单项目岗位人员快照。

外部系统需要负责人信息时，回传工单当前快照中的业务人员信息，不回传项目区域分工内部结构。

技术诊断进入独立受控诊断工具，不进入履约配置页面。

## 13. 工单岗位人员快照统一规则

流程、SLA、审核、整改、通知和工单页面统一消费：

- 当前工单客服经理；
- 当前工单项目经理；
- 当前工单项目助理。

这些值来自工单保存的人员快照。

统一规则：

- 页面打开时不重新计算；
- 流程运行时不重新计算；
- SLA 触发时不重新计算；
- 审核和通知不重新计算；
- 区域分工变化不静默修改历史工单；
- 人员缺失时进入“项目人员待确认”；
- 调整已有工单人员必须使用明确操作并保留审计记录。

详细决策见 `decisions/DEC-001-work-order-project-personnel-snapshot.md`。

## 14. 版本与发布

### 14.1 履约配置版本范围

项目履约配置版本整体包含：

- 服务范围；
- 唯一履约工作流；
- 表单与资料；
- SLA、预约和服务日历；
- 项目网点准入、覆盖和师傅能力规则；
- 审核与整改规则；
- 集成、通知和回传规则。

履约配置版本不包含：

- 项目成员；
- 项目区域人员分工；
- 工单岗位人员快照；
- 网点和师傅主数据；
- 实时排班、负载和产能；
- 已有工单的责任网点和师傅分配记录。

### 14.2 草稿与生效版本

每个项目最多存在：

- 一个当前生效版本；
- 一个正在编辑的草稿版本。

生效版本不可修改。新修改基于最新生效版本创建草稿，发布后形成新的不可变履约配置版本。

### 14.3 发布步骤

1. 保存草稿；
2. 执行跨模块完整性校验；
3. 预览履约结果；
4. 查看新旧版本业务差异；
5. 执行影响分析；
6. 填写发布说明；
7. 最终确认；
8. 发布；
9. 展示发布结果。

不得跳过校验和影响分析直接发布。

### 14.4 工单版本绑定

项目发布新版本后：

- 新工单使用新版本；
- 进行中工单继续使用创建时绑定的版本；
- 新配置不静默修改已有工单人员、责任网点、责任师傅或已确认预约；
- 当前阶段不提供自动迁移存量工单。

## 15. 页面层级规则

统一遵守：

- 项目列表是项目的唯一权威列表；
- 项目团队和履约配置都从项目详情进入；
- Admin 工作台只提供跨项目待办和深链接；
- 模块默认先展示概览或列表；
- 详情通过列表进入；
- 编辑器通过“新建”或“编辑草稿”进入；
- 节点和连线通过同一流程工作区的属性面板编辑；
- 版本对比、影响分析和发布采用明确步骤；
- 面包屑必须反映项目上下文；
- 返回时保留筛选、分页和滚动位置。

## 16. 当前阶段明确废弃

- Admin 一级菜单“配置中心”；
- 第二套“项目配置列表”；
- 项目内流程列表；
- 一个项目多个工作流；
- 独立流程版本和独立流程发布；
- 履约配置中的“角色与责任”；
- 项目自定义业务角色；
- 通用责任矩阵；
- 各模块自行按区域匹配项目人员；
- 深色履约配置导航；
- `流程列表 / 流程版本 / 节点设计 / 流线规则` 同级菜单；
- 独立节点设计首页；
- 多张重复节点编辑页面；
- 履约配置首页直接展开表单编辑器；
- 在项目配置中复制网点、师傅、排班或实时产能数据；
- 技术 JSON、内部 ID 和英文枚举直接展示给运营人员。

此前生成的相关图片仅作为探索记录，不构成产品事实源。

- 表单模板；
- 资料要求模板；

3. 版本与发布细化；
4. Admin 工作台配置待办；

在页面规格和领域关系未确认前，不继续批量生成高保真图片。

## 19. 验收标准

- Admin 主导航中不存在“配置中心”；
- 项目只有一个权威列表；
- 项目团队与区域分工和履约配置职责明确；
- 一个项目当前只支持一个工作流；
- 项目岗位人员通过工单快照统一提供给流程、SLA、审核和通知；
- 工作流、表单和 SLA 不独立发布；
- 项目履约配置作为完整不可变版本发布；
- 工单绑定创建时的履约配置版本；
- 项目人员分工和实时资源数据不进入履约配置版本；
- 新配置不静默修改已有工单人员和分配结果；
- 公共模板明确延期到下一期；
- 所有页面使用统一浅色导航和 A+ 产品语言；
- 后续规格、读模型和实现均以本文件和已接受决策为事实源。
