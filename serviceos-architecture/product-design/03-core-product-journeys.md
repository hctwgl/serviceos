# ServiceOS 核心产品旅程

状态：Accepted
用途：约束完整纵向切片，避免按接口、字段或孤立页面交付。

## 1. 工单责任链

每张工单同时存在四类责任：

```text
项目协同人员：客服经理、项目经理、项目助理
履约组织责任：责任网点
现场执行责任：责任师傅
流程任务责任：当前任务处理人
```

四类责任不能合并为通用“负责人”。项目人员由项目与行政区域匹配并写入工单快照；责任网点和责任师傅由不同业务命令建立；当前任务处理人由流程运行产生。

## 2. Admin 工单运营旅程

### 用户目标

平台运营人员需要迅速发现高风险工单，理解当前责任和阻塞，并执行有权限的下一步。

### 完整旅程

```text
工作台风险或待办
→ 工单中心业务队列
→ 工单工作区
→ 查看阶段、责任、SLA 与阻塞
→ 执行派网点、协调、审核、整改或异常处理
→ 服务端命令完成
→ 工作区与工作台投影刷新
```

### 不变量

- 工作台和工单中心是查询投影，不保存第二套工单状态；
- 工单工作区是理解单张工单和进入正式操作的权威页面；
- 候选必须由服务端给出资格结果、排序依据和排除原因；
- 候选过期或工单变化时命令失败关闭，前端要求刷新；
- 平台派网点后不自动派师傅；
- 改派网点必须结束旧网点和旧师傅的当前责任，并保留历史。

## 3. 项目建立与履约配置旅程

### 用户目标

项目人员建立可履约项目，配置完整业务规则，校验影响后一次性发布。

### 完整旅程

```text
创建项目
→ 维护项目团队与区域分工
→ 打开项目详情的履约配置
→ 配置服务范围和唯一工作流
→ 配置表单资料、SLA 预约、网点资源、审核整改与集成规则
→ 跨模块校验
→ 版本对比与影响分析
→ 原子发布
→ 新工单绑定新版本
```

### 不变量

- 项目列表是唯一项目权威目录；
- 当前一个项目只有一份履约配置和一个工作流；
- 项目人员和区域分工属于日常项目主数据，不随履约版本发布；
- 履约配置所有模块组成一个不可变生效版本，不单独发布；
- 发布全部成功或原生效版本完全不变；
- 新版本只用于新工单，进行中工单继续使用创建时绑定版本；
- 公共流程、表单和 SLA 模板不在当前期实施。

## 4. Network 履约协作旅程

### 用户目标

网点负责人和调度人员处理当前由本网点负责的工单，为其分配合格师傅并保证预约与履约风险受控。

### 完整旅程

```text
平台分配责任网点
→ 网点工作台出现待办
→ 打开待分配工单
→ 查看本网点合格师傅候选及冲突
→ 逐单分配师傅
→ 协调预约并跟踪履约
→ 处理整改或异常
→ 必要时向平台发起跨网点协调
```

### 不变量

- 一个网点可以服务多个项目，但只能处理当前由自己负责的工单；
- 网点只能选择本网点合格师傅；
- 网点不能改派责任网点，也不能把工单直接转给其他网点；
- 网点不修改项目配置、项目人员快照或平台审核事实；
- 现场作业开始后，普通改派师傅被阻止，进入明确交接或异常流程；
- 工作台、预约、整改和异常页面消费同一服务端业务事实。

## 5. 原生 iOS 现场作业旅程

### 用户目标

师傅在弱网或断网现场仍能可信地完成任务、采集资料、恢复上传并处理冲突。

### 完整旅程

```text
登录与设备校验
→ 下载今日任务和可离线工作包
→ 查看任务详情
→ 联系客户并确认预约
→ 到场定位与开始作业
→ 按步骤执行、填表和采集资料
→ 提交前完整性检查
→ 保存本地命令并提交
→ 系统后台上传资料
→ 服务端确认或返回冲突
→ 用户完成冲突/整改处理
```

### 不变量

- iOS 只处理当前分配给本人的任务；
- 本地保存成功与服务端提交成功必须使用不同状态表达；
- 页面始终显示网络、同步、待上传和本地保存状态；
- 离线命令具有幂等标识、前置版本和明确重放顺序；
- 后台上传不依赖页面持续运行；
- 系统杀进程后可以恢复任务、草稿、命令和上传状态；
- 服务器事实与本地修改冲突时不得静默覆盖，必须显式处理；
- H5 不伪装支持离线、后台上传和设备可信能力。

## 6. 审核整改旅程

```text
现场提交表单和资料快照
→ 完整性校验
→ 创建审核任务
→ 通过 / 退回补充 / 发起整改 / 转异常
→ 整改责任方提交新快照
→ 创建复核任务
→ 复核通过或继续明确处置
```

原始提交、审核意见和每轮整改快照不可被后续提交覆盖。审核人与提交人的职责分离、最大整改轮次和异常出口属于核心业务语义，实施前必须以已接受事实或用户确认为准。

## 7. 外部协同旅程

项目履约配置保存业务映射与回传规则；系统管理保存连接、凭据和技术协议；实际入站、发送、确认、失败与重试属于工单运行事实。三者不得合并。

回传失败重试使用首次生成的冻结业务快照，不能按最新配置重新拼装；需要更正业务事实时，先通过工单业务命令产生新事件。

## 8. 纵向切片验收

每条核心旅程的交付必须同时证明：

- 真实角色可以从入口完成业务目标；
- 允许与拒绝路径均由服务端授权；
- 页面读模型不依赖浏览器跨领域拼装；
- 命令具有并发、幂等、审计和失败关闭语义；
- 正常、空、错误、无权限、只读、处理中和冲突状态可复现；
- 中文名称、错误和下一步建议对业务用户可理解；
- 对应契约、迁移、测试和实施状态已同步；
- 真实运行页面经过产品负责人验收，而非只审阅文档或截图。
