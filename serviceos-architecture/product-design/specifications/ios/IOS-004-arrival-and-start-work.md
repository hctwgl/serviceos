# IOS-004 到场与开始作业

状态：Accepted
设计等级：A

## 1. 页面定位

“到场与开始作业”用于当前责任师傅在服务现场主动采集一次定位、完成到场签到，并在平台确认到场事实后进入现场作业流程。

页面必须始终区分以下业务状态：

- 已打开导航；
- 已到达现场但尚未采集定位；
- 本机已经采集定位；
- 本机已经保存离线签到命令；
- 平台已经确认到场并创建或恢复 `Visit`；
- 已进入作业步骤页面；
- 现场作业事实已经由后续作业命令建立。

点击导航、采集到定位、保存离线命令和平台确认签到都不等于“已经开始作业”。“开始作业”在本页面只负责进入 IOS-005 作业步骤；真正的步骤开始、表单保存、资料采集和作业完成由后续页面及对应领域命令负责。

## 2. 产品边界

IOS-004 负责：

- 展示当前任务、预约、地址和到场要求；
- 使用系统地图打开导航；
- 在用户明确点击后采集一次当前位置；
- 展示定位质量、围栏判断和项目签到政策；
- 在政策允许时提交到场签到；
- 处理范围外、低精度、定位不可用和预约时间异常；
- 在线确认 `Visit`，或在有效工作包允许时保存离线签到命令；
- 展示到场事实、同步可信度和当前可执行动作；
- 在到场事实满足要求后进入 IOS-005 作业步骤；
- 将责任、预约、版本或离线冲突导向 IOS-010。

IOS-004 不负责：

- 持续或后台定位；
- 自动判断师傅已经到场；
- 自动修改预约时间或客户爽约状态；
- 修改责任网点、责任师傅或项目人员；
- 在客户端硬编码签到半径；
- 通过手填经纬度、截图或自由文本绕过定位政策；
- 在平台未确认时显示“签到成功”；
- 创建没有真实作业引用的签退或完成记录；
- 直接完成工单、流程节点或现场作业。

## 3. 访问与责任校验

在线读取和签到必须同时满足：

```text
authenticated principal
+ trusted Technician Context
+ ACTIVE TechnicianProfile
+ ACTIVE NetworkTechnicianMembership
+ current ACTIVE technician responsibility
+ required capability
+ current task allowed-action
+ appointment and visit policy
```

服务端必须重新校验当前责任、任务状态、预约状态、工单绑定的履约配置版本和命令并发版本。客户端显示了按钮，不代表服务端可以省略校验。

离线签到还必须满足：

- 当前账号和设备隔离有效；
- 有效工作包显式包含到场离线动作；
- 工作包包含当前任务、责任、预约、签到政策和地址快照；
- 工作包未过期且未被 tombstone、撤权或设备撤销覆盖；
- 当前不存在未解决的同任务到场冲突；
- 稳定的设备命令标识和依赖版本完整。

无法证明当前责任时统一失败关闭，不通过差异化错误泄露其他师傅或其他网点任务是否存在。

## 4. 页面结构

```text
任务与预约摘要 + 同步可信度
→ 服务地址与打开导航
→ 到场要求和安全提示
→ 定位权限与一次性定位采集
→ 定位质量和签到范围结果
→ 范围例外说明与资料要求
→ 到场签到结果
→ 开始作业入口
→ 最近到场事实与冲突入口
```

固定底部区域最多突出一个主操作：

```text
采集当前位置
提交到场签到
等待平台确认
开始作业
处理冲突
```

页面不能同时突出“签到”和“开始作业”，避免用户跳过当前尚未成立的事实。

## 5. 任务与预约摘要

首屏展示：

- 工单号和任务名称；
- 当前项目或服务名称；
- 当前责任网点；
- 预约日期、时间窗口和业务时区；
- 服务地址和受控导航入口；
- 允许签到的时间范围；
- 当前网络、工作包和同步状态；
- 当前到场或 `Visit` 状态；
- 影响签到的阻止原因。

设备时间不能作为预约窗口和签到时限的唯一依据。在线使用服务器时间，离线使用工作包中的服务器时间偏移并明确显示“截至最近同步”。

预约未确认、已经取消、已经过期或被改约时，页面必须读取最新预约事实或有效工作包，不得沿用页面首次打开时的旧时间窗口。
## 6. 打开导航

点击“打开导航”只把当前受控服务地址或地图点交给系统地图，不创建任何到场、签到或联系事实。

导航入口必须遵守客户字段策略：

- 只提供当前任务必要的地址；
- 不把地址写入普通日志、埋点或剪贴板；
- 任务撤权后立即清除受控地址和导航入口；
- 地址无法解析时明确提示平台或网点维护，不能使用历史地址兜底。

从地图返回 App 后，页面只提示“请确认已到达现场并采集当前位置”，不能根据返回时间或地图状态自动提交签到。

## 7. 定位授权

App 只在用户点击“采集当前位置”时请求“使用 App 期间”定位授权。

禁止：

- 启动 App 时无业务上下文地申请定位；
- 为签到申请持续后台定位；
- 在用户未操作时周期性刷新位置；
- 用推送、定时器或页面轮询秘密采集位置；
- 将拒绝定位授权解释为范围外。

授权说明必须使用中文解释：

> 用于核验本次服务是否到达预约现场。ServiceOS 不会为该功能持续跟踪您的位置。

权限被拒绝或系统定位关闭时，页面展示恢复方法和项目政策允许的替代处置。不能反复弹出系统授权，也不能直接把签到标记为成功。

## 8. 一次性定位采集

一次采集至少形成以下受控事实：

```text
capturedAt
coordinate
horizontalAccuracyMeters
locationSource
deviceCommandId
task and appointment reference
responsibility and policy version
```

其中坐标、精度和设备信息属于受控敏感数据：

- 只用于签到政策计算、审计和必要争议处理；
- 不在普通日志、崩溃报告或产品埋点中输出；
- 不在普通业务页面长期显示原始经纬度；
- Network Portal 和 Admin 默认只看到业务化范围结论；
- 数据保留和访问遵循签到证据政策。

页面展示的定位状态使用中文业务表达：

- 正在定位；
- 已获取，可用于核验；
- 定位精度不足；
- 超出签到范围；
- 定位服务不可用；
- 定位权限未授予；
- 定位结果已过期，请重新采集。

位置达到政策规定的有效时长后自动失效。提交签到时不得使用已经过期的采集结果。

## 9. 签到范围与政策结果

签到范围由工单绑定的履约配置版本或有效工作包提供，客户端不能硬编码统一半径。

政策计算必须能够解释：

- 使用的现场位置来源；
- 允许范围；
- 本次定位精度；
- 范围结论；
- 适用政策版本；
- 当前允许动作；
- 需要的原因、资料或审批。

业务结果至少区分：

```text
范围内，可签到
范围外，允许说明后签到
范围外，需要补充资料
范围外，需要平台或网点确认
范围外，禁止签到
定位精度不足，需要重新采集
定位不可用，按项目政策处理
```

定位不可用、精度不足和范围外是三个不同事实，不能合并成一个“定位异常”。

客户端可以展示本地预评估，但平台确认的在线签到必须由服务端使用同一政策版本复核。客户端结果和服务端结果不一致时，以服务端为准并解释需要重新定位或处理冲突。

## 10. 范围外和定位异常处置

项目政策允许范围外签到时，服务端返回受控原因、说明和资料要求。页面按要求收集：

- 标准原因；
- 必要业务说明；
- 必要现场资料引用；
- 是否需要网点或平台确认；
- 当前允许继续的范围。

禁止使用一段自由文本代替标准原因，也禁止上传地图截图作为唯一定位证据。

项目政策要求人工确认时，页面创建明确待办或异常入口，并显示“等待确认”，不能让师傅通过重复点击跳过确认。

项目政策禁止签到时，页面提供“发起现场异常”或“联系网点协调”入口。异常处理必须回到权威任务或异常领域，不在本页本地修改签到政策。

## 11. 预约窗口与到场关系

预约和到场是关联但独立的事实：

- 打开导航不会修改预约；
- 提前到达不自动提前预约；
- 迟到不自动取消预约；
- 师傅未签到不自动认定客户爽约；
- 客户爽约必须经过 IOS-003 定义的预约处置；
- 改约后旧预约位置和时间不能继续用于新签到。

提前或超出预约窗口签到时，项目政策可以：

- 允许并记录说明；
- 要求客户确认；
- 要求网点或平台确认；
- 阻止并要求调整预约；
- 转入异常处理。

客户端不能自行把预约时间异常降级为普通警告。

## 12. 在线到场签到

在线签到流程为：

```text
采集一次有效定位
→ 展示政策结果和提交影响
→ 用户明确确认签到
→ 提交稳定 deviceCommandId 和当前版本
→ 服务端重新校验责任、预约、政策和定位
→ 在同一事务中确认 Visit、审计、幂等结果和 Outbox
→ 返回 Visit 标识、状态和版本
→ 页面显示“平台已确认到场”
```

平台接收时间 `receivedAt` 由服务端生成，与设备业务发生时间 `capturedAt` 分开保存。

重复点击或网络重试使用同一 `deviceCommandId`，必须返回已有结果，不能创建第二个 `Visit` 或重复到场记录。

同一任务和当前责任已经存在有效 `Visit` 时，页面恢复该 `Visit`，不能新建一个平行现场访问。存在相互冲突的 Visit 状态时进入 IOS-010，不由客户端选一个继续。

只有服务端返回权威 `Visit` 状态和版本后，页面才能显示“平台已确认到场”。

## 13. 离线到场签到

有效工作包明确允许时，离线签到按以下状态表达：

```text
本机已采集定位
→ 本机已保存签到命令
→ 等待平台确认
→ 平台已确认到场 / 需要处理冲突
```

离线状态不得显示“签到成功”。固定文案为：

> 本机已记录到场信息，联网后仍需平台确认。

离线命令至少保存：

- 稳定 `deviceCommandId`；
- 工作包和任务引用；
- 责任、预约、配置和签到政策版本；
- `capturedAt` 和服务器时间偏移；
- 定位、精度和来源；
- 本地政策评估；
- 标准例外原因；
- 必要资料的本地受控引用；
- 命令依赖和本地状态。

恢复网络后，服务端必须重新校验当前责任、预约、配置和 Visit 状态。责任已变化、预约已改约、工作包已失效或已经存在冲突 Visit 时，不得创建伪造的历史 Visit；本机成果进入 IOS-010 受控隔离和人工处置。

只有普通缓存而没有有效工作包时，可以查看最近非敏感摘要，但不能离线签到。

## 14. 到场后的页面状态

平台确认到场后，页面展示：

- 到场时间；
- 当前 Visit 状态；
- 范围核验结果；
- 是否使用例外原因；
- 当前同步状态；
- 后续允许动作；
- 最近平台确认时间。

默认不展示：

- 原始经纬度；
- 设备标识；
- 内部 Visit ID；
- 命令 ID；
- 技术政策码；
- 原始异常堆栈。

受权限控制的业务人员可以从权威 Visit 历史查看业务化到场结果，但不能通过本页面获取无关敏感定位数据。

## 15. 开始作业

“开始作业”与“到场签到”必须分离。

平台已确认到场且服务端 `allowed-actions` 允许时，点击“开始作业”进入 IOS-005。进入页面本身不自动完成流程节点，也不伪造第一项现场操作。

IOS-005 根据当前任务和 Visit 版本展示第一个可执行步骤；真正改变作业状态的命令由该步骤的领域语义决定。

离线签到仍等待平台确认时：

- 工作包未允许后续离线作业：只允许查看等待状态和同步入口；
- 工作包明确允许依赖该离线签到继续：可以进入 IOS-005，但页面必须持续显示“到场待平台确认”；
- 后续离线命令必须声明对签到命令的依赖，按顺序同步；
- 签到最终冲突时，依赖命令全部停止并进入 IOS-010，不能绕过签到单独提交。

客户端不能把“打开 IOS-005”记录为正式开始作业事实。

## 16. Visit 生命周期边界

IOS-004 只处理当前现场访问的到场入口和安全恢复，不承担完整 Visit 生命周期管理。

明确边界：

- 到场确认创建或恢复当前 Visit；
- 一个任务当前责任下不得存在两个并行有效 Visit；
- 中断、恢复、签退和完成由后续作业流程及真实业务引用驱动；
- 签退必须引用真实完成的现场作业事实；
- 不得为通过接口校验而创建占位 `operationRef`；
- 发布新配置不能改写已有 Visit 的政策和历史结论。

存在已中断 Visit 时，服务端必须返回“恢复原 Visit”或“不能继续”的明确动作，客户端不能自行创建新 Visit。

## 17. 页面读模型

建议使用页面级读模型：

```text
TechnicianArrivalWorkspaceView
├── taskSummary
├── responsibilitySummary
├── appointmentSummary
├── serviceLocationSummary
├── arrivalPolicy
├── currentVisit
├── localArrivalState
├── syncSummary
├── allowedActions
└── blockingReasons
```

`arrivalPolicy` 至少提供：

- 位置有效时长；
- 精度要求；
- 签到范围和业务化说明；
- 提前或超时签到政策；
- 范围外和定位不可用处置；
- 所需原因、资料和确认；
- 政策版本。

前端不能分别请求地址、预约、责任、政策、Visit 和流程接口后自行拼装“是否可以签到”。

## 18. 命令边界

IOS-004 允许调用的业务命令仅包括：

- 到场签到；
- 提交政策要求的到场例外信息；
- 发起明确的现场异常或协调；
- 在线恢复已有 Visit；
- 将允许的离线签到命令进入受控队列。

以下不是 IOS-004 的命令：

- 修改预约；
- 调整责任网点或师傅；
- 强制签到；
- 手工修改 Visit 状态；
- 通用“开始作业”；
- 签退或完成工单；
- 修改签到范围或项目政策。

“采集当前位置”和“打开作业步骤”是客户端动作，不应伪装成业务写命令。

## 19. 并发、幂等与冲突

签到提交必须携带：

- 稳定 `deviceCommandId`；
- 当前任务或 Visit 并发版本；
- 责任版本；
- 预约 Revision；
- 配置和政策版本；
- 业务发生时间；
- 定位证据摘要。

以下变化必须拒绝旧提交：

- 责任师傅或责任网点变化；
- 预约被取消或改约；
- 任务进入不可签到状态；
- 当前 Visit 已被其他受控操作推进；
- 工作包被撤销或过期；
- 设备或账号失效；
- 到场政策版本不再适用于当前任务。

拒绝后保留本机受控成果，并进入 IOS-010。不得重写命令版本重试，也不得以最新责任身份提交旧师傅采集的定位。

## 20. 错误与恢复状态

页面至少覆盖：

| 状态 | 产品表现 | 允许动作 |
|---|---|---|
| 定位中 | 显示采集进度 | 取消采集 |
| 定位精度不足 | 解释精度要求 | 重新采集、按政策处置 |
| 定位权限拒绝 | 解释用途和恢复方法 | 打开系统设置、协调 |
| 范围外可说明 | 显示命中政策 | 填写标准原因和资料 |
| 范围外被阻止 | 显示阻止原因 | 发起异常、联系网点 |
| 在线签到处理中 | 防止重复提交 | 等待、查看网络状态 |
| 平台已确认到场 | 显示权威 Visit 摘要 | 开始作业 |
| 本机已记录 | 明确尚未平台确认 | 查看同步状态 |
| 责任或预约变化 | 停止继续签到 | 处理冲突 |
| 已存在有效 Visit | 恢复权威 Visit | 继续作业 |
| 外部依赖不可用 | 明确不能判断 | 重试、协调，不伪造成功 |

技术错误统一转换为中文业务说明、稳定错误码和下一步动作。普通用户界面不显示 HTTP 状态码、异常类名或原始 JSON。

## 21. 安全、隐私与审计

定位、签到、例外和 Visit 变更必须可审计，至少记录：

- 租户、项目、工单和任务；
- 当前责任网点和师傅；
- 操作者和受信 Context；
- 业务动作和稳定错误码；
- `capturedAt`、`receivedAt` 和服务器时间基准；
- 定位质量和业务化范围结论；
- 使用的政策、预约和责任版本；
- 例外原因和必要资料引用；
- 在线或离线命令来源；
- 幂等结果和 Visit 版本。

日志使用中文事件说明和稳定英文字段，但不得记录完整地址、原始坐标、客户联系方式、Authorization、完整设备标识或资料正文。

App 必须避免在应用切换器快照、通知和锁屏中暴露完整地址和定位结果。

## 22. 无障碍与移动交互

- 主操作触控区域不小于 iOS 推荐尺寸；
- 定位中、成功、警告和失败不能只依赖颜色；
- VoiceOver 能读出范围结论、同步可信度和下一步；
- 动态字体放大后，主操作和阻止原因仍可完整阅读；
- 定位和网络等待提供明确进度，不使用无限无说明加载；
- 确认签到前完整说明将提交的事实和异常信息；
- 低电量和弱网时仍能可靠保存允许的离线命令。

## 24. 验收标准

### 正常链路

- [ ] 用户主动点击后才采集一次位置；
- [ ] 页面展示定位质量和业务化范围结果；
- [ ] 在线签到由服务端重新校验责任、预约和政策；
- [ ] 重复提交相同命令不会创建第二个 Visit；
- [ ] 只有平台返回权威结果后才显示“平台已确认到场”；
- [ ] 到场后“开始作业”只进入 IOS-005，不伪造作业完成事实。

### 异常和例外

- [ ] 定位不可用、精度不足和范围外分别处理；
- [ ] 范围外处置使用服务端受控原因和资料要求；
- [ ] 提前、超时或预约变化不会被客户端静默忽略；
- [ ] 已存在有效 Visit 时恢复原 Visit，不创建平行 Visit；
- [ ] 无法获得权威判断时明确失败，不显示成功。

### 离线和冲突

- [ ] 只有有效工作包明确允许时才能离线签到；
- [ ] 离线状态显示“本机已记录，等待平台确认”；
- [ ] 后续离线作业命令声明对签到命令的依赖；
- [ ] 责任、预约或版本冲突不会创建伪 Visit；
- [ ] 本机成果进入 IOS-010 受控处置，不被静默丢弃或改写。

### 安全与隐私

- [ ] 不申请持续后台定位；
- [ ] 原始坐标不进入普通日志、通知或埋点；
- [ ] 撤权后不再展示地址、定位和签到入口；
- [ ] 审计能够关联操作者、责任、预约、政策、位置质量和幂等结果；
- [ ] 普通界面不展示内部 ID、英文枚举、技术错误和原始经纬度。

## 25. 禁止实现

禁止：

- 进入页面自动签到；
- 打开导航即签到；
- 使用固定半径覆盖项目政策；
- 通过手填坐标、地图截图或自由文本强制签到；
- 定位权限拒绝时默认范围内；
- 离线保存后显示“签到成功”；
- 到场签到同时完成流程节点或工单；
- 用占位作业引用完成 Visit 签退；
- 责任变化后以新责任身份提交旧师傅位置；
- 捕获定位或提交异常后返回默认成功；
- 为 H5、旧客户端或开发数据保留双轨到场模型。
