# IOS-003 联系与预约

状态：Accepted
设计等级：A

## 1. 页面定位

“联系与预约”用于当前责任师傅联系客户、记录真实联系结果，并在项目规则允许的范围内提议、确认、改约、取消或处理爽约。

它维护的是与 Admin、Network Portal 共用的同一组 `ContactAttempt` 和 `Appointment` 事实，不在 iOS 本地建立平行预约状态机。

页面必须始终区分：

- 点击拨号；
- 实际联系结果；
- 本地已保存的联系记录；
- 服务器已接收的联系事实；
- 本地预约草稿；
- 已提交的预约提议；
- 客户或授权主体已经确认的预约；
- 通知是否发送成功。

## 2. 产品边界

IOS-003 负责：

- 展示当前客户联系入口和最小联系历史；
- 使用系统拨号能力发起联系；
- 在返回 App 后记录标准联系结果；
- 展示服务端提供的可预约时间窗；
- 创建预约提议；
- 在权限和项目政策允许时确认、改约或取消预约；
- 在预约窗口结束后按规则处理客户爽约；
- 展示 Appointment Revision、并发、通知和 SLA 影响；
- 离线保存联系和预约草稿，并按工作包政策排队允许的命令。

IOS-003 不负责：

- 维护客户或联系人主数据；
- 由客户端手填联系人引用；
- 自动录音或根据通话时长猜测联系结果；
- 根据设备日历自行生成可预约时段；
- 在改约时静默改派责任师傅；
- 把取消预约等同取消工单；
- 把师傅未签到自动认定为客户爽约；
- 用本机草稿覆盖服务器当前预约。

## 3. 访问和责任校验

在线读写必须同时满足：

```text
authenticated principal
+ trusted Technician Context
+ ACTIVE TechnicianProfile
+ ACTIVE NetworkTechnicianMembership
+ current ACTIVE technician responsibility
+ required capability
+ current task allowed-action
```

离线写入还必须满足：

- 当前账号和设备隔离有效；
- 有效工作包包含对应离线动作；
- 任务、责任、联系人快照和预约版本未过期；
- 命令依赖和幂等键完整；
- 本机未收到 tombstone、设备撤销或责任失效。

进入页面后责任发生变化时，立即停止拨号授权、隐藏客户敏感信息、禁用预约命令，并将未同步成果送入 IOS-010。

## 4. 页面结构

```text
任务摘要 + 同步可信度
→ 客户联系卡
→ 联系结果记录
→ 当前预约
→ 可预约时间窗
→ 提议 / 确认 / 改约 / 取消
→ 冲突、SLA 和通知影响
→ 联系与预约时间线
→ 固定底部操作
```

页面可以使用“联系记录”和“预约安排”两个页内区块，但不能拆成两套互不关联的任务事实。

## 5. 联系对象的权威来源

客户端不能提交任意 `contactedPartyRef`，也不能从手机号文本反推联系人。

服务端必须基于当前任务、工单客户快照和字段策略返回受控联系人：

```text
contactAccessToken or controlledContactRef
displayNameMasked
relationshipLabel
allowedChannels
validUntil
fieldPolicyVersion
```

联系人引用只用于服务端命令和短期系统拨号授权，不作为普通界面技术字段显示，也不写入剪贴板、日志或普通偏好。

联系人缺失、失效或字段策略不允许时，页面明确显示“当前没有可用的授权联系人”，并提供网点或平台协调入口，不能使用项目经理、历史号码或其他联系人兜底。

## 6. 客户联系卡

展示：

- 客户脱敏姓名；
- 联系人关系；
- 脱敏手机号；
- 允许联系的渠道；
- 客户联系偏好；
- 当前允许联系时间；
- 最近联系结果；
- 下次建议联系时间；
- “拨打电话”主入口。

默认不长期展示完整手机号。点击拨号前再次校验当前责任、字段策略和联系人授权，再将号码交给系统拨号能力。

禁止：

- 将完整号码复制到剪贴板；
- 在应用切换器快照中暴露完整号码；
- 将号码写入本地日志或埋点；
- 任务撤权后继续保留拨号入口；
- 未经明确合规要求自动录音或保存通话音频。

## 7. 系统拨号流程

```text
点击拨打电话
→ 服务端或有效工作包再次确认联系权限
→ 使用系统拨号能力
→ 返回 App
→ 提示记录真实联系结果
→ 本地保存或提交 ContactAttempt
```

以下事实不能自动推断：

- 用户点击拨号不等于电话已接通；
- 系统返回 App 不等于联系成功；
- 通话时长不等于客户确认预约；
- 师傅未填写结果不等于无人接听。

App 在拨号后被杀进程或长时间未返回时，可以保留“联系结果待补充”本机提醒，但不能创建正式联系事实。

## 8. 联系结果

联系记录只追加，不能修改或删除历史。

标准结果使用中文业务名称：

- 已联系；
- 无人接听；
- 占线；
- 联系方式有误；
- 客户要求稍后联系；
- 客户拒绝；
- 联系方式无效；
- 需要其他渠道跟进。

每次记录至少包含：

- 联系渠道；
- 业务发生时间 `capturedAt`；
- 联系开始和结束时间，可用时；
- 标准结果；
- 标准原因；
- 必要业务说明；
- 下次联系时间，可选；
- 权威联系人引用；
- 实际操作者；
- 稳定 `deviceCommandId`；
- 本机创建时间和服务器接收时间。

实际操作者来自认证主体，客户端不能伪造。联系人引用来自服务端受控任务快照，客户端不能手填。

## 9. 联系说明

业务说明必须配合标准结果使用，不能用一段自由文本替代结果分类。

页面应根据结果动态要求：

- 客户要求稍后联系：下次联系时间；
- 联系方式有误：错误类型和是否需要平台维护；
- 客户拒绝：拒绝原因和下一步协调；
- 需要其他渠道：建议渠道；
- 已联系：是否产生预约意向。

说明字段限制长度，并提示不得填写身份证、银行卡、与当前任务无关的隐私或技术信息。

## 10. 离线联系记录

有效工作包允许时，联系结果可以离线排队：

```text
本地已保存
→ 待同步
→ 服务器已接收
→ 联系事实已确认
```

离线联系记录必须携带：

- 工作包和责任版本；
- 权威联系人快照引用；
- `capturedAt`；
- 服务端时间偏移；
- 稳定 `deviceCommandId`；
- 标准结果和必要字段。

服务器恢复网络后重新校验责任和联系人策略。责任变化时不得把旧师傅联系事实写入新责任上下文，记录进入冲突或受控人工处理。

只有普通缓存而没有有效工作包时：在线状态可以重新取得短期拨号授权，但离线状态不提供号码或拨号入口，也不能保存正式联系命令。页面可以保留不含敏感信息的结果草稿，并提示联网后重新校验，不能伪造已提交。

## 11. 联系历史

页面展示最小安全历史：

- 联系时间；
- 渠道；
- 标准结果；
- 下次联系时间；
- 是否由本人或网点协同完成；
- 服务端确认状态。

默认不展示完整联系人引用、完整电话、录音、内部操作者 ID 或不必要的自由文本。

Network Portal、Admin 和 iOS 读取同一不可变 `ContactAttempt`。不同端重复提交相同幂等命令时返回既有结果，不创建重复联系记录。

## 12. 当前预约

当前预约卡展示：

- 预约类型；
- 当前状态；
- 当前时间窗口；
- 业务时区；
- 当前 Revision；
- 提议方；
- 确认主体和确认渠道；
- 实际操作人角色；
- 最近变更时间；
- 通知结果；
- 允许动作。

技术 Revision ID 和聚合 ID 不直接显示。用户可以看到“第 3 次调整”等业务表达。

当前预约与本机预约草稿必须并列区分，不能用草稿替换服务器卡片。

## 13. 可预约时间窗

时间窗由服务端综合以下事实生成：

- 工单绑定的预约政策和服务日历；
- 当前流程阶段；
- 当前责任网点和师傅；
- 师傅排班、请假和已有预约；
- 网点产能与停派状态；
- 最小提前量和最大预约范围；
- 服务时长；
- 项目时区；
- 特殊工作日、节假日和停服日期；
- 当前 Appointment Revision；
- 可用性生成时间和版本。

客户端不得读取设备日历后自行生成正式时段，也不得将过期缓存时段作为当前可用结果。

每个时段至少展示：

- 开始和结束时间；
- 业务时区；
- 是否与本人其他任务冲突；
- 产能或规则限制；
- 可用性截至时间；
- 选择后对 SLA 的影响摘要。

数据源不可用时显示“暂时无法获取可预约时间”，不得提供固定默认时段。

## 14. 本机预约草稿

师傅可以在本机准备：

- 预约类型；
- 期望时间窗口；
- 提议方；
- 客户意向；
- 确认渠道；
- 改约或取消原因；
- 必要说明。

本机草稿只表示输入已保存，不代表时段仍可用，也不代表客户、网点或平台已经确认。

离线展示：

```text
预约草稿已保存到本机
联网后需要重新校验时间和当前预约版本
```

不得显示“预约成功”或“客户已确认”。

## 15. 提议预约

提交提议前展示：

- 当前任务和责任；
- 选择的时间窗口；
- 可用性版本和有效期；
- 当前 Appointment Revision；
- 师傅日程冲突；
- SLA 影响；
- 提议方；
- 需要谁确认；
- 通知渠道。

预约提议形成新的不可变 Revision，并进入“待确认”。

离线时：

- 默认只保存本机草稿；
- 只有工作包明确允许、时段依据仍在有效期且项目政策允许时才可排队提议命令；
- 排队后显示“预约提议待同步”，不能显示已提议或已确认；
- 服务端仍需重新校验责任、时段和并发版本。

## 16. 确认预约

确认必须区分三个身份：

```text
actualActor       实际操作的师傅
confirmedSubject  被记录为确认预约的权威主体
confirmationChannel 客户或授权主体使用的确认渠道
```

实际操作者始终来自 JWT。确认主体必须来自服务端授权的联系人或业务主体，客户端不能手填引用，也不能把师傅本人伪装成客户。

项目政策要求客户确认时，必须具备可审计的确认渠道或凭据，例如：

- 已记录的联系结果；
- 客户短信或受控消息确认；
- 授权客服或网点代确认记录；
- 明确的外部业务确认引用。

如果项目政策允许师傅代录客户口头确认，页面必须明确显示“师傅代录客户确认”，同时保存实际操作者、客户主体、渠道、时间和授权依据。

离线确认默认只保存草稿。只有项目政策和有效工作包明确允许排队时，才可以进入“确认待同步”；在服务端接受前仍不能显示“客户已确认”。

## 17. 改约

改约创建新 Revision，不修改历史。

提交前必须展示：

- 当前预约时间窗；
- 新时间窗；
- 发起方和标准原因；
- 当前和剩余改约次数；
- 最晚允许改约时间；
- 当前师傅日程冲突；
- 是否需要重新客户确认；
- SLA 暂停、延续或重算结果；
- 通知对象和渠道；
- 当前 Revision 和可用性版本。

改约后原确认事实不会自动带到新 Revision。需要客户重新确认时，状态回到待确认。

改约不能静默改变责任师傅。新时间与当前师傅冲突时，只能选择其他有效时段、返回网点调度或创建协调待办。

离线改约默认保存草稿；服务端未接受前，今日任务、日程和客户通知仍以当前服务器预约为准。

## 18. 取消预约

取消必须记录：

- 取消方；
- 标准原因；
- 实际操作者；
- 取消时间；
- 当前 Revision；
- 对工单流程、SLA 和本人日程的影响；
- 是否需要客户或平台确认；
- 通知对象和结果。

取消预约不等于取消工单。取消后下一步由服务端流程和 allowed-actions 决定，可能需要重新联系、重新预约或进入异常协调。

离线取消只有在明确工作包策略允许时才可排队，并始终显示“取消待同步”；否则只能保存原因草稿。

## 19. 客户爽约

客户爽约必须同时满足：

- 当前预约已经确认；
- 预约窗口已经结束或项目规则允许在明确容差后判定；
- 当前责任仍属于本人；
- 已记录必要联系或到场核验事实；
- 当前用户拥有相应 allowed-action；
- 当前 Revision 未变化。

必须记录：

- 标准爽约原因；
- 相关联系尝试；
- 到场或 Visit 摘要，可用时；
- 证明材料，按规则；
- 客户通知结果；
- 下一步业务动作。

客户爽约、师傅迟到、师傅未到和双方无法联系是不同事实。App 不得仅因当前没有签到就自动认定客户爽约。

爽约只改变 Appointment，不自动取消工单或伪造 Visit。

## 20. Appointment Revision 与并发

时间窗口、地址快照、确认事实、改约原因、取消和爽约均形成不可变 Revision 或历史。

所有写命令携带：

- 当前 Appointment Version 或 ETag；
- 当前 Revision；
- 责任版本；
- 可用性版本，可适用时；
- 工作包版本，离线时；
- 稳定 `deviceCommandId`；
- 服务端时间偏移和 `capturedAt`。

发生并发变化时：

1. 拒绝旧命令；
2. 保留本机未提交输入；
3. 展示服务器最新预约和变化摘要；
4. 说明草稿中的哪些内容需要重新确认；
5. 进入 IOS-010 或在最新事实基础上重新提交；
6. 不提供“仍然覆盖”。

## 21. SLA 影响

页面展示服务端计算结果：

- 当前 SLA 是否继续计时；
- 是否进入已批准暂停；
- 改约后延续还是重算；
- 新截止时间；
- 预警和升级；
- 需要平台处理的例外。

客户端不计算或修改 SLA，也不因离线、通知失败或内部异常静默暂停 SLA。

本机草稿不改变 SLA。只有服务端接受预约业务命令后，相关 SLA 变化才成立。

## 22. 通知与提醒

预约命令结果和通知结果分别展示：

```text
预约业务事实已成立
客户通知发送中 / 已送达 / 失败待处理
```

通知失败不回滚已经成立的预约，也不能把已确认预约显示为未确认。

关键通知失败时创建明确协同待办。普通师傅只能看到业务说明和恢复入口，不能看到供应商凭据、HTTP 响应、内部消息 ID 或队列状态。

## 23. 联系和预约时间线

按业务时间展示：

- 联系尝试；
- 预约提议；
- 客户或授权主体确认；
- 改约；
- 取消；
- 爽约；
- 通知结果；
- 并发冲突和重新提交；
- 网点或平台协调结果。

每条记录展示中文事件、业务发生时间、实际操作者角色、确认主体或联系结果和服务端确认状态。

不展示完整联系人引用、内部 Actor ID、技术事件名、消息队列对象或数据库版本。

## 24. 服务器事实与本机状态

```text
服务器事实
├── 权威联系人和字段策略
├── ContactAttempt 历史
├── 当前 Appointment 与 Revision
├── 可预约时间窗
├── SLA 影响和 allowed-actions
└── 通知与协调结果

本机状态
├── 拨号后待补结果
├── 联系结果草稿
├── 预约草稿
├── 待同步命令
└── 并发冲突
```

本机状态只能叠加，不能替换服务器当前联系人、预约、确认、SLA 或通知事实。

## 25. 页面读模型与本机投影

建议服务端读模型：

```text
TechnicianContactAppointmentWorkspaceView
TechnicianAuthorizedContactView
TechnicianContactHistoryView
TechnicianCurrentAppointmentView
TechnicianAppointmentAvailabilityView
TechnicianAppointmentImpactView
TechnicianAppointmentTimelineView
TechnicianContactAppointmentAllowedActionView
```

建议本机投影：

```text
LocalPendingContactResult
LocalContactAttemptDraft
LocalAppointmentDraft
LocalAppointmentCommandSummary
LocalAppointmentConflictSummary
```

服务端直接返回中文业务名称、权威联系人、时区、时间窗、冲突、SLA 影响、通知结果和阻止原因。客户端不得调用多个底层接口自行重建预约规则。

## 26. 业务命令

建议 Technician 专用业务入口委托 Appointment 领域命令：

```text
RecordTechnicianContactAttempt
ProposeTechnicianAppointment
ConfirmTechnicianAppointment
RescheduleTechnicianAppointment
CancelTechnicianAppointment
MarkTechnicianCustomerNoShow
CreateAppointmentCoordinationRequest
```

客户端不能提交 tenant、project、network、operator 或任意联系人引用。服务端从可信上下文、当前任务责任和客户快照解析这些事实。

不提供：

```text
UpdateAppointment
SetAppointmentStatus
OverwriteAppointmentRevision
MarkContactSuccessfulFromDial
ConfirmAsCustomerWithoutEvidence
ForceAvailableTimeSlot
```

## 27. 离线命令和幂等

每条允许排队的离线命令必须拥有稳定 `deviceCommandId`、依赖关系、工作包版本、责任版本和 capturedAt。

重试同一命令返回既有结果，不创建重复 ContactAttempt 或 Appointment Revision。

命令顺序必须显式，例如：

```text
记录联系成功
→ 提交预约提议
→ 等待服务端结果
→ 客户确认
```

后续命令依赖前一命令服务端资源时，在依赖成立前保持队列等待，不能使用本地临时 ID 冒充正式资源。

## 28. 页面状态

必须覆盖：

- 有授权联系人；
- 无授权联系人；
- 联系权限受限；
- 拨号后待记录结果；
- 联系结果本地已保存；
- 联系结果待同步；
- 联系记录已确认；
- 无预约；
- 预约草稿；
- 待确认；
- 已确认；
- 改约草稿或待同步；
- 已取消；
- 已爽约；
- 可用时段加载中；
- 可用时段不可用或已过期；
- 日程冲突；
- 通知发送中或失败；
- 离线且工作包有效；
- 离线但只有普通缓存；
- Appointment 并发冲突；
- 责任被改派或撤销；
- 客户端版本不支持；
- 只读、无权限和加载失败。

## 29. 错误和恢复

错误使用中文业务说明和明确恢复建议：

- 联系人授权失效：刷新任务或联系网点；
- 预约版本变化：查看最新预约并重新确认；
- 时段已不可用：重新加载可预约时间；
- 当前师傅冲突：选择其他时段或联系网点调度；
- 责任变化：停止操作并处理本机成果；
- 通知失败：预约事实保留，进入通知协同；
- 离线命令被拒绝：进入 IOS-010；
- 服务不可用：保留草稿并稍后重试。

错误提供可复制 traceId，不显示异常类名、HTTP 状态码或内部字段。

## 30. 权限与敏感信息

至少校验：

- 查看受控联系人；
- 发起系统拨号；
- 查看最小联系历史；
- 记录联系尝试；
- 查看预约；
- 查看可用时段；
- 提议预约；
- 确认预约；
- 改约；
- 取消预约；
- 标记客户爽约；
- 发起协调。

各动作分别授权，不能因为可以查看预约就默认可以确认、改约或取消。

完整电话、联系人引用、联系说明和确认凭据按最小必要原则处理。登出、改派、撤权或设备撤销后，本地敏感数据和拨号授权立即失效。

## 31. 可访问性与现场可用性

- Dynamic Type 下时间窗、客户状态、联系结果和主操作不得截断；
- VoiceOver 清楚朗读脱敏联系人、预约状态、时间窗、时区、冲突和下一动作；
- 关键状态同时使用文字、图标和颜色；
- 拨号、保存结果和预约主操作触控区域不小于 44×44pt；
- 时间选择不能只依赖拖拽；
- 跨时区同时提供清楚文字说明；
- 弱网时保存草稿不阻塞返回任务详情；
- 预约确认和取消使用明确影响确认，不依赖瞬时 Toast。

## 32. 审计与可观测性

必须记录：

- 联系人字段策略和短期拨号授权；
- 实际操作者、权威联系人、渠道和联系结果；
- capturedAt、receivedAt 和服务端时间偏移；
- Appointment Revision、并发版本和可用性版本；
- 实际操作者、确认主体、确认渠道和授权依据；
- 改约、取消和爽约原因；
- SLA 影响；
- 通知和协调结果；
- 离线命令、幂等返回和冲突；
- 改派、撤权和本地敏感数据失效；
- traceId、correlationId、clientKind 和 clientVersion。

日志不得记录 Token、完整电话、完整地址、联系人内部引用、通话内容、录音、表单敏感值或通知供应商凭据。

## 33. 验收标准

1. 客户联系人来自服务端权威任务快照，客户端不能手填引用；
2. 默认只显示脱敏电话，拨号前重新校验当前责任和字段策略；
3. 点击拨号、返回 App 和通话时长都不会自动生成联系成功；
4. 联系结果使用标准分类，ContactAttempt 只追加不可修改；
5. 实际操作者来自认证主体，不能由客户端伪造；
6. 离线联系记录明确区分本地保存、待同步和服务器确认；
7. 可预约时间窗由服务端生成，过期或不可用时不使用固定兜底；
8. 本机预约草稿不会覆盖当前 Appointment 或显示预约成功；
9. 预约提议、客户确认、通知送达和平台确认分别表达；
10. actualActor、confirmedSubject 和 confirmationChannel 分别保存；
11. 师傅不能伪装成客户确认预约；
12. 改约形成新 Revision，不静默继承旧确认或改派师傅；
13. 取消预约不等于取消工单；
14. 客户爽约、师傅迟到、师傅未到和无法联系分别表达；
15. 预约窗口未结束时不能标记客户爽约；
16. 并发变化拒绝旧命令并保留本机草稿，不允许覆盖；
17. 通知失败不回滚已经成立的预约事实；
18. 离线命令使用稳定 deviceCommandId，重试不产生重复联系或预约修订；
19. 改派、撤权或设备撤销后联系人、预约命令和本地敏感访问立即失效；
20. 页面不展示内部 ID、英文枚举、异常堆栈或技术队列状态。

## 34. 与其他规格的关系

- IOS-001 今日任务：展示待联系、待预约和预约风险入口；
- IOS-002 任务详情：展示当前预约、最近联系和下一动作摘要；
- IOS-004 到场与开始作业：消费已确认预约并执行 Visit 到场；
- IOS-005 作业步骤：处理客户拒绝、无法施工和现场异常；
- IOS-009 离线恢复：恢复联系、预约草稿和离线命令；
- IOS-010 同步冲突：处理责任、预约 Revision、时段和联系人策略冲突；
- NW-005 今日预约：网点端读取和维护同一 Appointment 与 ContactAttempt；
- AD-008 项目 SLA 与预约：定义预约政策、服务日历和 SLA 影响；
- DEC-005：责任师傅和当前任务处理人独立建模；
- DEC-006：改派和失权后旧师傅命令必须失败关闭。
