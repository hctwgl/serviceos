# 工单工作区状态矩阵

状态：Draft  
视觉方向：A+ 经典专业任务型

## 1. 目的

工单工作区不能只设计“正常履约”状态。实现前必须覆盖项目协同人员、责任网点、责任师傅、当前任务、预约、现场履约、审核整改和外部协同的关键状态，并明确首屏信息、允许动作、阻塞原因和恢复方式。

统一责任模型见 `decisions/DEC-005-work-order-responsibility-and-assignment-boundaries.md`。

## 2. 责任状态矩阵

| 状态 | 首屏必须显示 | 主操作 | 禁止行为 |
|---|---|---|---|
| 项目人员待确认 | 缺失的客服经理、项目经理或项目助理，命中区域，受影响动作 | 调整项目关联人员 | 不使用系统管理员、项目创建人或其他区域人员兜底 |
| 待分配责任网点 | 服务地址、标准行政区域、服务类型、配置版本、候选摘要、SLA | 分配责任网点 | 不自动选择默认网点，不把项目经理当作责任网点 |
| 无合格网点候选 | 覆盖、准入、服务类型、产能或数据不可用的具体原因 | 创建协调待办或处理异常 | 不扩大服务区域，不使用其他项目网点 |
| 责任网点已分配，待分配师傅 | 责任网点、师傅能力要求、预约时限、候选摘要 | 分配责任师傅 | 不从其他网点直接选择师傅，不自动选择第一个候选 |
| 无合格师傅候选 | 技能、资质、排班、负载或数据不可用的具体原因 | 等待产能、改派网点或转异常 | 不忽略技能资质，不使用无效人员 |
| 责任网点和师傅已分配 | 两层责任、选择依据、分配时间、下一流程任务 | 进入下一业务动作 | 不把当前任务处理人和责任师傅混为一人 |
| 当前任务处理人待确认 | 当前任务、所需处理对象、缺失原因、截止时间 | 指定或处理缺失责任 | 不静默转给前一处理人或管理员 |
| 改派网点待确认 | 原网点、原师傅、新网点、预约、SLA、任务和通知影响 | 确认改派网点 | 不保留原网点师傅作为新网点当前师傅 |
| 改派师傅待确认 | 原师傅、新师傅、责任网点、预约和任务影响 | 确认改派师傅 | 不改变责任网点，不覆盖原师傅已完成事实 |
| 现场作业已开始，普通改派被阻止 | 当前现场状态、已完成步骤、交接风险 | 进入异常或交接流程 | 不通过普通改派直接替换现场责任 |

## 3. 履约状态矩阵

| 状态 | 首屏必须显示 | 主操作 | 禁止行为 |
|---|---|---|---|
| 已派师傅待预约 | 责任网点、责任师傅、客户联系方式、预约时限 | 确认预约 | 不绕过预约政策、师傅可用性或服务范围校验 |
| 已预约待上门 | 预约时间、地址、联系人、网点和师傅、变更记录 | 调整预约 | 不静默覆盖客户已确认时间 |
| 履约进行中 | 当前步骤、已完成步骤、待提交资料、现场执行人、同步状态 | 查看现场进度 | Admin 不代替师傅完成定位、拍照、签名等现场动作 |
| 待审核 | 提交人、提交快照、审核人、检查项、截止时间 | 进入审核 | 不把接口成功或自动校验通过等同人工审核通过 |
| 整改中 | 原审核结论、问题项、责任对象、截止时间、整改轮次 | 跟踪或处理整改 | 不隐藏原问题，不创建平行整改单 |
| 待复核 | 整改提交快照、复核人、原问题和截止时间 | 进入复核 | 不直接覆盖原审核和整改记录 |
| SLA 高风险 | 剩余时间、风险来源、当前阻塞、责任链和升级对象 | 处理风险 | 不只显示红色标签，必须给出下一步 |
| 流程阻塞 | 当前节点、阻塞条件、依赖责任或资料、恢复入口 | 处理阻塞 | 不静默跳过节点或伪造完成条件 |
| 外部协同异常 | 外部协同方、业务状态、失败原因、尝试记录和允许动作 | 处理协同异常 | 不展示技术队列替代业务原因，不编辑冻结回传事实 |
| 已完成 | 完成时间、结果、关键资料、配置版本和审计记录 | 查看结果 | 不继续提供改变履约结果的普通动作 |
| 已取消 | 取消原因、操作人、影响范围、外部通知结果 | 查看记录 | 不直接恢复为进行中，除非另有明确重开流程 |

## 4. 通用页面状态

| 状态 | 首屏必须显示 | 主操作 | 禁止行为 |
|---|---|---|---|
| 无权限 | 用户可见的工单摘要和权限说明 | 返回列表 | 不泄露敏感字段、权限代码或内部 ID |
| 只读 | 只读原因、当前责任链和业务状态 | 查看相关记录 | 不以大量置灰按钮模拟可编辑状态 |
| 并发冲突 | 已变化的业务对象、最新操作人和时间 | 刷新并重新确认 | 不自动覆盖服务端最新责任或状态 |
| 数据不完整 | 缺失的业务字段、受影响模块和追踪标识 | 重试或返回 | 不用 ID、英文枚举或空字符串兜底 |
| 关联模块部分不可用 | 不可用模块、最后成功更新时间、其他可用内容 | 重试对应模块 | 不把未知数据显示为零或正常 |
| 系统错误 | 中文错误标题、业务影响、操作追踪标识 | 重试或返回 | 不展示堆栈、SQL、Java 类名或第三方原始错误 |

## 5. 页面共同结构

所有状态保持稳定的对象头部：

- 工单编号；
- 客户与项目；
- 工单绑定的履约配置版本；
- 当前阶段和当前任务；
- 客服经理、项目经理、项目助理完整性摘要；
- 责任网点；
- 责任师傅；
- 当前任务处理人；
- 预约和 SLA；
- 当前最高优先级阻塞；
- 主要允许动作。

主内容区根据当前状态突出当前任务，不把所有模块平均展示。右侧上下文栏固定承载联系人、地址、责任链、风险、数据更新时间和下一步建议。

## 6. 动作规则

所有动作来自服务端 `allowed-actions`。前端不得自行推导可操作性。

不可执行动作原则上不进入主操作区；用户需要理解原因时，展示中文阻塞解释，包括：

- 当前状态；
- 缺失条件；
- 当前责任对象；
- 需要的权限；
- 建议处理入口。

派网点、派师傅、改派网点和改派师傅使用不同动作和确认页面，不提供通用“修改负责人”。

## 7. 原型评审要求

仅在复杂交互仍有理解分歧时生成少量最终原型，优先评审：

1. 待分配责任网点；
2. 责任网点已分配、待分配师傅；
3. 改派网点影响确认；
4. 现场作业已开始、普通改派被阻止；
5. SLA 高风险且存在责任缺失；
6. 审核驳回与整改；
7. 并发冲突和数据部分不可用。

原型必须使用一致的 A+ 设计语言，并标注责任来源、主操作、状态来源和服务端读模型。
