# AD-001 Admin 工作台

状态：Draft for Product Review  
设计等级：A

## 1. 产品位置

Admin 工作台是平台运营人员登录后的任务首页。

它负责把用户权限范围内需要处理的工单、审核、整改、派单、外部协同和项目配置问题聚合为可行动的任务入口。

它不是：

- 第二套项目列表；
- 第二套工单中心；
- “配置中心”；
- 项目履约配置编辑器；
- 统计报表首页；
- 系统监控或技术诊断控制台。

所有业务事实仍由对应业务模块维护：

```text
工作台任务投影
├── 工单风险 → Admin 工单中心 / 工单工作区
├── 派单待办 → 派单与改派
├── 审核整改 → 工单审核与整改任务
├── 配置待办 → 项目详情 / 履约配置
└── 外部协同异常 → 工单外部协同记录或受控诊断
```

工作台只提供摘要、排序、解释和深链接，不复制业务对象，也不在首页实现另一套状态流转。

## 2. 目标用户

- 平台运营人员；
- 履约运营负责人；
- 审核和异常管理负责人；
- 项目配置和发布人员；
- 具有全局、组织或项目范围数据权限的管理人员。

工作台必须严格按照当前用户的数据范围和操作权限返回内容。前端不得先获取全量数据再自行隐藏无权内容。

## 3. 核心任务

用户进入系统后，应在一分钟内判断：

- 今天最需要优先处理什么；
- 哪些工单即将或已经违反 SLA；
- 哪些工单等待派单、审核、整改或异常处理；
- 哪些项目配置阻止项目正常接单或发布；
- 哪些外部协同失败需要人工介入；
- 自己下一步应进入哪个业务工作区。

工作台必须回答：

> 当前最紧急的问题是什么，为什么需要我处理，我应进入哪里完成处理。

## 4. 页面结构

### 4.1 页面头部

页面头部展示：

- 标题：工作台；
- 当前数据范围；
- 数据生成时间；
- 数据新鲜度状态；
- 刷新；
- 工作台视图设置入口，可选。

“当前数据范围”必须使用业务名称，例如“全部项目”“华东区域项目”“我可管理的项目”，不能展示权限代码、组织 ID 或技术范围表达式。

工作台数据不是实时交易承诺。页面必须展示生成时间，并对过旧数据明确提示。

### 4.2 今日优先处理

首屏使用一张主任务卡或紧凑任务列表表达真正需要立即处理的事项。

至少聚合：

- 已超时工单；
- 两小时内可能超时工单；
- 无责任网点或无责任师傅；
- 项目岗位人员缺失；
- 流程阻塞；
- 发布失败且尚未解决的项目配置；
- 无生效配置但已经接近服务开始时间的项目；
- 对方拒绝或重试用尽的关键外部回传。

每项必须提供：

- 业务对象；
- 风险原因；
- 严重程度；
- 发生或持续时间；
- 当前责任范围；
- 推荐下一步；
- 明确进入入口。

点击后进入已经应用相应筛选或定位到具体问题的业务页面。

### 4.3 待办摘要

使用可点击的摘要卡展示：

- 待派单；
- 待审核；
- 待整改；
- 待处理异常；
- 配置待办；
- 外部协同待处理。

摘要卡必须显示：

- 待办数量；
- 其中高风险数量；
- 最早到期或持续时间最长的事项；
- 数据更新时间；
- 点击后的目标业务视图。

摘要数据不可用时显示“暂时无法获取”，不得显示为零。

### 4.4 我的待办

按照当前用户可处理的业务责任展示：

- 需要我处理的工单任务；
- 需要我审核或复核的任务；
- 需要我完成的整改或异常处理；
- 需要我确认的发布警告；
- 我有权限发布且已准备完成的项目配置；
- 需要我处理的外部协同异常。

“我的待办”只显示服务端已经判断当前用户可处理的事项。前端不得根据岗位名称、项目成员关系或页面权限自行推断任务归属。

### 4.5 风险与预警

按业务严重程度展示：

- SLA 即将超时；
- 已超时未处理；
- 流程阻塞；
- 长时间无人负责；
- 项目配置阻止接单或发布；
- 项目服务区域没有可用网点候选；
- 审核或整改超过时限；
- 外部工单接收失败；
- 外部回传被拒绝或重试用尽；
- 客户投诉或高风险标记。

每项必须给出明确业务动作，例如：

- 立即处理；
- 查看阻塞原因；
- 补充项目人员；
- 修复配置错误；
- 继续发布准备；
- 查看对方拒绝原因。

### 4.6 配置待办

配置待办是工作台中的跨项目任务投影，不是独立“配置中心”。

它的权威来源为每个项目的履约配置生命周期、校验、影响分析和发布状态。

点击任何配置待办，都必须进入：

```text
客户与项目 / 对应项目 / 履约配置 / 对应问题位置
```

不得进入第二套“项目配置详情”。

#### 4.6.1 待办类型

当前阶段支持：

1. 待首次配置；
2. 有未发布草稿；
3. 草稿保存失败；
4. 存在阻塞错误；
5. 校验结果已过期；
6. 等待影响分析；
7. 影响分析已过期；
8. 等待警告确认；
9. 已具备发布条件；
10. 发布处理中；
11. 发布失败；
12. 外部依赖不可用并阻止校验或发布。

“最近发布成功”属于最近活动，不属于待办。

#### 4.6.2 一行代表什么

配置待办列表一行代表：

> 一个项目当前最需要处理的一项履约配置问题或发布动作。

同一项目存在多个校验错误时，工作台聚合为一条主要待办，显示阻塞错误数量和最高优先级问题。进入项目履约配置后再查看完整错误列表。

工作台不得为同一项目的每个字段错误生成大量重复待办。

#### 4.6.3 列表字段

至少展示：

- 项目名称；
- 客户名称；
- 项目状态；
- 服务开始时间或当前业务阶段；
- 当前生效版本；
- 当前草稿及其基线版本；
- 待办类型；
- 主要问题；
- 其他阻塞错误或警告数量；
- 最近保存人和时间；
- 最近发布人和时间；
- 数据更新时间；
- 推荐下一步；
- 当前用户允许执行的入口动作。

不展示：

- 原始项目 ID；
- 草稿内部 ID；
- 并发版本号；
- 校验规则内部编码；
- 发布任务技术状态；
- 完整异常堆栈。

#### 4.6.4 筛选与排序

至少支持：

- 待办类型；
- 项目名称；
- 客户名称；
- 项目状态；
- 是否存在生效版本；
- 是否存在阻塞错误；
- 是否已具备发布条件；
- 最近更新时间；
- 当前用户是否具有处理权限。

默认排序优先考虑业务影响，而不是简单按修改时间：

1. 发布失败且影响项目接单；
2. 无生效配置且临近或已经进入服务期；
3. 草稿保存失败；
4. 存在阻塞错误；
5. 已具备发布条件；
6. 等待警告确认或影响分析；
7. 普通未发布草稿；
8. 待首次配置但尚未临近服务期。

服务端必须返回排序依据和优先级说明，前端不得自行拼装优先级。

#### 4.6.5 允许操作

工作台配置待办只提供导航型动作：

- 开始配置；
- 继续草稿；
- 修复错误；
- 重新校验；
- 执行影响分析；
- 确认警告；
- 进入发布确认；
- 查看发布失败；
- 查看生效版本；
- 查看发布历史。

工作台不允许：

- 新建或删除项目；
- 直接编辑流程、表单、SLA 或映射；
- 在列表中修改配置状态；
- 绕过项目页面直接发布；
- 批量发布多个项目；
- 批量放弃草稿；
- 修改项目成员或区域分工。

“进入发布确认”仍然进入对应项目的版本与发布页面，并执行 AD-013 定义的完整最终检查。

#### 4.6.6 空状态

没有配置待办时显示：

- 当前范围内暂无需要处理的项目配置；
- 最近一次数据更新时间；
- 可进入项目列表查看全部项目。

不得显示“新建配置”按钮，也不得引导建立独立配置对象。

### 4.7 最近活动

最近活动用于解释近期发生的重要变化，例如：

- 项目履约配置发布成功；
- 发布失败；
- 高风险配置错误被修复；
- 工单重大异常已解决；
- 审核或整改完成；
- 外部协同恢复。

它是只读活动流，不与待办混合计数。点击活动进入对应业务记录。

### 4.8 趋势和分布

只展示能支持运营判断的趋势：

- 近七日新建和完成工单趋势；
- 当前工单阶段分布；
- SLA 风险分布；
- 项目或网点异常排行；
- 配置待办类型分布，可选。

图表不是首屏首要内容。不存在明确行动价值的装饰性图表不进入当前阶段。

## 5. 优先级与推荐动作

首要动作由服务端根据用户权限范围内的业务风险计算，不固定为“新建工单”。

默认业务优先级为：

1. 已经产生客户或履约影响的高风险事项；
2. 已超时或即将超时；
3. 无负责人、无网点或流程阻塞；
4. 发布失败、保存失败或无法接单的项目配置；
5. 等待审核、整改、确认或发布；
6. 普通运营待办。

工作台必须展示排序原因，例如：

> 该项目已进入服务期，但尚无生效履约配置。

不能只返回不可解释的分数。

## 6. 深链接规则

每个任务项必须由服务端返回稳定业务目标和允许动作。

深链接必须：

- 进入用户有权访问的正式业务页面；
- 保留工作台当前数据范围和返回位置；
- 尽可能定位到具体工单、流程节点、配置模块或错误；
- 目标已处理时显示最新业务状态，而不是继续显示过期待办；
- 目标无权访问时显示明确权限说明；
- 目标不存在时显示业务记录已失效或已删除，不显示技术 404。

前端不得根据任务类型拼接内部路由或对象 ID。

## 7. 数据刷新与一致性

工作台是聚合读模型，允许与业务页面存在短暂刷新延迟，但必须可解释。

统一规则：

- 每个模块展示数据生成时间；
- 用户点击任务进入业务页面时，以业务页面最新状态为准；
- 已处理任务在下次刷新后从待办消失；
- 工作台不得通过本地操作先行伪造任务已完成；
- 某个来源不可用时只标记对应模块失败，不影响其他模块展示；
- 数据过旧时不得继续显示为“实时”或“最新”。

配置待办尤其不得缓存成为第二套配置状态。项目履约配置状态始终是唯一事实源。

## 8. 页面读模型

建议使用：`AdminWorkbenchView`

必须包含：

- `scope`：当前数据范围及业务名称；
- `generatedAt`：整体数据生成时间；
- `freshness`：数据新鲜度；
- `prioritySummary`：今日优先处理摘要；
- `priorityItems`：高优先事项；
- `taskSummaries`：各类待办摘要；
- `myTasks`：当前用户可处理待办；
- `riskItems`：风险与预警；
- `configurationTaskSummary`：配置待办摘要；
- `configurationTasks`：配置待办列表；
- `recentActivities`：最近活动；
- `trendSeries`：经聚合的趋势数据；
- `distributionSeries`：分布数据；
- `moduleAvailability`：各聚合模块是否可用；
- `allowedActions`：页面级允许动作。

`configurationTasks` 每项至少提供：

- 项目和客户中文业务上下文；
- 待办类型和中文说明；
- 优先级及其解释；
- 当前生效版本和草稿摘要；
- 阻塞错误与警告摘要；
- 推荐下一步；
- 正式业务目标；
- 当前用户允许动作；
- 数据生成时间。

所有业务对象必须同时提供稳定标识和中文展示名称。页面不允许使用 ID、技术状态或英文枚举兜底。

## 9. 业务查询与命令

工作台主要使用查询，不拥有配置编辑或业务任务流转命令。

至少需要：

- 查询当前用户工作台；
- 按数据范围重新查询；
- 查询更多优先事项；
- 查询更多配置待办；
- 刷新指定工作台模块；
- 获取某一任务的最新导航目标。

所有实际处理动作必须跳转到对应业务工作区执行。

当前阶段不提供：

- 从工作台直接发布配置；
- 从工作台直接改派工单；
- 从工作台直接完成审核或整改；
- 通过批量接口修改多个领域对象；
- 在工作台保存第二套任务完成状态。

## 10. 权限

至少区分：

- 查看工作台；
- 查看跨项目工单摘要；
- 查看审核和整改摘要；
- 查看配置待办；
- 查看外部协同异常；
- 查看跨项目趋势；
- 进入具体业务对象。

能看到配置待办不代表能编辑或发布项目配置。每条任务的动作必须根据目标页面权限返回。

无权查看项目名称或客户信息时，该项目不应进入当前用户工作台，而不是使用脱敏占位项目制造不可操作待办。

## 11. 必须设计的页面状态

- 正常有数据；
- 当前范围无任何待办；
- 仅存在配置待办；
- 配置待办为空；
- 用户无业务数据范围；
- 部分聚合模块暂不可用；
- 配置状态来源暂不可用；
- 页面整体加载失败；
- 数据生成中；
- 数据更新时间过旧；
- 只读模式；
- 深链接目标已处理；
- 深链接目标无权限；
- 深链接目标已失效。

部分模块失败时必须明确标记失败模块和最后成功更新时间，不得显示为零。

## 12. 当前阶段不包含

- 独立“配置中心”；
- 第二套项目配置列表；
- 工作台内联编辑履约配置；
- 工作台直接发布；
- 多项目批量发布；
- 用户自定义任意工作台查询脚本；
- 技术监控、日志、队列或接口诊断首屏；
- 无行动价值的装饰性图表；
- 演示数据初始化入口。

## 13. 验收标准

- 首屏首先呈现任务和风险，不是导航入口集合；
- 用户可以从摘要进入已筛选或已定位的正式业务页面；
- 配置待办只投影项目履约配置状态，不建立第二套项目或配置对象；
- 配置待办不能直接编辑或绕过 AD-013 发布；
- 同一项目的多个配置问题在工作台合理聚合；
- 任务排序具有中文业务解释；
- 用户只能看到其权限范围内且可理解的业务对象；
- 部分数据失败不会被显示为零；
- 工作台数据过旧时有明确提示；
- 目标业务页面状态与工作台不一致时，以业务页面为准；
- 不展示环境、客户端版本、Capability、Portal Context；
- 不展示 UUID、英文状态、内部字段、认证秘密或技术诊断信息；
- 1280 和 1440 宽度下主要内容不横向溢出；
- 200% 缩放下仍可完成主要操作；
- 空状态不引导用户创建独立配置或初始化演示数据。
