# AD-001 Admin 工作台

状态：Draft for Product Review  
设计等级：A

## 1. 目标用户

- 平台运营人员；
- 履约运营负责人；
- 审核和异常管理负责人；
- 具有全局或项目范围数据权限的管理人员。

## 2. 核心任务

用户进入系统后，应在一分钟内判断：

- 今天最需要优先处理哪些事项；
- 哪些工单即将或已经违反 SLA；
- 哪些事项等待审核或整改；
- 哪些工单无人负责或流程阻塞；
- 自己下一步应进入哪个工作队列。

## 3. 页面结构

### 3.1 页面头部

- 标题：工作台；
- 当前数据范围；
- 数据更新时间；
- 自定义工作台入口。

### 3.2 今日优先处理

使用一张主卡表达真正需要立即处理的事项：

- 优先事项总数；
- 两小时内可能超时数量；
- 已超时数量；
- 点击后进入已应用筛选条件的工单中心。

### 3.3 辅助摘要

- 待审核；
- 待整改；
- 待派单；
- 等待外部处理。

卡片必须支持点击进入对应业务视图，不能只展示统计数字。

### 3.4 风险与预警

按严重程度展示：

- SLA 即将超时；
- 已超时未处理；
- 流程阻塞；
- 长时间无人负责；
- 外部回传失败；
- 客户投诉或高风险标记。

每项必须提供明确动作，如“立即处理”“查看阻塞原因”。

### 3.5 我的待办

按照用户责任范围展示：

- 我负责的工单；
- 我负责的审核；
- 我发起的整改；
- 需要我确认的发布；
- 需要我处理的异常。

### 3.6 趋势和分布

仅展示能支持判断的趋势：

- 近七日工单趋势；
- 当前阶段分布；
- 风险等级分布；
- 项目或网点异常排行。

不展示没有行动价值的装饰性图表。

## 4. 首要动作

首要动作由用户当前最紧急事项决定，不固定为“新建工单”。

默认动作优先级：

1. 处理已超时；
2. 处理两小时内超时；
3. 处理无人负责；
4. 处理阻塞；
5. 处理普通待办。

## 5. 页面读模型

建议使用：`AdminWorkbenchView`

必须包含：

- `scope`：当前数据范围；
- `generatedAt`：数据生成时间；
- `prioritySummary`：今日优先处理摘要；
- `taskSummaries`：各类待办摘要；
- `riskItems`：风险列表；
- `myTasks`：当前用户待办；
- `trendSeries`：经聚合的趋势数据；
- `distributionSeries`：分布数据；
- `allowedActions`：页面级允许动作。

所有业务对象必须同时提供稳定 ID 和中文展示名称，不允许前端使用 ID 兜底。

## 6. 状态

必须设计：

- 正常有数据；
- 当前范围无待办；
- 用户无业务范围；
- 部分统计数据暂不可用；
- 页面数据整体加载失败；
- 只读模式；
- 数据更新时间过旧。

部分模块失败时必须明确标记失败模块，不得显示为零。

## 7. 验收标准

- 首屏首先呈现任务和风险，不是导航入口；
- 用户可以从任意摘要直接进入已筛选的工作队列；
- 不展示环境、客户端版本、Capability、Portal Context；
- 不展示 UUID、英文状态或内部字段；
- 1280 和 1440 宽度下主要内容不横向溢出；
- 200% 缩放下仍可完成主要操作；
- 空状态不引导用户初始化演示数据。
