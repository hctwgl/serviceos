# AD-003 工单工作区

状态：Draft for Product Review  
设计等级：A

## 1. 产品位置

工单工作区是单张工单的权威业务工作页面。

它负责让用户在一个明确上下文中完成：

- 判断工单是什么、进行到哪里；
- 查看项目协同人员和履约执行责任；
- 查看当前流程任务、风险和阻塞；
- 进入派单、预约、审核、整改和外部协同等正式操作；
- 查看完整业务历史和操作结果。

它不是：

- 超长字段详情页；
- 项目配置编辑器；
- 网点、师傅或项目人员主数据维护页面；
- 技术消息、日志或流程引擎调试台；
- 前端自行拼装业务状态的聚合页。

正式关系：

```text
工单中心 / 工作台
        ↓
工单工作区
        ├── 派单与改派
        ├── 预约与履约
        ├── 审核与整改
        ├── 项目关联人员调整
        └── 外部协同处理
```

## 2. 目标用户

- 平台运营人员；
- 项目运营人员；
- 平台调度人员；
- 审核和整改人员；
- 具有工单查看或处理权限的管理人员。

页面必须按照当前用户的数据范围、敏感信息权限和行级允许动作返回内容。

## 3. 核心判断

用户进入页面后，应在首屏判断：

- 这是什么工单；
- 当前绑定哪个项目和履约配置版本；
- 当前处于哪个流程阶段和任务；
- 客服经理、项目经理和项目助理是谁；
- 当前责任网点和责任师傅是谁；
- 当前任务由谁处理；
- 是否存在 SLA、人员、派单、预约、资料、审核、整改或外部协同风险；
- 为什么被阻塞；
- 自己下一步可以执行什么动作。

页面必须回答：

> 当前责任链是否完整，工单为什么停在这里，下一步由谁做什么。

## 4. 工单责任链

工单工作区必须区分四层责任：

```text
项目协同人员快照
├── 客服经理
├── 项目经理
└── 项目助理

履约组织责任
└── 责任网点

现场执行责任
└── 责任师傅

流程任务责任
└── 当前任务处理人
```

### 4.1 项目协同人员

- 由“项目 + 工单标准行政区编码”在工单创建时匹配；
- 保存为工单项目岗位人员快照；
- 页面打开时不重新计算；
- 区域分工变化不静默修改已有工单；
- 调整使用独立“调整项目关联人员”命令。

### 4.2 责任网点

- 由平台派单确定；
- 必须来自工单绑定配置版本和当前运营数据生成的合格候选；
- 代表承担该工单履约组织责任的网点；
- 项目经理不能代替责任网点。

### 4.3 责任师傅

- 由责任网点或具有明确跨组织调度权限的人员确定；
- 当前阶段原则上必须属于责任网点；
- 必须满足工单绑定版本定义的技能和资质要求；
- 责任网点变化时，原师傅责任必须在同一改派命令中明确结束。

### 4.4 当前任务处理人

- 由工单绑定的流程版本和当前运行状态决定；
- 可以是项目岗位人员、网点、师傅、审核人、人工指定人员或系统任务；
- 不等同于责任网点或责任师傅；
- 同一工单可以有稳定责任网点和师傅，但当前任务由项目经理或审核人处理。

四层责任不能互相静默兜底。

## 5. 页面结构

### 5.1 对象头部

必须展示：

- 工单编号；
- 客户名称；
- 项目名称；
- 服务类型；
- 外部单号，可选；
- 当前阶段；
- 当前业务状态；
- 当前任务；
- 当前任务处理人；
- 责任网点；
- 责任师傅；
- 预约摘要；
- SLA 风险和截止时间；
- 当前最高优先级阻塞；
- 工单绑定的履约配置版本；
- 页面首要动作；
- 更多操作。

对象头部不能展示内部流程实例 ID、任务 ID、配置内部 ID 或英文技术状态。

### 5.2 责任与风险摘要

对象头部下方使用紧凑摘要明确展示：

- 项目岗位人员是否完整；
- 是否已分配责任网点；
- 是否已分配责任师傅；
- 当前任务是否有明确处理人；
- 是否存在即将超时或已超时；
- 是否存在阻塞；
- 当前推荐下一步。

缺失状态必须使用明确文案：

- 缺少项目经理；
- 待分配责任网点；
- 责任网点已确定，待分配师傅；
- 当前任务处理人待确认。

不得统一显示为“无人负责”。

### 5.3 主内容区

按业务工作顺序组织：

1. 履约进度；
2. 当前任务；
3. 派单与责任；
4. 预约信息；
5. 现场作业；
6. 表单与资料；
7. 审核与整改；
8. 外部协同；
9. 操作记录。

页面默认突出当前阶段和当前任务，不把所有模块以相同权重平铺。

### 5.4 右侧上下文区

持续展示：

- 客户和联系人；
- 服务地址和标准行政区域；
- 服务对象摘要；
- 项目岗位人员快照；
- 责任网点和责任师傅；
- 当前任务处理人；
- SLA 依据；
- 当前阻塞原因；
- 下一步建议；
- 关键数据更新时间。

### 5.5 动作区

动作来自服务端 `allowed-actions`，例如：

- 分配责任网点；
- 改派责任网点；
- 分配责任师傅；
- 改派责任师傅；
- 调整项目关联人员；
- 确认预约；
- 调整预约；
- 进入当前任务；
- 提交审核；
- 审核通过；
- 退回补充；
- 发起整改；
- 提交整改；
- 复核；
- 重新发送外部回传；
- 标记外部已确认；
- 转交异常处理；
- 请求取消或取消工单。

页面只显示当前最重要的一个首要动作。其他动作进入次级操作区。

不允许执行的动作应说明：

- 当前状态为什么不允许；
- 缺少什么条件；
- 应由谁处理；
- 下一步进入哪里。

前端不得自行推断动作权限或业务可执行性。

## 6. 履约进度

履约进度同时表达：

- 已完成阶段；
- 当前阶段；
- 后续阶段；
- 阻塞阶段；
- 每个阶段的业务责任方；
- 关键完成时间；
- 退回、整改和异常分支；
- 当前绑定的流程版本。

点击阶段可查看该阶段的任务、处理人、表单资料、SLA、审核结果和操作记录。

不得只展示技术流程节点名称或状态码。

## 7. 当前任务

当前任务区域至少展示：

- 任务业务名称；
- 任务目的；
- 任务来源阶段；
- 当前处理对象；
- 实际处理人；
- 创建时间；
- 截止时间；
- 当前状态；
- 完成条件；
- 必填表单和资料；
- 阻塞原因；
- 当前允许动作。

如果当前任务依赖项目经理、责任网点或责任师傅，但相应责任缺失，页面必须明确显示依赖关系和处理入口。

## 8. 派单与责任

“派单与责任”区域展示：

- 当前责任网点；
- 网点分配时间；
- 分配人；
- 分配依据摘要；
- 当前责任师傅；
- 师傅分配时间；
- 分配人；
- 能力匹配摘要；
- 责任变更历史；
- 当前派单状态；
- 候选数据更新时间；
- 允许动作。

派单状态使用：

- 待分配责任网点；
- 责任网点已分配，待分配师傅；
- 责任网点和师傅已分配；
- 网点候选不可用；
- 师傅候选不可用；
- 改派处理中；
- 责任变更待确认；
- 当前阶段不需要师傅。

点击分配或改派进入 AD-004 定义的上下文工作区。工单工作区不自行实现第二套候选筛选逻辑。

## 9. 项目岗位人员快照

必须展示：

- 客服经理；
- 项目经理；
- 项目助理；
- 每个岗位的匹配状态；
- 实际命中的行政区域；
- 是否通过上级区域继承；
- 是否使用备份负责人；
- 匹配时间；
- 最近人工调整记录。

人员缺失时显示“项目人员待确认”，列出具体岗位和受影响动作。

禁止：

- 页面实时重新按地址匹配；
- 使用系统管理员或项目创建人兜底；
- 因派单或改派网点而自动替换项目岗位人员。

## 10. 预约与现场作业

预约区域展示：

- 当前预约状态；
- 预约时间和时区；
- 客户确认；
- 网点和师傅可用性摘要；
- 改约历史；
- 预约与 SLA 的关系；
- 当前允许动作。

现场作业区域展示：

- 作业步骤；
- 当前步骤；
- 到场和开始时间；
- 当前现场执行人；
- 表单资料完成度；
- 离线或同步状态摘要；
- 异常和暂停原因。

Admin 不代替师傅完成受设备、定位、拍照或签名能力约束的现场动作。

## 11. 审核与整改

展示同一工单的权威运行记录：

- 当前审核任务；
- 本次提交快照；
- 审核人和截止时间；
- 检查项和问题；
- 审核结论；
- 整改记录；
- 整改责任对象；
- 整改截止时间；
- 整改轮次；
- 复核任务和结论；
- 逾期与升级记录。

Admin、网点端和 iOS 端展示同一任务和记录，不复制平行审核或整改状态。

## 12. 外部协同

使用业务语言展示：

- 外部协同方；
- 外部业务类型；
- 外部单号；
- 入站结果；
- 最近回传；
- 对方确认状态；
- 当前失败原因；
- 尝试次数；
- 最近尝试时间；
- 使用的履约配置版本；
- 当前允许动作；
- 协同操作记录。

业务状态：

- 已接收；
- 待发送；
- 发送中；
- 已送达；
- 对方已确认；
- 对方拒绝；
- 暂时失败，等待重试；
- 重试已用尽；
- 等待人工处理；
- 已人工解决。

普通运营用户不得看到认证秘密、完整原始报文、技术队列对象、内部消息标识或异常堆栈。

重新发送只能发送已经冻结的同一业务快照。业务事实需要更正时，先执行工单业务命令产生新事件。

## 13. 操作记录

操作记录必须能够解释：

- 谁；
- 在什么时间；
- 对工单执行了什么业务动作；
- 动作前后的责任、状态或数据变化；
- 使用了哪个配置版本或候选依据；
- 是否成功；
- 失败原因；
- 相关操作追踪标识。

至少记录：

- 项目岗位人员匹配和人工调整；
- 责任网点分配和改派；
- 责任师傅分配和改派；
- 预约确认与改约；
- 流程任务创建、领取、完成和转交；
- 表单资料提交；
- 审核、整改和复核；
- 外部入站和回传；
- 工单取消和异常处理。

## 14. 页面读模型

建议使用：`WorkOrderWorkspaceView`。

必须包含：

- `header`；
- `configurationBinding`；
- `progress`；
- `currentTask`；
- `responsibilityChain`；
- `projectPersonnelSnapshot`；
- `assignment`；
- `appointment`；
- `fieldWork`；
- `formsAndEvidence`；
- `reviewAndCorrection`；
- `externalCollaboration`；
- `sla`；
- `blockers`；
- `recommendedAction`；
- `activityTimeline`；
- `moduleAvailability`；
- `allowedActions`；
- `version`。

`responsibilityChain` 至少提供：

- 三个项目岗位人员及完整性；
- 当前责任网点；
- 当前责任师傅；
- 当前任务处理对象和实际处理人；
- 每层责任来源；
- 生效时间；
- 当前状态；
- 缺失或冲突；
- 允许处理入口。

读模型必须直接提供业务名称和中文状态。前端不得通过多个目录、流程和资源接口自行拼装责任链。

## 15. 业务命令

工单工作区本身可以承载或进入以下命令：

- 调整项目关联人员；
- 分配责任网点；
- 改派责任网点；
- 分配责任师傅；
- 改派责任师傅；
- 确认或调整预约；
- 执行当前流程任务；
- 提交审核；
- 执行审核结论；
- 提交整改；
- 执行复核；
- 重新发送冻结回传；
- 标记外部确认或人工解决；
- 请求取消或取消工单。

所有命令必须支持：

- 幂等键；
- 预期工单版本；
- 权限校验；
- 允许动作校验；
- 当前责任和任务校验；
- 中文业务错误；
- 明确操作结果；
- 刷新建议；
- 可追踪操作标识。

派单和改派命令必须遵守 AD-004；项目人员调整必须遵守 DEC-001；审核整改必须遵守 AD-011；外部协同必须遵守 AD-012 和 DEC-002。

## 16. 权限

至少区分：

- 查看工单；
- 查看客户敏感信息；
- 查看服务地址；
- 查看项目岗位人员；
- 调整项目关联人员；
- 分配或改派责任网点；
- 分配或改派责任师傅；
- 查看或调整预约；
- 执行当前任务；
- 审核、整改和复核；
- 查看外部协同；
- 执行外部协同人工操作；
- 查看操作记录；
- 进入技术诊断。

只读用户使用专门只读呈现，不以大量置灰控件模拟可编辑页面。

## 18. 当前阶段不包含

- 在工单工作区维护项目区域分工；
- 在工单工作区维护网点和师傅主数据；
- 重新按最新项目配置迁移工单；
- 通过派单自动改变项目岗位人员；
- 使用模糊“负责人”字段替代责任链；
- 在浏览器自行判断候选资格；
- 技术流程实例、消息队列和数据库对象首屏；
- Admin 代替师傅完成设备能力相关现场动作。

## 19. 验收标准

- 首屏能看清当前阶段、责任链、风险和下一步；
- 项目人员、责任网点、责任师傅和当前任务处理人清晰区分；
- 项目岗位人员来自工单快照，不实时重算；
- 派网点和派师傅是两个明确阶段；
- 所有责任变更有来源、生效时间和历史记录；
- 派单和改派进入 AD-004 的统一候选与影响确认；
- 审核、整改和复核使用同一权威任务状态；
- 外部协同使用业务状态，不展示技术队列对象；
- 页面不退化为超长字段详情表；
- 所有动作由服务端返回并具有中文结果；
- 不展示内部 ID、技术字段或英文枚举；
- 用户无需跳转多个技术页面才能判断工单；
- 重要状态不仅依赖颜色；
- 1280、1440 和 200% 缩放下可完成主要任务；
- 操作记录能够解释责任和状态为何发生变化。
