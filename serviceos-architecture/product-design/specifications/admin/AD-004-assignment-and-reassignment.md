# AD-004 派单与改派

状态：Draft for Product Review  
设计等级：A

## 1. 使用场景

派单与改派是工单工作区和网点工作台中的上下文操作，不设计成独立技术页面。

适用场景：

- 平台将工单分配给责任网点；
- 网点将工单分配给师傅；
- 因产能、技能、距离、请假或异常原因进行改派；
- 批量处理同一项目或区域内的待分配工单。

## 2. 核心任务

用户需要快速判断：

- 当前工单需要什么技能和服务能力；
- 哪些网点或师傅符合条件；
- 候选人的当前负载、距离和可用时间；
- 派单是否会影响预约或 SLA；
- 改派会影响哪些当前责任和进行中的任务。

## 3. 交互形态

### 3.1 单个工单

使用右侧抽屉，保持工单上下文可见。

抽屉结构：

1. 工单摘要；
2. 派单条件；
3. 候选筛选；
4. 候选列表；
5. 影响提示；
6. 确认操作。

### 3.2 批量派单

使用独立批量工作区，必须展示：

- 选中工单数量；
- 可统一分配数量；
- 不满足条件数量；
- 不能分配的具体原因；
- 部分成功后的结果明细。

## 4. 候选信息

网点候选至少显示：

- 网点名称；
- 服务范围匹配情况；
- 当前待处理量；
- 近期 SLA 表现；
- 可接单状态；
- 预计响应时间。

师傅候选至少显示：

- 姓名；
- 有效技能；
- 当前任务量；
- 今日可用时段；
- 距离或服务区域；
- 当前状态；
- 近期履约质量。

不得显示内部 ID 代替名称。

## 5. 推荐与筛选

系统可以给出推荐排序，但必须解释主要依据，例如：

- 技能完全匹配；
- 当前负载较低；
- 距离服务地址较近；
- 可在预约时间前到达；
- 近期履约质量稳定。

不得使用不可解释的“智能推荐”标签。

## 6. 改派影响

改派确认前必须说明：

- 原责任方；
- 新责任方；
- 是否取消原预约；
- 是否影响当前任务；
- 是否需要通知客户；
- 是否可能影响 SLA；
- 是否需要填写改派原因。

进行中的现场作业不得通过普通改派流程直接变更责任人。

## 7. 读模型与命令

建议读模型：

- `AssignmentWorkspaceView`；
- `AssignmentCandidateView`。

命令：

- `AssignWorkOrderCommand`；
- `ReassignWorkOrderCommand`；
- `BatchAssignWorkOrdersCommand`。

命令必须包含幂等键、预期版本、责任范围和改派原因。

## 8. 状态

必须设计：

- 存在可用候选；
- 无可用候选；
- 候选数据过期；
- 工单状态变化；
- 原责任方已有进行中操作；
- 派单成功；
- 派单失败；
- 批量部分成功；
- 无派单权限；
- 只读查看。

## 9. 验收标准

- 用户可以理解候选为什么适合；
- 改派前明确展示业务影响；
- 不允许仅凭名称或 ID 盲选；
- 不允许前端自行判断可分配范围；
- 无候选时给出明确原因和下一步处理方式；
- 命令失败不得静默关闭抽屉；
- 所有操作结果进入工单操作记录。
