# AD-006 流程配置与版本发布

状态：Draft for Product Review  
设计等级：A

## 1. 核心任务

用户需要设计一个可执行的履约流程，并在发布前理解流程完整性和业务影响。

用户必须能够：

- 查看当前生效流程；
- 编辑新草稿；
- 增删和调整流程节点；
- 配置节点责任和进入条件；
- 发现无出口、无责任人和循环等问题；
- 对比草稿与生效版本；
- 分析发布影响；
- 发布不可变的新版本。

## 2. 页面结构

### 2.1 顶部版本区

展示：

- 项目名称；
- 当前生效版本；
- 当前草稿版本；
- 草稿保存状态；
- 校验状态；
- 发布准备状态；
- 保存草稿、预览流程、发布版本。

### 2.2 中央流程画布

- 节点必须使用业务名称；
- 连线表达明确的流转条件；
- 当前选中节点具有明显但克制的视觉状态；
- 支持缩放、平移和适合视图；
- 支持键盘选择和删除；
- 不允许把数据库 ID 显示在节点上。

### 2.3 节点属性面板

右侧面板根据节点类型展示：

- 节点名称；
- 节点类型；
- 责任角色；
- 进入条件；
- 完成条件；
- SLA 或时限；
- 需要的表单和资料；
- 允许的下一步；
- 是否允许跳过；
- 业务说明。

不直接展示后端 JSON 配置。

### 2.4 组件面板

允许添加的组件：

- 开始；
- 人工任务；
- 系统任务；
- 审核任务；
- 网点任务；
- 师傅任务；
- 条件分支；
- 并行分支；
- 汇聚节点；
- 等待外部结果；
- 结束。

组件名称必须使用业务语言。

## 3. 校验规则

至少校验：

- 是否存在开始和结束节点；
- 是否存在不可到达节点；
- 是否存在没有出口的非终态节点；
- 是否存在不允许的循环；
- 是否缺少责任角色；
- 是否缺少完成条件；
- 是否引用不存在或未发布的表单；
- SLA 配置是否冲突；
- 外部任务是否缺少失败处理；
- 条件分支是否存在覆盖空缺。

校验结果必须定位到具体节点和属性。

## 4. 版本对比

发布前必须展示：

- 新增节点；
- 删除节点；
- 责任变化；
- 条件变化；
- SLA 变化；
- 表单与资料变化；
- 外部集成变化；
- 可能影响正在履约工单的变化。

不得只展示原始 JSON Diff。

## 5. 影响分析

影响分析必须回答：

- 哪些项目范围将使用新版本；
- 新创建工单何时开始使用；
- 进行中工单是否继续使用旧版本；
- 哪些网点和角色受到影响；
- 是否需要新的权限、表单或外部配置；
- 是否存在阻止发布的问题。

在没有明确迁移策略时，进行中工单继续绑定创建时的流程版本，不自动切换。

## 6. 发布流程

正式流程：

1. 保存草稿；
2. 完整性校验；
3. 业务预览；
4. 版本对比；
5. 影响分析；
6. 填写发布说明；
7. 最终确认；
8. 发布；
9. 展示发布结果。

不允许跳过校验和影响分析直接发布。

## 7. 页面读模型

建议使用：

- `WorkflowDesignerView`；
- `WorkflowValidationView`；
- `WorkflowChangeComparisonView`；
- `WorkflowImpactAnalysisView`；
- `WorkflowReleaseView`。

所有读模型必须使用业务名称和中文说明，不把底层流程引擎结构直接暴露给页面。

## 8. 状态

必须设计：

- 无草稿；
- 草稿编辑中；
- 自动保存中；
- 保存失败；
- 校验失败；
- 校验有警告；
- 版本冲突；
- 只读查看；
- 发布处理中；
- 发布失败；
- 发布成功；
- 依赖资源失效。

## 9. 验收标准

- 非技术产品和实施人员能够理解流程图；
- 节点属性使用业务语言，不显示 JSON；
- 校验问题能够直接定位和修复；
- 发布前能够理解业务影响；
- 生效版本不可修改；
- 不提供旧流程转换或兼容模式；
- 所有版本和发布操作具有完整审计记录。
