# ServiceOS 产品设计基线

状态：Accepted

本目录是 ServiceOS 所有用户界面和交互设计的唯一事实源。

## 已确认方向

ServiceOS 采用 **A+：经典专业任务型** 产品设计方向：

- 以经典企业级视觉为基础；
- 以用户任务、责任、风险和下一步动作组织页面；
- 保持适合长期高频使用的中高信息密度；
- Admin、网点端和原生 iOS 师傅端使用统一产品语言，但不强行使用相同布局；
- 不以数据库表、接口或技术对象组织产品页面；
- 不显示原始 UUID、英文枚举、内部字段、调试数据和技术上下文；
- 不为旧页面、旧路由和旧交互保留兼容设计；
- 先确认领域关系、页面规格、状态、读模型和命令，再决定是否需要高保真图片。

## 已确认项目配置关系

```text
客户
└── 项目
    └── 履约配置
        └── 一个履约工作流
```

- 不设置 Admin 一级菜单“配置中心”；
- 项目列表是项目主数据的唯一权威列表；
- 履约配置位于项目详情中；
- 一个项目当前只支持一个履约工作流；
- 工作流、表单、SLA 等内容随项目履约配置整体发布；
- 公共模板作为下一期候选，不进入当前阶段。

## 首批设计范围

1. Admin 工作台；
2. Admin 工单中心；
3. 工单工作区；
4. 派单与改派流程；
5. 项目详情—履约配置；
6. 项目唯一履约流程设计。

## 原型使用原则

高保真图片只用于验证复杂交互和视觉方向，不是产品事实源。

不得根据图片自动补造：

- 业务实体；
- 菜单层级；
- 字段；
- 状态；
- 权限；
- 流转规则；
- 版本关系。

发生冲突时，以已确认的领域决策和正式页面规格为准。

## 完成定义

页面只有同时满足以下条件，才可以进入开发：

- 明确所属业务对象和入口；
- 明确目标角色和核心任务；
- 明确信息优先级和主要动作；
- 已定义正常、空、错误、无权限、只读和冲突状态；
- 已定义页面读模型和业务命令；
- 复杂交互已经通过规格、流程图或少量最终原型消除理解分歧；
- 已通过产品负责人明确确认；
- 不包含演示、调试、公共模板预实现或旧系统兼容设计。
