# ServiceOS 产品设计实施基线

状态：Accepted
适用范围：Admin、网点 Portal、原生 iOS 师傅端

## 1. 本目录解决什么问题

本目录只保存已经确认、足以约束产品实现的长期产品事实：

- 三个产品分别服务谁、完成什么任务；
- 核心业务旅程和责任边界；
- 统一视觉、交互和中文表达基线；
- 哪些内容已经确认，哪些仍是设计假设；
- 实施过程中何时必须请产品负责人确认。

本目录不再保存逐页面、逐字段、逐异常分支的长篇推演稿，也不承担工程实施状态管理。当前代码、契约、迁移和测试是否已经实现，以
[实施状态总览](../docs/implementation-status.md)及对应里程碑证据为准。

## 3. 事实优先级

实现时按以下顺序判断：

1. 用户在当前任务中明确确认的最新业务决定；
2. 已接受 ADR 和本目录 `decisions/` 中的产品决策；
3. OpenAPI、事件 Schema、Flyway 与自动化测试证明的机器事实；
4. Architecture Book 中 Accepted/Implemented 的设计与里程碑文档；
5. 本目录的页面结构和交互建议；
6. 当前代码和历史探索稿。

页面建议与高优先级工程事实冲突时，不得用页面稿覆盖机器契约；应先判断是产品模型需要修正，还是工程实现需要补齐。

## 4. 已确认的产品方向

- 采用 A+“经典专业任务型”视觉语言，桌面端使用统一浅色侧栏，不使用深色导航；
- Admin、网点 Portal 和原生 iOS 是三个独立产品，不共享整页实现；
- 原生 iOS 是正式现场作业端；Technician H5 只承担受限在线能力；Android 延后评估；
- 正式页面围绕用户任务组织，不围绕数据库表、接口或领域对象组织；
- 调试、演示、Token、伪造上下文和技术诊断不进入正式产品导航；
- 不兼容旧页面、旧路由、旧开发数据和过期文档，不引入静默兜底；
- 项目列表是唯一项目权威入口；履约配置属于项目详情，不设置一级“配置中心”；
- 当前一个项目只有一份履约配置和一个履约工作流；公共模板延期；
- 平台分配责任网点，责任网点分配本网点师傅，网点不能自行跨网点转单；
- 一个网点可以服务多个项目；
- 师傅端必须支持离线保存、可靠上传、恢复和显式冲突处理。

## 5. 文档如何进入开发

每个交付批次按完整用户旅程实施，而不是按字段或页面数量实施：

```text
确认本批次用户目标
    ↓
核对现有契约、代码和工程证据
    ↓
补齐页面读模型与业务命令
    ↓
完成真实可操作页面
    ↓
验证正常、空、错、权限和冲突状态
    ↓
由产品负责人走查真实运行页面
```

开发人员不得仅凭本目录推演新的核心业务语义。涉及责任链、状态机、SLA、审核、数据权限、离线冲突和破坏性契约时，遵守仓库 `AGENTS.md` 的确认规则。

## 6. 产品负责人参与方式

不要求产品负责人逐篇审阅技术文档。只在以下节点参与：

1. 核心业务语义发生变化；
2. 一个完整纵向切片首次形成可运行页面；
3. 高风险动作、异常处置或权限边界需要选择；
4. 页面准备成为后续页面复用母版；
5. 试点或发布前的最终产品验收。

字段命名、组件拆分、查询缓存和测试组织等工程细节，由实现者依据事实源直接完成。

## 7. 停止条件

若某项内容尚无实现任务、真实业务证据或近期验收计划，不继续扩写独立页面规格。设计细节优先在真实纵向切片中验证，验证通过后再回填为稳定产品事实。

历史逐页规格已经从当前事实源删除，Git 历史仍保留其探索过程。本目录不建立 `legacy`、`deprecated` 或双轨文档。
