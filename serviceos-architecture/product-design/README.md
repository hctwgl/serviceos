# ServiceOS 产品设计基线

状态：Accepted

本目录是 ServiceOS 所有用户界面和交互设计的唯一事实源。

## 已确认方向

ServiceOS 采用 **A+：经典专业任务型** 产品设计方向：

- 以经典企业级视觉为基础；
- 以用户任务、责任、风险和下一步动作组织页面；
- 保持适合长期高频使用的中高信息密度；
- Admin、网点端和原生 iOS 师傅端使用统一产品语言，但不强行使用相同布局；
- 不以数据库表、接口或技术对象组织产品页面；
- 不显示原始 UUID、英文枚举、内部字段、调试数据和技术上下文；
- 不为旧页面、旧路由和旧交互保留兼容设计；
- 先确认真实业务关系、页面规格、状态、读模型和命令，再决定是否需要高保真图片。

## 已确认项目关系

```text
客户
└── 项目
    ├── 项目团队与区域分工
    ├── 履约配置
    │   ├── 当前生效版本
    │   ├── 当前活动草稿
    │   └── 一个履约工作流
    └── 项目工单
        ├── 绑定的履约配置版本
        ├── 项目岗位人员快照
        ├── 责任网点
        ├── 责任师傅
        ├── 审核、整改与复核记录
        └── 外部协同记录
```

- 不设置 Admin 一级菜单“配置中心”；
- 项目列表是项目主数据的唯一权威列表；
- Admin 工作台只聚合任务投影，不保存第二套业务状态；
- 配置待办点击后进入对应项目详情的履约配置；
- 项目团队与区域分工位于项目详情中；
- 履约配置位于项目详情中；
- 一个项目当前只支持一个履约工作流；
- 工单根据项目和地址行政区域匹配客服经理、项目经理和项目助理；
- 工单保存匹配时的人员快照；
- 流程、SLA、审核、整改、通知和回传统一消费工单人员快照；
- 项目团队与区域分工不随履约配置版本发布；
- 项目网点准入、服务覆盖和师傅能力规则随履约配置整体发布；
- 网点、师傅、排班、负载和实时产能是运营数据，不进入履约配置版本；
- 审核与整改规则随履约配置发布，实际任务和记录属于工单运行数据；
- 项目集成业务规则随履约配置发布，外部连接属于系统管理，实际收发属于工单运行数据；
- 履约配置只能作为完整项目版本原子发布；
- 新工单绑定新版本，进行中工单继续使用创建时版本；
- 公共模板作为下一期候选，不进入当前阶段。

## 已接受跨模块决策

- `DEC-001-work-order-project-personnel-snapshot.md`：工单项目岗位人员快照是流程、SLA、审核、整改、通知、回传和页面展示获取项目岗位人员的唯一来源；
- `DEC-002-project-integration-boundaries.md`：项目集成业务规则、系统连接和工单外部协同运行记录分层；
- `DEC-003-project-configuration-atomic-publishing.md`：项目履约配置原子发布，工单绑定创建时版本，历史规则恢复形成新版本；
- `DEC-004-admin-workbench-task-projection.md`：Admin 工作台只做跨模块任务投影和业务深链接，不维护第二套业务状态。

发生人员缺失时，不允许静默选择默认用户；已有工单换人必须通过明确操作并保留审计记录。

发布失败时当前生效版本必须保持不变，不允许任何模块部分生效。

工作台与目标业务页面状态不一致时，以目标业务页面最新状态为准。

## 原型使用原则

高保真图片只用于验证复杂交互和视觉方向，不是产品事实源。

不得根据图片自动补造：

- 业务实体；
- 菜单层级；
- 字段；
- 状态；
- 权限；
- 流转规则；
- 版本关系；
- 人员匹配和资源匹配逻辑；
- 审核、整改和复核任务关系；
- 集成连接与运行记录关系；
- 工作台任务归属和优先级。

发生冲突时，以已确认的真实业务、决策记录和正式页面规格为准。

## 完成定义

页面只有同时满足以下条件，才可以进入开发：

- 明确所属业务对象和入口；
- 明确目标角色和核心任务；
- 明确信息优先级和主要动作；
- 已定义正常、空、错误、无权限、只读和冲突状态；
- 已定义页面读模型和业务命令；
- 已与真实业务操作方式一致；
- 跨模块人员、版本、资源、任务、集成和工作台投影边界与已接受决策一致；
- 复杂交互已经通过规格、流程图或少量最终原型消除理解分歧；
- 已通过产品负责人明确确认；
- 不包含演示、调试、公共模板预实现或旧系统兼容设计。
