# DEC-019：ServiceOS AI Agent 开发协作规范 V1.0

状态：Accepted

版本：1.0.0

日期：2026-07-24

适用范围：

- Codex Agent
- Claude Code Agent
- 自动化开发 Agent
- ServiceOS 全部研发任务


---

# 1. 决策背景

ServiceOS 采用 AI Agent 辅助研发模式。

Agent 负责：

- 代码实现
- 测试执行
- 文档维护
- 技术分析


负责人负责：

- 产品方向
- 架构决策
- 任务拆解
- PR 验收


Agent 不是自由开发者。

Agent 必须在 ServiceOS 产品决策体系约束下执行。


---

# 2. Agent 工作原则


## 2.1 先理解，再开发


任何任务开始前：

必须读取：

```
serviceos-architecture/product-design/

```

至少包括：

- 当前相关 DEC 文档
- 产品事实源
- 架构约束


禁止：

直接根据任务标题开发。


---

## 2.2 产品优先


Agent 必须优先理解：

用户是谁？

为什么使用？

核心任务是什么？

下一步动作是什么？


禁止：

围绕数据库表设计页面。


---

## 2.3 最小增量原则


优先：

复用已有能力。


禁止：

- 新建重复领域模型
- 新建重复服务
- 重复实现已有能力


---

# 3. 标准任务流程


每个 Sprint 必须遵循：

```
需求理解

↓

方案分析

↓

实施计划

↓

编码

↓

测试

↓

PR

↓

验收

```


---

# 4. Step 1：需求分析


Agent 首先输出：

```
docs/agent-analysis/

```

内容：

- 当前问题理解
- 影响范围
- 现有能力分析
- 实施方案


禁止：

直接修改大量代码。


---

# 5. Step 2：方案设计


必须说明：

## 产品层

- 用户是谁
- 页面目标
- 业务流程


## 技术层

- 修改模块
- API影响
- 数据影响
- 测试方案


---

# 6. Step 3：开发要求


## 后端


必须遵守：

- Spring Modulith 边界
- API 契约
- Flyway 数据迁移
- Event Schema


禁止：

跨模块直接访问 Repository。


---

## 前端


必须遵守：

- ServiceOS Design System
- 业务组件复用
- 页面任务导向


禁止：

- 默认 Vben 页面风格
- CRUD堆页面
- 技术字段直接展示


---

# 7. PR 提交规范


PR 必须包含：


## 标题


格式：

```
feat/fix/refactor: xxx

```


---

## 描述


必须包含：

### What changed

改了什么。


### Why

为什么。


### Impact

影响范围。


### Validation

如何验证。


---

# 8. 截图要求


涉及 UI：

必须提供：

- 页面截图
- 关键流程截图
- 不同分辨率验证


---

# 9. 验收规则


验收分三个层次。


## 产品验收


检查：

- 是否符合 DEC
- 是否解决用户问题
- 是否提升业务效率


---

## 技术验收


检查：

- 架构边界
- 测试
- 性能
- 可维护性


---

## 体验验收


检查：

- 信息密度
- 操作路径
- 视觉一致性


---

# 10. Agent 禁止事项


禁止：

1. 自行改变产品模型。

2. 创建未经批准的新领域。

3. 为展示效果制造虚假业务逻辑。

4. 用 Mock 数据掩盖真实能力缺失。

5. 删除已有业务约束。

6. 绕过权限。

7. 修改核心状态机而不说明。


---

# 11. Agent 必须主动发现问题


如果发现：

- 当前模型不足
- API不足
- 权限不足
- 产品设计冲突


必须：

提出问题。

输出：

```
问题

影响

建议

等待确认

```


禁止：

自行扩大范围。


---

# 12. Sprint 管理规则


每个 Sprint：

必须只有一个核心目标。


例如：

正确：

```
实现履约方案版本管理

```

错误：

```
完善整个 ServiceOS

```


---

# 13. 分支规则


推荐：

```
codex/sprint-name

```


禁止：

直接修改 master。


---

# 14. Commit 规则


Commit 应体现业务价值。


例如：

正确：

```
feat: add fulfillment version publishing workflow

```


错误：

```
update files

```


---

# 15. ServiceOS 开发优先级


优先级顺序：


1.

业务模型正确


2.

核心流程闭环


3.

用户体验


4.

性能优化


5.

高级能力


---

# 16. Agent 最终目标


不是：

完成更多页面。


而是：

建设一个长期演进的新能源履约操作系统。


所有代码必须服务于：

```
项目

↓

履约方案

↓

版本

↓

工单

↓

任务

↓

现场执行

↓

数据沉淀

```

---
