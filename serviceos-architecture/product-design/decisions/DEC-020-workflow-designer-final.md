# 1. 决策目标

本决策冻结 ServiceOS 履约方案中 Workflow Designer 的产品模型、领域模型、交互方式和一期能力边界。

核心目标：

> 履约配置人员可以从空白画布开始，围绕纵向流程和业务节点，完成阶段、节点、任务、表单、证据、SLA、系统动作、事件等待、条件流转、校验、模拟和版本发布。

Workflow Designer 不是：

- 静态流程展示页
- 功能菜单集合
- 技术 BPMN 编辑器
- JSON 配置工具
- 运行工单监控页面

---

# 2. 核心模型

```text
Fulfillment Plan
  ↓
Fulfillment Version
  ↓
Fulfillment Workflow
  ↓
Fulfillment Phase
  ↓
Node
  ↓
Task / Form / Evidence / SLA / Rule
  ↓
Node Completion Result
  ↓
Transition
  ↓
Next Node
```

核心职责：

| 对象 | 职责 |
|---|---|
| Fulfillment Version | 冻结完整履约规则 |
| Workflow | 定义完整业务路径 |
| Fulfillment Phase | 组织业务阶段、展示与统计 |
| Node | 执行或控制流程 |
| Task | 表达人工责任与动作 |
| Form | 定义数据采集 |
| Evidence | 定义证明材料 |
| SLA | 定义时间约束 |
| Transition | 根据节点结果选择下一节点 |

---

# 3. Fulfillment Phase

Fulfillment Phase 是业务阶段容器，不是流程执行节点。

示例：

- 客户确认阶段
- 现场勘测阶段
- 施工安装阶段
- 验收交付阶段

Phase 用于：

- 组织流程
- 工单进度展示
- 项目驾驶舱展示
- 阶段耗时统计
- 阶段阻塞分析
- 运营指标

Phase 不负责：

- 生成任务
- 执行条件判断
- 选择下一节点
- 执行 SLA
- 触发流程流转

一个 Phase 可以包含多个 Node。

除开始和结束节点外，其他节点必须属于一个 Phase。

---

# 4. 默认流程方向

流程画布默认采用纵向、自上而下布局。

```text
开始
  ↓
联系客户
  ↓
预约上门
  ↓
现场勘测
  ↓
勘测审核
  ↓
安装施工
  ↓
资料审核
  ↓
结束
```

条件分支向左右展开：

```text
             勘测审核
                 ↓
              审核结果
             ↙       ↘
           驳回       通过
            ↓          ↓
        补充资料     安装施工
```

一期只维护纵向主布局，不同时维护横向和纵向两套编辑模式。

必须支持：

- 纵向自动布局
- 适应画布
- 缩放
- 小地图
- 局部分支横向展开

# 5. 节点分类

Node 分为执行节点和控制节点。

## 5.1 执行节点

### 5.1.1 人工作业节点

用于人工完成业务动作。

示例：

- 联系客户
- 预约上门
- 现场勘测
- 安装施工
- 资料提交
- 客户确认

规则：

- 一期对应一个主 Task Template。
- 运行时创建一个主 Task Instance。
- 可绑定 Form、Evidence、SLA、完成条件和业务输出。

### 5.1.2 审核节点

用于人工审核业务结果。

示例：

- 勘测审核
- 安装资料审核
- 验收审核
- 车企人工审核

规则：

- 审核也是任务。
- 必须绑定审核 Task Template。
- 运行时创建审核 Task Instance。
- 至少支持 `PASS` 与 `REJECT`。
- 可直接按审核结果连接后续节点，不必强制增加条件节点。
- 可配置驳回目标节点。

### 5.1.3 系统动作节点

用于系统自动执行命令。

示例：

- 调用车企接口
- 自动派单
- 发送通知
- 生成资料包
- 更新外部状态
- 计算结算结果

规则：

- 不创建人工 Task Instance。
- 必须配置输入、输出、幂等、重试、成功和失败策略。
- 可输出 `SUCCESS`、`FAILED`、`RETRY_EXHAUSTED` 等结果。

## 5.2 控制节点

### 5.2.1 开始节点

规则：

- 一个 Workflow 只能有一个。
- 不属于 Phase。
- 不创建 Task。
- 必须至少有一条出边。

### 5.2.2 条件判断节点

用于根据业务数据选择唯一路径。

判断来源：

- 上一节点业务输出
- 工单上下文
- 表单字段
- 审核结果
- 系统动作结果
- 外部事件数据

规则：

- 不创建 Task。
- 支持多个互斥输出。
- 必须配置默认分支，或证明条件完全覆盖。
- 一期不支持并行分叉。

### 5.2.3 事件等待节点

只用于流程主动暂停并等待外部事件或时间条件。

适用：

- 等待车企异步回执
- 等待第三方接口回调
- 等待客户小程序确认
- 等待物业审批结果
- 等待约定时间或定时器到期

不适用：

- 等待工程师执行安装
- 等待客服领取审核任务
- 等待普通任务完成
- 等待任务排期

这些属于 Node 或 Task 的运行状态，不应创建事件等待节点。

事件等待节点规则：

- 通常不创建 Task。
- 必须配置等待事件、匹配规则、超时策略。
- 可配置是否生成提醒任务。
- 收到事件后输出事件数据并流转。

### 5.2.4 结束节点

规则：

- 支持多个。
- 不属于 Phase。
- 可表示正常完成、取消、异常终止等结果。
- 不创建 Task。

# 6. 一期串行约束

V1 Workflow Definition 严格采用串行语义：

- 任意时刻最多一个活跃节点。
- 每次节点完成只激活一个下一节点。
- 所有分支均为互斥分支。
- 不支持 Fork / Join。
- 不支持多个并行节点。
- 不支持并行跨 Phase。
- 不支持多主任务同时推进。

允许：

- 审核通过 / 驳回互斥分支
- 条件判断互斥分支
- 显式返工回路
- 多个结束节点

返工回路必须显式配置，并在运行时保留每次节点执行记录。

# 7. Node 与 Task 的设计关系

Node 不等于 Task。

```text
Node
  ↓
Task Template
  ↓
Task Instance
```

Node 定义：

- 流程位置
- 节点类型
- 所属 Phase
- 进入条件
- 完成条件
- 业务输出
- 后续流转

Task 定义：

- 谁执行
- 如何执行
- 输入与输出
- 表单
- 证据
- SLA
- 操作权限

一期：

- 人工作业节点一个主任务。
- 审核节点一个审核任务。
- 节点需要多个独立责任动作时，应拆成多个串行节点。

# 8. 节点资产的创建方式

新拖入的节点默认只有：

- 节点类型
- 临时名称
- 所属 Phase
- 草稿状态

默认不存在：

- Task
- Form
- Evidence
- SLA
- Completion Rule

用户必须在节点设计面板中逐步创建。

每类配置提供两种方式：

## 8.1 当前版本内新建

例如：

- 创建“现场勘测任务”
- 创建“勘测检查表”
- 创建“现场照片证据规则”
- 创建“24 小时勘测 SLA”

## 8.2 从公共模板复制

例如：

- 从公共任务模板复制
- 从公共表单模板复制
- 从公共 SLA 模板复制
- 从公共业务节点模板创建

复制后成为当前 Fulfillment Version 内的独立配置快照。

禁止：

- 已发布版本动态引用可变公共模板
- 公共模板修改后自动影响已发布版本
- 让用户误以为只能从已存在资产中选择

公共业务节点资产库用于提高配置效率。

第一阶段分类：

## 作业类

- 联系客户
- 预约上门
- 现场勘测
- 安装施工

- 客户确认

## 审核类

- 勘测审核
- 安装资料审核
- 验收审核

## 系统动作类

- 发送客户通知
- 调用车企接口
- 生成资料包
- 自动派单
- 更新外部状态

## 事件等待类

- 等待客户确认

# 16. 连线与 Transition

每个可连接节点显示输入、输出锚点。

操作：

1. 从源节点输出锚点拖动
2. 连接目标节点输入锚点
3. 创建 Transition
4. 配置结果或条件

普通作业节点可按业务结果配置多条互斥出边。

审核节点可直接配置：

```text
PASS   → 安装预约
REJECT → 补充资料
```

条件节点配置：

- 数据来源
- 字段
- 比较符
- 值
- 分支名称
- 默认分支

系统必须：

- 阻止非法连接
- 检查不可达路径
- 检查无终点路径
- 对循环提供风险提示
- 允许显式返工回路

# 18. 配置顺序与引导

设计器必须通过步骤状态和缺失提示引导：

1. 创建或复制方案版本
2. 配置适用范围
3. 创建 Fulfillment Phase
4. 搭建纵向流程骨架
5. 添加 Node
6. 连接 Node
7. 创建任务并配置责任
8. 创建表单和证据要求
9. 配置 SLA
10. 配置系统动作或事件等待
11. 配置节点结果和条件分支
12. 完整性校验
13. 模拟运行
14. 发布版本

禁止依赖长篇解释文字引导用户。

# 19. 配置完整度

设计器必须显示版本完整度。

示例：

```text
流程结构：完成
节点责任：7/7
任务配置：6/7
表单配置：4/5
证据规则：3/5
SLA：5/7
模拟运行：未完成
```

节点卡片必须显示配置摘要：

- 节点类型
- 责任角色
- 任务是否已创建
- 表单是否已配置
- 证据是否完整
- SLA 是否完整
- 配置完整 / 缺失状态

节点卡片不得展示运行态任务状态或工单数据。

---

# 20. 完整性校验

发布前必须校验：

- 唯一开始节点
- 至少一个结束节点
- 不存在孤立节点
- 不存在不可达节点
- 不存在无终点路径
- 所有分支为互斥分支
- 人工作业节点具备任务
- 审核节点具备审核任务和结果路径
- 系统动作节点具备成功、失败和重试策略
- 事件等待节点具备事件和超时配置
- 条件判断节点具备分支和默认策略
- 关键节点具备 SLA
- 必填表单和证据完整
- 返工回路可解释且不会自动无限循环

校验结果必须：

- 定位到具体 Phase
- 定位到具体 Node
- 定位到具体配置项
- 提供直接修复入口

---

# 21. 模拟运行

模拟运行用于发布前验证，不创建真实工单。

输入：

- 项目
- 品牌
- 服务类型
- 区域
- 设备或车型等匹配条件
- 节点模拟结果

输出：

- 命中方案和版本
- 执行的 Phase
- 执行的 Node
- 创建的 Task
- 使用的 Form / Evidence / SLA
- 分支选择原因
- 系统动作和事件等待
- 预计流程路径

---

# 22. 版本与发布

版本生命周期：

- Draft
- Testing
- Active
- Archived

规则：

- Active 版本只读。
- 修改必须创建新 Draft。
- 发布时整体冻结 Workflow、Phase、Node、Transition、Task、Form、Evidence、SLA 和规则。
- 新版本只影响新工单。
- 历史工单继续使用创建时绑定版本。
- 公共模板后续变化不得影响已发布版本。

# 23. 技术决策

使用 Vue Flow 作为流程画布基础。

必须支持：

- 自定义节点
- 自定义 Edge
- 拖拽
- 纵向自动布局
- 缩放
- 适应画布
- 小地图
- 键盘删除和复制
- 撤销与重做
- Phase 分区
- 节点选中定位
- 校验问题定位

不采用完整 BPMN 标准。

ServiceOS 使用自身业务模型保存 Workflow Definition。

禁止：

- 用静态 HTML / CSS 模拟流程设计器
- 使用 localStorage 冒充持久化
- 使用 Mock 冒充真实保存
- 刷新后丢失流程

---

# 24. 一期能力范围

支持：

- 纵向串行流程
- 互斥条件分支
- 审核通过 / 驳回分支
- 显式返工回路
- 多结束节点
- Phase 分区
- 节点复制
- 公共业务节点模板
- 系统动作节点
- 事件等待节点
- 自动布局
- 完整性校验
- 模拟运行

不支持：

- 多节点并行
- Fork / Join
- 复杂会签
- 多实例任务
- 任意脚本表达式
- 完整 BPMN
- 跨流程事务补偿编排

---

# 25. Agent 开发约束

禁止：

- 用 Phase 代替 Node
- 让 Phase 驱动流程
- 把 Task 当成 Node
- 把普通等待状态建成事件等待节点
- 把系统动作伪装成人工任务
- 将任务、表单、SLA做成与流程平行的功能菜单
- 让用户只能选择预先存在资产而不能创建
- 将公共模板动态绑定到已发布版本
- 将工单数据和运行统计放入设计器
- 一期引入并行流程

必须：

- 默认纵向流程
- Node 作为设计中心
- 右侧节点设计面板统一配置节点
- 新节点资产初始为空，由用户创建或从模板复制
- 审核节点创建审核任务
- 系统动作节点不创建人工任务
- 事件等待节点只用于外部事件或时间
- Transition 根据业务结果流转
- 已发布版本不可变

# 26. 验收用例

必须通过：

1. 新建空白草稿版本。
2. 创建四个 Fulfillment Phase。
3. 添加开始节点。
4. 添加联系客户、现场勘测、勘测审核、安装施工。
5. 给现场勘测节点新建任务、表单、证据和 SLA。
6. 给勘测审核节点创建审核任务。
7. 创建 PASS → 安装、REJECT → 补资料分支。
8. 添加提交车企系统动作节点。
9. 添加等待车企回执事件等待节点。
10. 添加条件判断并配置唯一命中分支。
11. 保存草稿并刷新，流程完整恢复。
12. 运行完整性校验并定位缺失节点。
13. 模拟运行并展示分支原因。
14. 发布版本后只读。
15. 基于 Active 版本创建新 Draft。
16. 全流程任意时刻最多一个活跃节点。
