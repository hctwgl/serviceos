# DEC-011：ServiceOS 调度中心 Dispatch Workspace 产品设计决策 V1.0

状态：Accepted

版本：1.0.0

日期：2026-07-24

适用范围：

- ServiceOS Admin Web
- Network Web
- Dispatch Workspace
- Task Assignment
- Technician Resource Management
- Appointment Coordination
- Field Execution

---

# 1. 决策背景

ServiceOS 调度中心不是简单派单页面。

它是连接：

```
WorkOrder

↓

Task

↓

Service Network

↓

Technician

↓

Field Execution
```

的资源协调工作台。

核心目标：

> 在满足履约规则、技能、区域、SLA和资源约束的情况下，将任务分配给最合适的执行主体。

---

# 2. 调度中心定位

Dispatch Workspace 解决的问题：

- 哪些任务需要处理？
- 哪些任务存在风险？
- 谁可以执行？
- 为什么推荐这个人？
- 当前资源是否冲突？
- 如何快速完成分配？


不是：

- 一个简单派单列表
- 一个选择工程师下拉框
- 一个任务状态页面

---

# 3. 核心模型

关系：

```
Project

↓

Fulfillment Version

↓

Workflow Task

↓

Dispatch

↓

Service Network

↓

Technician

```

---

# 4. Dispatch 输入

调度必须基于业务事实。

输入包括：

## 工单任务

- 任务类型
- 服务阶段
- SLA要求
- 优先级
- 客户要求


## 履约约束

来自：

Fulfillment Version

包括：

- 技能要求
- 资质要求
- 服务区域
- 时间要求
- 证据要求


## 资源能力

包括：

- 网点覆盖范围
- 工程师技能
- 工程师资质
- 当前任务
- 可用时间

---

# 5. 调度工作台页面结构

采用专业任务型 Workspace。

```
------------------------------------------------

顶部：
筛选 / 日期 / 区域 / 项目 / 状态

------------------------------------------------

左侧：
待调度任务池

中间：
资源与任务匹配区域

右侧：
候选资源详情

底部：
时间安排

------------------------------------------------
```

---

# 6. 待调度任务池

展示：

- 工单编号
- 客户
- 服务类型
- 当前阶段
- SLA剩余时间
- 优先级
- 区域
- 所需技能


重点突出：

风险任务。

例如：

```
安装任务

剩余 SLA：2小时

需要：高压资质

区域：山东济南
```

---

# 7. 候选资源模型

候选资源包括：

## 服务网点

展示：

- 网点名称
- 覆盖区域
- 当前容量
- 可承接业务


## 技术人员

展示：

- 姓名
- 技能
- 资质
- 当前任务
- 可用时间

---

# 8. 推荐规则

系统可以推荐，但必须可解释。

禁止：

```
智能评分：96分
```

但不知道为什么。


推荐必须说明：

```
推荐李师傅

原因：

✓ 服务区域匹配
✓ 具备安装资质
✓ 当前空闲
✓ 最近任务距离较近
△ 今日已有3个任务
```

---

# 9. 派单流程

标准流程：

```
任务进入待调度

↓

规则筛选

↓

生成候选资源

↓

人工确认

↓

创建 Assignment

↓

通知执行人员

↓

任务开始执行
```

---

# 10. Assignment 分配模型

Assignment 表示：

任务与执行主体之间的责任关系。

包含：

- Task
- Service Network
- Technician
- 分配时间
- 分配人
- 接单状态

---

# 11. 改派规则

改派不是简单修改字段。

需要记录：

- 原责任人
- 新责任人
- 原因
- 操作人
- 时间

形成业务事件。

---

# 12. Appointment 协同

调度需要结合预约。

关系：

```
Task

↓

Appointment

↓

Technician Schedule
```


预约信息包括：

- 客户时间窗口
- 工程师时间
- 预计到达时间
- 调整原因

---

# 13. 调度与履约方案关系

Dispatch 不定义业务规则。

规则来源：

Fulfillment Version。

例如：

履约版本要求：

```
安装任务

需要：
高压电工证

区域：
山东
```

调度负责寻找满足条件的人。

---

# 14. 调度页面禁止事项

禁止：

1. 做成普通任务列表。

2. 只提供人工选择人员。

3. 隐藏推荐原因。

4. 不展示资源冲突。

5. 用状态字段代替责任关系。

---

# 15. Agent 开发约束

必须：

- 调度围绕任务展开。
- 推荐必须解释。
- 责任必须可追溯。
- 改派必须形成事件。
- 资源能力必须结构化。

禁止：

- if/else硬编码派单规则。
- 把调度做成下拉框页面。
- 把工程师列表当资源管理。

---

# 16. 页面验收标准

打开调度中心后，用户应该立即知道：

1. 当前有哪些任务需要处理？

2. 哪些任务风险最高？

3. 谁最适合执行？

4. 为什么推荐？

5. 分配后是否影响SLA？

如果只能看到：

```
任务列表
选择人员
点击派单
```

则认为设计失败。
