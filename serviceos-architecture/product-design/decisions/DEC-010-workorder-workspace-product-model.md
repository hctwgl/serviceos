# DEC-010：ServiceOS 工单 Workspace 运行态产品模型决策 V1.0

状态：Accepted

版本：1.0.0

日期：2026-07-24

适用范围：

-   ServiceOS Admin Web
-   Network Web
-   Technician Web / iOS
-   WorkOrder Runtime
-   Task Execution
-   Field Service
-   Evidence
-   SLA
-   Exception

------------------------------------------------------------------------

# 1. 决策背景

ServiceOS 不是简单工单管理系统。

工单不是业务配置中心，而是：

> 履约方案版本的一次实际执行实例。

核心关系：

    Project

    ↓

    Fulfillment Plan

    ↓

    Fulfillment Version

    ↓

    WorkOrder

    ↓

    Workflow Instance

    ↓

    Task Instance

    ↓

    Field Execution

------------------------------------------------------------------------

# 2. WorkOrder 定义

WorkOrder（工单）表示：

一次具体新能源现场服务履约。

例如：

    客户：
    张先生


    车辆：
    比亚迪王朝


    服务：
    家充安装


    绑定方案：

    比亚迪王朝勘测安装 V3

------------------------------------------------------------------------

# 3. 工单与履约版本关系

工单创建时必须冻结履约版本。

例如：

2026-07-24：

创建：

WO001

绑定：

比亚迪王朝勘测安装 V3

之后：

方案升级到 V4。

结果：

    WO001

    继续使用 V3


    新工单

    使用 V4

------------------------------------------------------------------------

# 4. 工单运行模型

运行结构：

    WorkOrder

        |

    Workflow Instance

        |

    Stage Instance

        |

    Task Instance

        |

    Field Execution

        |

    Result

------------------------------------------------------------------------

# 5. WorkOrder 生命周期

状态由履约流程驱动。

示例：

    创建

    ↓

    待处理

    ↓

    执行中

    ↓

    等待外部条件

    ↓

    审核中

    ↓

    完成

    ↓

    关闭

禁止：

通过一个 status 字段承载所有业务含义。

------------------------------------------------------------------------

# 6. WorkOrder 页面定位

工单页面不是字段详情页。

它是：

> 履约执行 Workspace。

目标：

用户进入后快速知道：

-   当前在哪里
-   谁负责
-   下一步是什么
-   是否有风险

------------------------------------------------------------------------

# 7. 工单 Workspace 页面结构

推荐布局：

    ------------------------------------------------

    工单摘要


    客户
    服务类型
    当前阶段
    负责人
    SLA状态


    ------------------------------------------------


    左侧：

    履约时间线


    中间：

    当前任务


    右侧：

    上下文信息


    ------------------------------------------------

------------------------------------------------------------------------

# 8. 顶部工单摘要

展示：

-   工单编号
-   客户
-   项目
-   履约方案版本
-   当前阶段
-   SLA状态
-   当前责任人

操作：

根据权限展示：

-   转派
-   提交
-   审核
-   异常处理

------------------------------------------------------------------------

# 9. 履约时间线

时间线展示业务事件。

不是系统日志。

示例：

    09:00

    客户联系完成


    10:30

    预约成功


    11:20

    勘测任务创建


    14:00

    工程师到场

事件来源：

-   Workflow
-   Task
-   Field Visit
-   Evidence
-   Review

------------------------------------------------------------------------

# 10. 当前任务区域

这是工单最核心区域。

展示：

当前 Task。

包括：

-   任务名称
-   责任人
-   截止时间
-   SLA
-   输入
-   输出要求
-   完成条件

示例：

    现场勘测任务


    责任：

    工程师


    SLA：

    24小时


    需要：

    填写勘测表

    上传现场照片

    提交结果

------------------------------------------------------------------------

# 11. Task Instance 任务实例

Task Instance 是执行对象。

包含：

-   所属工单
-   所属流程节点
-   责任角色
-   执行人
-   状态
-   开始时间
-   完成时间

状态：

    待执行

    执行中

    提交审核

    已完成

    已驳回

    已取消

------------------------------------------------------------------------

# 12. Field Execution 现场执行

现场任务包含：

-   到场签到
-   定位
-   现场填写
-   图片采集
-   视频采集
-   客户确认

------------------------------------------------------------------------

# 13. Form Instance 表单实例

工单执行产生表单数据。

关系：

    Form Template

           ↓

    Form Instance

例如：

模板：

安装施工表 V3

实例：

张先生安装施工记录。

------------------------------------------------------------------------

# 14. Evidence Instance 证据实例

证据属于执行结果。

关系：

    Evidence Rule

           ↓

    Evidence Instance

例如：

规则：

安装必须：

-   铭牌照片
-   施工照片
-   测试视频

实例：

实际上传资料。

------------------------------------------------------------------------

# 15. SLA Runtime

SLA 在工单运行态体现。

包括：

-   当前倒计时
-   是否超时
-   风险等级
-   升级状态

配置：

来自：

Fulfillment Version

运行：

属于：

WorkOrder

------------------------------------------------------------------------

# 16. Exception 异常模型

异常不是备注。

异常是独立业务对象。

包含：

-   异常类型
-   发现时间
-   责任人
-   影响范围
-   处理状态
-   恢复时间
-   根因

生命周期：

    发现

    ↓

    分派

    ↓

    处理中

    ↓

    恢复

    ↓

    关闭

------------------------------------------------------------------------

# 17. 工单与项目关系

Project：

负责运营边界。

WorkOrder：

负责执行。

关系：

    Project

       |

    多个 WorkOrder

------------------------------------------------------------------------

# 18. 工单与履约方案关系

WorkOrder 不直接引用方案。

必须引用版本。

正确：

    WorkOrder

    ↓

    Fulfillment Version

原因：

版本必须可追溯。

------------------------------------------------------------------------

# 19. 工单页面禁止事项

禁止：

1.  做成字段详情页。

2.  展示大量数据库字段。

3.  只展示状态。

4.  将所有操作堆在顶部。

5.  将系统日志当业务时间线。

------------------------------------------------------------------------

# 20. 工单页面必须体现

必须体现：

-   当前阶段
-   当前任务
-   当前责任
-   下一动作
-   风险
-   履约证据

------------------------------------------------------------------------

# 21. Admin / Network / Technician 边界

## Admin

关注：

项目运营和整体履约。

## Network

关注：

服务商协作、派单、任务分配。

## Technician

关注：

自己的现场任务执行。

------------------------------------------------------------------------

# 22. Agent 开发约束

禁止：

-   WorkOrder 页面复制数据库字段。
-   一个 status 字段驱动所有流程。
-   把时间线做成操作日志。
-   把任务做成附件列表。

必须：

-   工单绑定履约版本。
-   流程驱动任务。
-   任务驱动执行。
-   执行产生证据。
-   SLA贯穿生命周期。

------------------------------------------------------------------------

# 23. 页面验收标准

打开一个工单：

用户应该立即知道：

1.  这是什么服务？

2.  当前做到哪一步？

3.  谁负责？

4.  下一步做什么？

5.  是否存在风险？

如果需要查看多个 Tab 才能回答：

失败。
