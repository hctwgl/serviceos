# DEC-007 项目多履约方案、独立版本与工单绑定

状态：Accepted

适用范围：ServiceOS 项目履约配置、履约方案匹配、版本发布、工单受理与版本绑定、Admin 与工单页面、外部接入建单

关联：取代 DEC-003 的项目级发布边界（改为方案级）、承接 ADR-091（ProjectFulfillmentProfile）、由 [AD-014 履约方案匹配与版本绑定](../../architecture/AD-014-fulfillment-plan-matching-and-version-binding.md) 提供工程设计

## 1. 决策背景

早期基线假设「一个项目只有一条履约流程、一个生效版本和一个草稿版本」。真实现场服务表明：同一个项目（同一客户、同一合同关系）下存在多种差异明显的服务场景，例如设备安装调试、普通故障维修、紧急抢修、定期巡检、设备拆除迁移。它们的流程、SLA、派单、表单、证据和验收规则完全不同，无法压缩进单一流程。

ADR-091 已经引入 `ProjectFulfillmentProfile` / `ProjectFulfillmentRevision`，允许项目按「工单类型（serviceProductCode）」维护多套履约配置并各自独立发布 Revision，正式建单由 `ProjectFulfillmentResolver` 失败关闭解析。本决策在此基础上把「履约方案」正式确立为可按结构化条件匹配的服务场景，并把版本约束、发布边界和工单绑定统一到方案维度。

## 2. 已接受决策

> ServiceOS 中，一个项目可以配置多个履约方案。每个履约方案代表一个清晰且可匹配的服务场景，并独立维护适用范围、流程、SLA、派单策略、作业要求和验收规则。
>
> 每个履约方案最多存在一个生效版本和一个草稿版本，并可保留多个不可变历史版本。发布以单个履约方案版本为原子边界。
>
> 工单在正式受理时，根据项目、品牌、业务类型、设备类型、服务区域、故障等级等结构化条件匹配唯一履约方案，并绑定该方案当时的具体生效版本。
>
> 新版本发布只影响之后受理的新工单。已经绑定版本的工单继续使用原版本，不自动迁移。
>
> 多方案命中时，先按匹配优先级、再按规则具体度确定结果。无法得到唯一结果时，不得随机选择，工单进入待确认状态。
>
> 现场作业开始后，原则上不得直接修改工单绑定的履约方案或版本。

## 3. 核心领域关系

```text
Project 1 ── N FulfillmentPlan
FulfillmentPlan 1 ── N FulfillmentPlanVersion
WorkOrder N ── 1 Project
WorkOrder N ── 1 FulfillmentPlan
WorkOrder N ── 1 FulfillmentPlanVersion
```

流程不再直接挂在项目上，而是属于某个履约方案版本。每个版本冻结一份完整履约配置：适用范围、流程定义、SLA 规则、派单规则、表单与证据要求、验收规则、通知与升级规则、结算规则引用。

## 4. 履约方案状态与版本状态分离

方案状态（描述方案是否参与新工单匹配）：

```text
ENABLED    已启用：参与新工单匹配
DISABLED   已停用：不再参与新工单匹配，不影响历史工单
ARCHIVED   已归档：仅保留查询与审计
```

版本状态（描述某个版本的生命周期）：

```text
DRAFT       草稿：可编辑，不可被工单引用
ACTIVE      当前生效：参与新工单匹配与绑定
HISTORICAL  历史：只读，可查看、比较、复制为新草稿；被工单引用时不可删除
```

约束（数据库层强制，不只依赖应用层）：

- 每个方案最多一个 `DRAFT`；
- 每个方案最多一个 `ACTIVE`；
- 可存在多个 `HISTORICAL`；
- 已发布版本不可原地编辑；修改生效配置从当前 `ACTIVE` 复制生成新 `DRAFT`；
- 草稿发布后成为新的 `ACTIVE`，原 `ACTIVE` 原子转为 `HISTORICAL`；
- 被工单引用的 `HISTORICAL` 不得删除。

停用或归档方案不得影响已绑定该方案版本的运行中工单。

## 5. 原子发布边界为单个履约方案版本

DEC-003 的原子发布核心原则继续保留，但发布单位从「项目履约配置」调整为「单个履约方案版本」：

- 不同履约方案可以独立发布，一个方案发布失败不影响其他方案；
- 每个方案最多一个 `ACTIVE`、一个 `DRAFT`；
- 方案版本发布冻结完整履约配置快照（第 3 节列出的全部内容），不允许单模块独立生效或部分成功；
- 运行中工单继续使用绑定版本。

一个方案版本内部仍然遵循 DEC-003 的原子性、发布准备结果绑定、发布失败语义、已生成运行事实保护和历史规则恢复约束，仅把「项目唯一生效版本」全部改为「方案唯一生效版本」。

## 6. 工单匹配时点为正式受理

工单草稿阶段不绑定履约方案，因为项目、品牌、设备类型、服务地址、业务类型、故障等级和优先级仍可能变化。绑定发生在正式受理：

```text
创建工单草稿 → 补充业务字段 → 提交受理 → 校验项目与服务范围
→ 匹配履约方案 → 绑定方案生效版本 → 创建 SLA 实例 → 创建流程实例 → 进入派单
```

## 7. 匹配、冲突与无匹配处理

- 候选范围：仅当前项目下、方案状态 `ENABLED` 且存在 `ACTIVE` 版本的方案；草稿与历史版本不参与新工单自动匹配。
- 匹配顺序：结构化硬匹配排除不满足的方案 → 按 `matchPriority` 降序 → 同优先级按规则具体度 → 得到唯一方案。
- 禁止用「数据库第一条、创建最早、方案 ID 最小、前端列表顺序」等方式决定方案。
- 多命中同优先级同具体度、或零命中时：不得随机选择、不得默认项目第一条方案；工单进入「待确认履约方案」或「履约配置异常」状态，由具备权限的客服经理、项目经理或平台管理员处理。
- 匹配模式：`AUTO` / `MANUAL` / `EXCEPTION` / `REMATCH`。人工指定必须校验方案归属、启用状态、存在生效版本、用户权限与工单生命周期，规则不匹配时填写例外原因，并保留原始自动匹配候选与解释。

匹配算法、具体度定义与发布期冲突检查详见 [AD-014](../../architecture/AD-014-fulfillment-plan-matching-and-version-binding.md) 第 3～5 节。

## 8. 工单绑定与运行读取

工单受理完成后至少保存 `projectId`、`fulfillmentPlanId`、`fulfillmentPlanVersionId`，并可增强保存 `fulfillmentPlanVersionNo`、`fulfillmentPlanVersionDigest`、`matchedAt`、`matchMode`、`matchedBy`、`matchExplanation`。

绑定后：流程、SLA、表单、证据和验收规则统一通过 `fulfillmentPlanVersionId` 读取。新版本发布不影响现有工单，历史工单必须能查询原方案版本内容。

```text
错误：workOrder.fulfillmentPlan.activeVersion（运行时读“当前生效版本”）
正确：workOrder.fulfillmentPlanVersion（读绑定版本）
```

不建议在工单上分别保存流程版本、SLA 版本、表单版本等多个可变配置 ID，避免相互错配。

## 9. 受理后更换方案

- 草稿阶段：业务字段可改，无需迁移。
- 已受理未派网点：通过 `RematchFulfillmentPlan` 重新匹配，替换版本绑定、重生成 SLA、重建流程实例并保留审计。
- 已派网点未开工：由具备权限角色执行 `AdjustFulfillmentPlan`，重新校验责任网点/责任师傅是否符合新方案、SLA 如何重算、已建任务如何取消、已填数据如何处理，并与责任链变化保持事务一致。
- 现场作业已开始：原则上禁止直接更换方案或版本，推荐终止原工单并重建，或进入明确的异常履约变更流程；不得直接修改数据库中的 `fulfillmentPlanVersionId`。

## 10. 与责任链和派单边界的关系

本决策不改变已确定的责任链：平台分配责任网点，网点分配责任师傅，网点不能把工单转给其他网点。履约方案匹配发生在责任网点分配之前，方案只规定候选网点范围、技能资质要求、是否自动派单、候选排序依据和 SLA 要求：

```text
工单受理 → 匹配履约方案版本 → 计算方案要求 → 计算候选服务网点
→ 平台分配责任网点 → 网点分配责任师傅 → 执行方案版本中的流程
```

DEC-001（岗位人员快照）、DEC-005 与 DEC-006（责任链与派单边界）中「工单绑定的履约配置版本」的语义保持不变，只是该版本现在明确为「工单绑定的履约方案版本」。

## 11. 现状与目标差距（不得外推为已实现）

当前工程基线（ADR-091 / M378～M453）与本目标的对应关系：

| 目标概念 | 当前实现 | 差距 |
|---|---|---|
| 履约方案 FulfillmentPlan | `ProjectFulfillmentProfile`（按 serviceProductCode 键） | 目标允许同一工单类型多方案并按结构化条件匹配 |
| 履约方案版本 FulfillmentPlanVersion | `ProjectFulfillmentRevision`（DRAFT/PUBLISHED） | 目标显式区分 ACTIVE/HISTORICAL |
| 方案状态 ENABLED/DISABLED/ARCHIVED | Profile status ACTIVE/SUSPENDED/RETIRED | 术语对齐待实现 |
| 结构化多维匹配 + matchPriority + 具体度 | 按 brand/product/province 维度解析唯一 Bundle/Revision | 结构化条件、优先级、具体度尚未实现 |
| 待确认履约方案 / 匹配模式 | 零/多命中失败关闭，无待确认队列与 AUTO/MANUAL/EXCEPTION/REMATCH | 待实现 |
| RematchFulfillmentPlan / AdjustFulfillmentPlan | 无（仅创建时冻结、MigrateConfiguration） | 待实现 |

结构化匹配、优先级、具体度、待确认状态、匹配模式和受理后重匹配/调整命令是本决策接受的目标方向，尚无完整工程证据，不得标记为 IMPLEMENTED。相关代码迁移在文档基线统一并通过一致性检查后按独立里程碑推进。

## 12. 验收标准（文档）

- 不再存在「一个项目只能有一条履约流程 / 一份履约配置 / 一个生效版本」的有效描述；
- 项目、履约方案、方案版本、流程定义与工单之间的关系一致；
- 所有版本约束明确为「每个履约方案」维度；
- 发布对象明确为完整履约方案版本快照；
- 工单绑定时点明确为正式受理，且保存具体方案版本而非动态读取当前版本；
- 多方案命中、无方案命中和人工指定都有明确处理方式；
- 方案调整与责任网点、责任师傅、SLA 和流程实例的关系明确；
- Admin 页面、工单页面与 API 设计与新模型一致；
- 比亚迪示例已更新为多履约方案模式；
- 原有责任链和平台/网点派单边界未被破坏。
