# 1. 决策背景

ServiceOS 是新能源现场履约操作系统。

平台不仅需要记录当前状态，还需要回答：

- 发生了什么？
- 谁做的？
- 为什么变化？
- 什么时候变化？
- 依据什么规则？
- 是否影响 SLA？

因此必须建立统一的数据事件和审计模型。

---

# 2. 核心原则

## 2.1 状态不是事实，事件才是事实

禁止只依赖状态字段描述业务历史。

错误：

```
work_order.status = COMPLETED
```

正确：

```
任务完成事件
审核通过事件
验收完成事件
回传成功事件
```

---

# 3. Event Model 定义

Business Event 表示业务事实发生。

结构：

```
Event
 |
 +-- 谁
 |
 +-- 什么对象
 |
 +-- 什么动作
 |
 +-- 什么时候
 |
 +-- 前后状态
 |
 +-- 关联上下文
```

---

# 4. 事件分类

## 4.1 履约事件

例如：

- 工单创建
- 方案匹配
- 版本冻结
- 流程启动
- 阶段完成

---

## 4.2 任务事件

例如：

- 任务创建
- 任务分配
- 接单
- 开始执行
- 提交
- 完成
- 驳回

---

## 4.3 现场事件

例如：

- 到场签到
- 定位成功
- 表单提交
- 图片上传
- 客户确认

---

## 4.4 质量事件

例如：

- 资料审核通过
- 资料审核驳回
- 整改创建
- 整改完成

# 7. Timeline 模型

业务时间线不是日志。

Timeline 展示业务事件。

例如：

```
09:00
客户联系完成

10:30
预约成功

11:20
勘测任务创建

14:00
工程师到场

16:00
资料提交
```

---

# 8. Audit 审计模型

Audit 用于追踪系统操作责任。

记录：

- 操作人
- 操作时间
- 操作对象
- 操作内容
- 修改前
- 修改后
- 请求来源
- 关联事件

---

# 9. 业务事件与审计区别

Business Event：

回答：

“业务发生了什么？”


Audit：

回答：

“谁修改了什么？”


二者不能混合。

---

# 10. SLA 数据来源

SLA 计算必须基于事件。

例如：

```
任务创建事件

开始计时

任务完成事件

停止计时
```

---

# 11. AI 数据基础

未来 AI 能力依赖事件模型。

包括：

- 异常分析
- 根因分析
- 履约预测
- 质量分析
- 自动摘要

---

# 12. Agent 开发约束

禁止：

1. 用日志代替业务事件。
2. 用状态变化代替历史事实。
3. 在页面自行拼接时间线。
4. 不记录关键业务动作。

必须：

1. 核心业务动作产生 Event。
2. 时间线基于 Event。
3. 审计独立记录。
4. 事件支持后续分析和 AI。

---

# 13. 页面验收标准

任何业务页面都应该能够回答：

- 当前状态是什么？
- 为什么是这个状态？
- 谁导致了变化？
- 下一步是什么？
