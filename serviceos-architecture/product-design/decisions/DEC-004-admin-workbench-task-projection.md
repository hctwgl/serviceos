# DEC-004 Admin 工作台只做任务投影与业务深链接

状态：Accepted  
适用范围：ServiceOS Admin 工作台、跨项目待办、配置待办、工单风险、审核整改和外部协同异常

## 1. 决策背景

Admin 工作台需要聚合多个业务模块的待办和风险，但这些数据分别属于工单、派单、审核整改、项目履约配置和外部协同。

如果工作台自行保存另一套任务状态或允许直接修改业务对象，会产生：

- 工作台与业务页面状态不一致；
- 配置待办演变成第二套“配置中心”；
- 同一个项目出现两套配置状态；
- 首页操作绕过业务模块校验和权限；
- 已处理任务仍停留在工作台，或工作台提前显示完成；
- 前端根据多个接口自行推断优先级、权限和路由；
- 聚合数据失败时被错误显示为零。

## 2. 已接受决策

Admin 工作台只保存和展示跨模块任务投影。

```text
业务事实源
├── 工单与流程任务
├── 派单和资源分配
├── 审核、整改与复核
├── 项目履约配置生命周期
└── 外部协同运行记录
        ↓
服务端工作台聚合读模型
        ↓
摘要、优先级、解释和业务深链接
```

工作台不是任何业务对象的权威写模型。

## 3. 配置待办

配置待办的唯一事实源为项目履约配置：

- 当前生效版本；
- 当前活动草稿；
- 保存状态；
- 校验状态；
- 版本对比和影响分析状态；
- 警告确认状态；
- 发布状态；
- 外部依赖可用性。

工作台不得建立：

- 独立配置记录；
- 独立项目配置状态；
- 独立发布状态；
- 独立任务完成标记；
- 第二套项目配置详情页。

配置待办点击后必须进入对应项目详情的履约配置页面，并尽可能定位到具体问题或发布步骤。

## 4. 任务聚合

工作台可以对业务事实进行只读聚合：

- 计算用户范围内的任务数量；
- 合并同一项目的多个配置问题；
- 选择最高优先级问题；
- 生成推荐下一步；
- 提供排序原因；
- 返回正式业务页面入口。

同一项目存在多个配置错误时，工作台原则上展示一条主要配置待办，并显示其他错误和警告数量。完整问题仍在项目履约配置中查看。

聚合结果不改变源业务对象，也不能反向覆盖源状态。

## 5. 允许和禁止的操作

工作台允许：

- 查询和刷新；
- 切换有权限的数据范围；
- 查看摘要和风险解释；
- 进入已筛选的工单或任务队列；
- 进入对应项目的配置模块；
- 进入对应工单的审核、整改或外部协同记录。

工作台禁止：

- 直接发布项目配置；
- 直接修改流程、表单、SLA 或集成映射；
- 批量发布多个项目；
- 直接完成审核或整改；
- 直接改派责任网点或师傅；
- 在工作台保存“已处理”而源业务对象未变化；
- 通过前端拼接内部路由绕过服务端允许动作。

## 6. 数据一致性

工作台是聚合读模型，允许存在短暂刷新延迟，但必须满足：

- 展示数据生成时间；
- 每个聚合模块可以独立报告可用性；
- 来源不可用时显示“暂时无法获取”，不得显示为零；
- 用户进入目标业务页面后，以目标页面最新状态为准；
- 已处理任务在工作台刷新后消失；
- 工作台不得通过本地乐观更新伪造业务完成；
- 数据过旧时明确标记，不得声称实时。

## 7. 权限与深链接

服务端工作台读模型必须直接返回：

- 用户有权看到的业务对象；
- 当前任务的中文业务说明；
- 优先级及其解释；
- 当前允许动作；
- 正式业务目标。

前端不得根据岗位、项目成员关系、任务类型或页面按钮自行推断权限。

用户无权访问的业务对象不应进入其工作台，而不是先返回后脱敏成不可操作任务。

## 8. 当前阶段不包含

- 工作台内联业务编辑；
- 第二套项目或工单列表；
- 独立配置中心；
- 多项目批量发布；
- 工作台通用任务引擎；
- 用户自定义脚本排序；
- 技术消息队列、日志和接口诊断首页。

## 9. 验收标准

- 工作台所有任务都能追溯到明确业务事实源；
- 工作台不拥有独立业务状态流转；
- 配置待办不形成第二套配置中心；
- 所有处理动作进入正式业务页面执行；
- 同一项目配置问题合理聚合而不重复轰炸；
- 聚合排序具有中文业务解释；
- 部分来源失败不会显示为零；
- 目标页面与工作台状态不一致时以目标页面为准；
- 前端不自行推断权限、优先级或内部路由。
