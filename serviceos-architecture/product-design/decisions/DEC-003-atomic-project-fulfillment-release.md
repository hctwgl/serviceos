# DEC-003 履约方案版本原子发布与工单版本绑定

状态：Accepted  
适用范围：ServiceOS 履约方案版本发布、工单受理与版本绑定、历史版本恢复和外部回传

关联：多方案模型与匹配边界由 [DEC-007 项目多履约方案、独立版本与工单绑定](DEC-007-multi-plan-fulfillment-and-work-order-version-binding.md) 确立；原子发布单位从“项目履约配置”调整为“单个履约方案版本”

## 1. 决策背景

每个履约方案的配置由流程、表单、资料、SLA、预约、网点规则、审核整改和集成规则共同组成。

如果这些内容分别发布或部分生效，会出现：

- 流程已经引用新表单，但表单仍是旧版本；
- SLA 已经变化，但流程节点仍使用旧规则；
- 审核结论已启用，但流程没有对应出口；
- 网点覆盖和服务范围不一致；
- 回传字段映射与工单业务状态不一致；
- 发布失败后无法判断哪些规则已经生效；
- 进行中工单被新规则静默改变。

## 2. 已接受决策

每个履约方案的配置只能作为完整方案版本原子发布；不同履约方案独立发布，一个方案发布失败不影响其他方案。

```text
该方案全部业务规则形成一个新的不可变生效版本
或者
该方案当前生效版本完全保持不变
```

不允许任何单模块独立生效，也不允许部分成功。

## 3. 版本范围

原子发布版本包含：

- 服务范围；
- 该方案的履约工作流；
- 表单与资料；
- SLA、预约和服务日历；
- 项目网点准入、覆盖和师傅能力规则；
- 审核与整改规则；
- 入站、字段状态映射、回传和业务通知规则。

版本不包含项目成员、区域人员分工、工单人员快照、实时网点和师傅运营数据、实际审核整改任务、实际外部协同记录或外部连接认证信息。

## 4. 工单版本绑定

工单正式受理时必须匹配唯一履约方案，并绑定该方案当时的生效版本（`fulfillmentPlanId` + `fulfillmentPlanVersionId`）。

该绑定决定该工单使用的：

- 流程结构；
- 表单和资料定义；
- SLA 和预约规则；
- 网点资格与匹配规则；
- 审核与整改规则；
- 集成与回传业务规则。

发布新版本后：

- 新工单绑定新版本；
- 进行中工单继续使用原绑定版本；
- 不自动迁移存量工单；
- 不静默修改已有工单人员、网点、师傅、预约、审核整改任务和外部回传快照。

## 5. 发布准备结果绑定

完整性校验、业务预览、版本对比和影响分析必须绑定当前草稿并发版本。

草稿发生任何保存后：

- 原校验结果失效；
- 原版本对比失效；
- 原影响分析失效；
- 发布条件重新变为未满足。

不得使用旧结果发布新草稿内容。

## 6. 发布失败语义

发布失败时：

- 当前生效版本保持不变；
- 不存在部分生效版本；
- 草稿默认保留；
- 同一发布请求重复执行不得创建多个版本；
- 页面必须明确说明失败阶段和下一步。

只有完整不可变版本已经成功建立并成为该方案唯一生效版本时，才能向用户显示“发布成功”。

## 7. 已生成运行事实

发布新配置不得改写已经发生的运行事实，包括：

- 工单岗位人员快照；
- 责任网点和责任师傅；
- 已确认预约；
- 审核、整改和复核记录；
- 已生成外部回传业务快照；
- 正在重试的外部回传内容。

尤其是外部回传：已经生成的回传继续使用生成时冻结的业务数据和映射，重试时不得按新版本重新拼装。

## 8. 历史规则恢复

不提供直接修改生效指针的一键回滚。

恢复历史业务规则必须：

1. 基于历史版本创建新的活动草稿；
2. 重新执行校验、预览、版本对比和影响分析；
3. 发布为新的更高版本编号。

例如：

```text
当前生效 V5
基于历史 V3 创建恢复草稿
发布后形成 V6
```

历史版本和历史工单绑定不被改写。

## 9. 当前阶段不包含

- 单模块发布；
- 定时发布；
- 灰度发布；
- 分区域发布；
- 同一方案并行多个草稿（不同方案可各自拥有一个草稿）；
- 自动迁移存量工单；
- 直接把方案生效指针切回旧版本；
- 发布审批 BPM；
- 修改历史版本内容。

## 10. 验收标准

- 一个履约方案只有一个当前生效版本；
- 一个履约方案最多只有一个活动草稿；
- 不同履约方案独立发布，互不影响；
- 发布全部成功或当前生效版本完全不变；
- 单个模块无法独立发布；
- 发布结果与准确草稿版本绑定；
- 重复发布请求不会产生多个版本；
- 新工单使用新版本，进行中工单继续使用原版本；
- 新发布不改写已有工单运行事实；
- 正在重试的回传内容不因新配置改变；
- 历史规则恢复形成新版本，不改写版本历史。
