# PR #214 产品设计实施就绪审计

状态：Accepted
审计日期：2026-07-22

## 1. 审计结论

这批产品设计文档对统一领域关系和产品方向有价值，但原结构不适合直接作为开发蓝图。

压缩前，本目录包含 45 份 Markdown、约 2.2 万行内容；PR #214 已累计一百余次提交。大量逐页规格由 Agent 根据既有决策继续推演，产品负责人并未逐项确认。若把“规格编写完成”直接解释为“产品已经定稿”，会产生虚假确定性。

因此本次将事实源压缩为：

- 一份设计基线；
- 一份产品地图；
- 一份核心旅程；
- 一份实施就绪审计；
- 六份已确认的跨模块产品决策。

逐页长规格不再作为当前事实源。Git 历史保留探索过程，不在仓库中建立旧版或兼容文档。

## 2. 四类内容及其使用方式

### 2.1 用户已经确认的产品事实

可以直接约束实现：

- A+ 经典专业任务型视觉方向和统一浅色侧栏；
- Admin、Network、原生 iOS 的独立产品边界；
- 原生 iOS 正式、H5 受限、Android 延后；
- 不兼容旧开发成果、不写静默兜底；
- 项目列表唯一，履约方案归项目详情，不设一级配置中心；
- 一个项目可配置多个履约方案，每个方案独立维护版本；
- 项目与行政区域解析客服经理、项目经理、项目助理并写入工单快照；
- 平台派网点、网点派本网点师傅、网点不可跨网点转单；
- 一个网点可服务多个项目；
- 公共模板下一期再做。

### 2.2 工程事实

必须实时从机器事实和实施状态获取，不能由本目录声明：

- 已存在的模块、API、事件、数据库结构和版本；
- 已实现的 Admin、Network、Technician Web 与 iOS 能力；
- 当前授权、Scope、Navigation 与 `allowed-actions`；
- 已有工作包、离线命令、上传、冲突和整改契约；
- 当前测试、契约、迁移和部署证据。

实施前从 [实施状态总览](../docs/implementation-status.md)和直接工程证据出发，禁止照旧页面规格重新发明一套平行契约。

### 2.3 设计假设

以下内容只有在真实页面或核心业务评审后才能固定：

- 每页的确切字段、卡片数量、列顺序和默认筛选；
- 页面密度、左右栏比例和抽屉/详情页选择；
- 次要动作、批量动作和快捷入口；
- 项目、合同和人员治理的完整生命周期；
- 审核轮次、异常处置、改派交接等尚无高优先级事实的规则；
- 离线工作包和冲突处理的用户文案与交互细节；
- 低频治理页面是否需要独立高保真母版。

实现者可以提出默认方案，但不得把推演当成已确认业务事实。

### 2.4 已废弃的探索方向

- 独立一级“配置中心”；
- 在项目层直接平铺多个工作流列表（工作流改为归属各履约方案）；
- 流程列表、节点设计和流转规则作为同级菜单；
- 项目自定义通用责任矩阵；
- 把表单列表和编辑器塞进同一个首页；
- 为每个管理对象编写数百行独立规格后才开始开发；
- 用大量静态原型代替真实页面验证；
- 用“规格编写完成”表示产品或工程已经完成。

## 3. 当前实施就绪度

| 维度 | 结论 | 使用方式 |
|---|---|---|
| 产品定位 | 就绪 | 可直接指导三个产品的职责划分 |
| 核心责任链 | 就绪 | 由 DEC-001、DEC-005、DEC-006 约束 |
| 项目配置版本边界 | 就绪 | 由 DEC-003 约束，细节核对现有契约 |
| 工作台定位 | 就绪 | 只做任务投影和深链接 |
| 页面信息架构 | 基本就绪 | 允许在真实实现中调整次级结构 |
| 页面字段与视觉细节 | 未定稿 | 在运行页面中验证，不先继续写长文档 |
| API、数据库和状态机 | 以工程事实为准 | 不从产品稿反向假设已经存在 |
| iOS 离线完整闭环 | 需要逐项核对证据 | 不能因设计稿存在而宣称已实现 |
| 产品验收 | 未完成 | 需要真实角色、真实数据和可运行页面 |

结论：已经足够开始或继续实现纵向切片，但不足以在无人走查真实产品的情况下声明最终产品完成。

## 4. 实施策略

### 4.1 不再先写完所有产品文档

从当前工程基线选择一个完整用户旅程，先核对已有实现，再只补本切片缺少的产品决定、读模型、命令、页面和测试。

推荐第一批产品验证切片：

```text
Admin 工单中心
→ 工单工作区
→ 分配责任网点
→ Network 待分配工单
→ 分配责任师傅
→ iOS 今日任务出现并可进入任务详情
```

这条链路同时检验三个产品、四层责任、授权、页面读模型和任务投影，最能暴露设计与现有工程的差距。

### 4.2 每批只维护必要事实

每个批次只新增或更新：

- 变更的产品决定；
- 机器契约和迁移；
- 实施状态与追踪关系；
- 真实页面验收结果（保存在 PR 或外部验收记录，不新建永久过程文档）。

不为没有近期实施计划的页面建立预防性规格。

### 4.3 产品负责人只参加四类决策门

| 决策门 | 触发条件 | 需要确认的内容 |
|---|---|---|
| 核心语义门 | 状态、责任、SLA、审核、权限或数据范围变化 | 真实业务规则 |
| 首个母版门 | 新页面模式第一次可运行 | 信息优先级、主操作和视觉方向 |
| 纵向切片门 | 一条用户旅程端到端可运行 | 是否解决真实用户任务 |
| 试点发布门 | 准备交给真实用户 | 可用性、风险和上线范围 |

产品负责人不需要审查组件拆分、缓存键、内部类名或常规测试组织。

## 5. Agent 可以自主完成的范围

在已确认业务边界内，Agent 可以自主完成：

- 核对文档、代码、契约、迁移和测试；
- 实现页面、后端读模型、命令和生成客户端；
- 补齐中文文案、错误、日志和数据库描述；
- 编写并运行单元、集成、契约、E2E 和视觉验证；
- 同步 Architecture Book、实施状态与追踪矩阵；
- 清理被替代代码、旧路由、旧文档和无依据兜底；
- 提交、推送并准确报告本地与远端验证。

Agent 不能自主决定：

- 尚无事实依据的核心业务语义；
- 生产数据破坏、授权放宽或敏感数据策略；
- 面向真实用户的最终产品满意度；
- 未经验证即把设计或局部切片声明为完整产品。
