# 破坏性变更与禁止兜底规范

状态：Accepted  
生效日期：2026-07-21

## 1. 适用原则

ServiceOS 当前处于探索开发阶段。旧开发代码、旧测试数据、旧数据库结构、旧接口、旧页面和旧文档不构成兼容义务。

发现不合理设计时，应直接：

- 修改正确模型；
- 删除旧实现；
- 删除旧字段和接口；
- 更新全部调用方；
- 重建开发数据和测试夹具；
- 覆盖或删除过期文档。

不得以降低短期修改量为理由建立兼容层。

## 2. 禁止的兼容设计

禁止新增：

- `legacy`、`compat`、`deprecated`、`old`、`v1Adapter` 目录或类型；
- 旧字段别名；
- 新旧 DTO 同时返回；
- 新旧接口双读或双写；
- 旧路由重定向；
- 新旧页面并行；
- 特性开关保留旧产品；
- 针对旧开发数据的迁移脚本；
- 为旧文档增加“Deprecated”后长期保留。

禁止示例：

```java
try {
    return newClient.query(request);
} catch (Exception exception) {
    return oldClient.query(request);
}
```

```java
String name = displayName != null
        ? displayName
        : legacyName != null
            ? legacyName
            : id.toString();
```

正确做法是只接受唯一新契约，缺少必要数据时明确失败。

## 3. 禁止静默兜底

禁止：

- 名称缺失时展示 ID；
- 中文标签缺失时展示英文枚举；
- 查询失败时返回空列表；
- 配置缺失时使用未经产品确认的默认配置；
- 未知状态按“正常”处理；
- 权限信息缺失时默认允许；
- 外部调用失败时伪造成功结果；
- 操作超时后直接提示成功；
- 捕获异常后仅记录日志并继续业务流程。

必要展示数据缺失时：

- 开发和测试环境应立即失败；
- 契约测试应指出缺失字段；
- 正式环境展示明确的“数据不完整”错误状态；
- 不得泄漏内部 ID、原始 JSON 或英文技术字段。

## 4. 合法状态不是兜底

以下属于正式产品能力，必须完整设计：

- 空状态；
- 错误状态；
- 无权限状态；
- 只读状态；
- 并发冲突状态；
- 网络离线状态；
- 异步操作处理中状态；
- 部分失败和可恢复失败状态。

这些状态必须说明原因、影响和下一步操作，不能用一个通用“操作失败”替代。

## 5. 允许的技术降级

只有在产品和架构事实源中明确批准，并满足以下条件时，才允许技术降级：

- 不改变业务事实；
- 不降低权限和数据安全；
- 用户能够明确看到当前限制；
- 有稳定错误代码和监控；
- 有明确删除条件；
- 不调用旧实现。

例如，H5 不支持后台可靠上传时，应明确提示“该操作需要使用师傅 App”，而不是在页面后台静默等待。

## 6. 代码评审要求

任何涉及破坏性变更的 PR 必须说明：

- 删除了哪些旧实现；
- 哪些调用方已经同步修改；
- 是否重建开发数据；
- 是否删除旧测试和旧文档；
- 是否仍存在同义旧入口；
- 是否出现默认值、双轨或 Adapter。

发现旧实现仍存在时，不得以“下一阶段再删”为常规理由合并。只有新能力尚未完成、且当前 PR 仅为独立工程基础时，才允许暂时不删除运行代码。
