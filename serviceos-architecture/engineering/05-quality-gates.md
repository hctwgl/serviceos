# 自动化质量门禁

状态：Accepted  
生效日期：2026-07-21

## 1. 原则

中文友好、数据库描述完整和禁止兜底不能只依赖人工评审，必须逐步接入 CI。门禁失败时不得合并。

本文件先定义目标规则。后续实现 PR 负责把规则转换为脚本、测试和流水线。

## 2. 后端代码门禁

CI 必须检查：

- 公共领域类、领域服务和应用服务是否具有中文 Javadoc；
- 复杂权限、状态机、SLA、幂等和事务代码是否有中文约束说明；
- 是否存在空的 `catch`；
- 是否捕获宽泛异常后返回空值、默认值或成功；
- 是否根据异常消息字符串判断业务类型；
- 是否出现新接口失败后调用旧接口；
- 是否存在无责任说明的 `TODO`、`FIXME`；
- 是否存在禁止性模糊命名和拼音命名；
- 测试是否具有中文 `DisplayName` 或等价业务描述。

## 3. 日志和错误门禁

CI 必须检查：

- 业务日志是否包含中文事件说明；
- 关键日志是否包含稳定结构字段；
- 错误代码是否唯一且使用稳定英文；
- Problem Details 是否包含中文标题和详情；
- 是否存在“操作失败”“系统异常”等无业务语境信息；
- 是否记录 Token、Authorization、完整手机号、地址等敏感信息；
- 是否仅记录第三方原始异常而缺少中文业务上下文。

敏感日志检查应结合静态规则和测试期日志扫描，不只依赖关键词。

## 4. OpenAPI 门禁

CI 必须检查：

- 每个接口是否存在中文 `summary` 和 `description`；
- 参数、请求和响应字段是否有中文描述；
- 枚举是否有明确中文业务含义；
- 错误响应是否说明原因、可重试性和下一步操作；
- 是否存在旧字段别名；
- 是否存在页面必须展示但契约只返回内部 ID 的字段；
- 生成客户端与当前 OpenAPI 是否一致。

## 5. 数据库门禁

迁移在临时 PostgreSQL 中执行完成后，必须通过 `pg_catalog` 检查：

- 新增表是否具有中文描述；
- 新增字段是否具有中文描述；
- 视图、约束、函数、触发器和序列是否具有描述；
- 数据库对象名称是否符合规范；
- 是否出现 `data`、`info`、`flag`、`ext`、`extra` 等模糊核心字段；
- 是否以 JSON 代替明确核心业务结构；
- 外键、唯一约束和检查约束是否命名清晰；
- 时间、金额和单位是否明确；
- 核心业务是否被隐藏在触发器中。

数据库描述完整性以实际数据库元数据为准，不只扫描 SQL 文本。

## 6. Web 产品门禁

CI 和产品验收必须检查：

- 页面是否展示 UUID、原始 JSON、内部字段或英文枚举；
- 是否存在 Demo Data、Golden Path、Portal Stub 和 Token 登录正式入口；
- 是否使用 ID 代替业务名称；
- 是否由页面连接多个领域接口拼装主要视图；
- 是否覆盖正常、空、错误、无权限、只读和冲突状态；
- 是否通过 TypeScript、组件测试、核心链路 E2E 和无障碍检查；
- 是否在 1280、1440 和 200% 缩放下可用；
- 是否经过产品负责人视觉和任务验收。

自动截图通过不能替代产品批准。

## 7. 原生 iOS 门禁

必须检查：

- 单元测试和任务状态机测试；
- 本地数据库和升级测试；
- 离线命令、幂等和冲突测试；
- 后台上传和网络切换测试；
- 杀进程恢复；
- 相机、定位和权限说明；
- Keychain 和敏感数据保护；
- 真机弱网、断网和低存储测试；
- Archive 构建和发布配置。

iOS 页面不得照搬 H5，不得以 H5 能力限制正式 App。

## 8. 分阶段落地

质量门禁按以下顺序实现：

1. 静态扫描和文档完整性；
2. OpenAPI 和错误模型检查；
3. PostgreSQL 元数据检查；
4. 禁止兜底和敏感日志扫描；
5. Web 产品泄漏扫描和视觉门禁；
6. iOS 真机与生命周期门禁。

在某项自动化尚未实现前，PR 模板必须提供人工检查结果；自动化完成后删除人工替代流程。
