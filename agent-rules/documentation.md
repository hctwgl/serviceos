# 文档、事实源与里程碑规则

在新建里程碑、改变领域语义、同步实施状态、维护 ADR/契约/追踪矩阵或修改 Context Pack 时读取。

## 事实优先级

1. 用户在当前任务中明确批准的最新决策；
2. Accepted ADR；
3. Accepted/Implemented 的架构与验收文档；
4. OpenAPI、事件 Schema、Flyway 等机器契约；
5. 自动化测试证明的行为；
6. 当前代码；
7. README、注释和历史说明。

代码与文档冲突时不得默认代码正确。涉及核心业务语义、破坏性契约、长期架构、安全或数据风险时必须确认。

## 状态语义

- `Draft`：形成中，不可作为研发承诺；
- `Proposed`：完整提案，尚未接受；
- `Accepted`：可指导实现，不代表代码完成；
- `Implemented`：声明范围已有代码、迁移、契约和适用自动化证据；
- `Partial`：已有可靠切片，但领域整体未闭环。

不得把单个样例、通用代码或局部测试外推为所有品牌、区域、服务类型和异常均已完成。

## Context Pack

新里程碑编码前应创建 `serviceos-architecture/context/milestones/<ID>.md`，包含：

- 目标、明确非目标；
- 风险等级、初始阅读预算和影响模块；
- 必须阅读、按需阅读、不需要阅读；
- 已冻结决策、代码入口、验证和待确认问题。

Context Pack 是路由索引，不复制完整业务规则。默认启动只读 `AGENTS.md`、当前轻量基线和 Context Pack；发生冲突或高风险触发条件后才展开完整历史。

## 按影响范围同步

| 变化 | 必须同步 |
|---|---|
| 纯实现修复且外部事实不变 | 测试；必要时修正注释/实现文档 |
| API/事件/数据库变化 | 机器契约或 Flyway + 对应架构/数据/验收文档 |
| 领域规则、状态机、模块职责变化 | 总体设计 + 重大时 ADR + 验收矩阵 |
| 新里程碑或 Implemented 范围变化 | Context Pack、实现文档、验收矩阵、追踪矩阵、状态总览和导航 |

不要为局部实现机械改写全部文档，也不得遗漏真正受影响的权威事实源。

## 状态与历史分层

- `context/current-baseline.md`：当前轻量快照，只保存当前基线、最近变化和主要未闭环范围；
- `docs/implementation-status.md`：完整能力状态和历史证据；
- `docs/implementation-traceability-matrix.md`：模块、不变量、契约和测试索引；
- `architecture/Mxx-*` 与 `testing/Mxx-*`：逐里程碑实现和验收证据；
- `context/modules/`：模块入口卡片；
- `.agent/session-state.md`：本地临时缓存，不是共享事实源。

历史文档不得删除或静默改写，但也不得作为每次任务的默认全文输入。

## 里程碑完成记录

完成新业务里程碑时至少同步：

1. Context Pack 状态；
2. 对应实现文档和验收矩阵；
3. 实施状态中的完成/未完成范围；
4. 追踪矩阵中的代码、契约、迁移和测试入口；
5. 当前轻量基线的最新里程碑与最近摘要；
6. 适用 OpenAPI、事件 Schema、Flyway 和部署迁移说明。

没有工程证据不得标记 `Implemented`。未知提交 SHA 可在功能提交后用紧随其后的状态提交回填。
